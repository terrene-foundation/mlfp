Lesson 3.1 Feature Engineering, ML Pipeline, and Feature Selection

Module 2 gave you statistics and a single linear model. In the next 75 minutes we will answer the question that decides whether a machine-learning project ships or dies: what goes into the model? You will engineer features from a messy multi-table ICU dataset, learn the seven stages of the ML pipeline, and apply three feature-selection families — filter, wrapper, and embedded — to shrink forty-five candidate features down to the ones that actually predict mortality. Along the way you will meet the single most expensive mistake in ML: feature leakage.

Why This Matters

There is an old saying among practitioners: "data beats models, and features beat data." An XGBoost model trained on the wrong features will lose to a logistic regression trained on the right ones — every time. The skill that separates a junior ML engineer from a senior one is not knowing more algorithms; it is knowing how to represent a real problem as a numeric tensor a model can learn from.

Singapore's clinical data illustrates this vividly. A raw ICU record has vital-sign time-series recorded at irregular intervals, medication start and stop times, laboratory panels that are ordered only when the doctor suspects a problem (so the presence of a lab test is itself a signal), and admission metadata. None of this is directly ingestible by a tree-based classifier. Before any model, there is engineering.

Core Concepts

Statistics vs ML: explaining the past vs predicting the future

In Module 2 you fitted a linear regression to HDB prices and reported the coefficients: "every additional square metre is worth about S\$4,200". That is a statistical exercise — the goal is to explain what is in the data using a small number of interpretable parameters.

Machine learning rewrites the goal. A machine-learning model does not care about coefficients; it cares about the expected loss on unseen data:

$$\text{ML objective:} \quad \min_{f} \; \mathbb{E}_{(x, y) \sim \mathcal{D}} \bigl[L\bigl(y, f(x)\bigr)\bigr]$$

where $L$ is a loss function (squared error, log loss, hinge), $f$ is the model, and $\mathcal{D}$ is the unknown distribution that produced both the training and the future data. The statistical exercise cares about inference on the parameters; the ML exercise cares about generalisation to new draws from $\mathcal{D}$. That single shift in objective is what justifies everything we are about to do: we will grow features like weeds because we do not care about their interpretability — we care about whether they help the loss.

🔵 Theory — The training vs test gap

Let $\hat{f}$ be the model fitted on a finite sample. The generalisation gap is $\mathbb{E}_{(x,y) \sim \mathcal{D}}[L(y, \hat{f}(x))] - \hat{L}_{\text{train}}(\hat{f})$. Every design decision in ML — regularisation, cross-validation, feature selection, early stopping — is ultimately a trick to estimate or minimise this gap. We will come back to it in Lesson 3.2 and formally decompose it into bias, variance, and irreducible noise.

Feature engineering: four families that cover most of the work

A feature is any deterministic function of the raw data that you hand to a model as a column. Most features fall into four families:

🟢 Foundations — A feature is a function

Do not think of a feature as "a column in the CSV". Think of it as a function applied to the raw data. The raw data is fixed; the feature space is something you design. The CSV that came out of your hospital database has maybe 30 raw fields; your final feature matrix will have 50–200. You are not transcribing — you are engineering.

Feature leakage: the most expensive mistake in ML

Leakage is when information that will not be available at prediction time sneaks into the training features. It makes your cross-validation scores look spectacular and your deployed model fall over as soon as the first real request arrives.

The rule is brutally simple: for a prediction made at time $t$, only use information produced strictly before $t$. Violate it and your model memorises the future. Enforce it and your cross-validation scores will drop — sometimes by a lot — but they will finally match what you see in production.

Feature selection: three families

You will often engineer many more features than you need. Feature selection is the act of keeping the predictive ones and dropping the noise. There are three classical families.

Filter: score each feature independently

Filter methods rank each feature against the target using a model-free statistic and drop the ones below a threshold. The most common statistic for a continuous target is mutual information:

$$I(X; Y) \;=\; \sum_{x}\sum_{y} p(x, y) \, \log \frac{p(x, y)}{p(x)\,p(y)}$$

$I(X; Y) = 0$ iff $X$ and $Y$ are independent; it captures non-linear dependencies that linear correlation misses. For a categorical target, the chi-squared statistic is the classic choice:

$$\chi^2 \;=\; \sum_{i} \frac{(O_i - E_i)^2}{E_i}$$

where $O_i$ is the observed count in cell $i$ of a contingency table and $E_i$ is the expected count under independence. Large $\chi^2$ means the feature's distribution differs from what independence would predict.

Wrapper: let the model vote

A wrapper method repeatedly re-trains the actual model with different feature subsets and keeps whichever subset scores best on cross-validation. The algorithms are greedy because trying every subset is $O(2^d)$ and intractable for $d > 20$:

  • Forward selection: start with no features; at each step add the feature whose addition most improves CV score. Stop when no addition helps.
  • Backward elimination: start with all features; at each step drop the feature whose removal hurts CV score least. Stop when every remaining feature is essential.
  • Recursive Feature Elimination (RFE): train the model, rank features by importance, drop the lowest, refit, repeat.

Embedded: selection falls out of training

Embedded methods fit one model and extract the selection from the fitted parameters. L1-regularised regression (Lasso) drives irrelevant coefficients to exactly zero — fit once, keep the nonzero ones. Tree ensembles expose a feature_importances_ array derived from total impurity reduction — again, one fit, one ranking. The cost is near zero; the tradeoff is that the selection is tied to the model family you used.

Worked Example: Engineering Features for ICU Mortality

📝 Goal

We have five tables from a Singapore ICU cohort: patients, admissions, vitals, medications, labs. The target is in-hospital mortality — a binary flag. The prediction is made at the end of the first 24 hours of admission. We need one row per admission, with features that obey point-in-time correctness.

Step 1.

Load every table and force the timestamp columns to proper datetimes so we can do temporal filtering later.

import polars as pl
from shared import MLFPDataLoader

loader = MLFPDataLoader()
patients    = loader.load("mlfp02", "icu_patients.parquet")
admissions  = loader.load("mlfp02", "icu_admissions.parquet")
vitals      = loader.load("mlfp02", "icu_vitals.parquet")
medications = loader.load("mlfp02", "icu_medications.parquet")
labs        = loader.load("mlfp02", "icu_labs.parquet")

admissions = admissions.with_columns(
    pl.col("admit_time").str.to_datetime("%Y-%m-%d %H:%M:%S"),
    pl.col("discharge_time").str.to_datetime("%Y-%m-%d %H:%M:%S"),
)
Step 2.

Clip every vital reading to the first 24 hours of the admission it belongs to. This is where point-in-time correctness is enforced: a vital recorded at hour 48 cannot be a feature for a hour-24 prediction.

vitals_24h = (
    vitals.join(admissions.select(["admission_id", "admit_time"]),
                on="admission_id")
          .with_columns(
              hours_since_admit=(
                  pl.col("timestamp") - pl.col("admit_time")
              ).dt.total_hours()
          )
          .filter(pl.col("hours_since_admit") <= 24)
)
Step 3.

Aggregate the time-series into a fixed-shape summary per admission. Mean, std, min, max of heart rate, systolic BP, SpO₂. This collapses an irregular time series into a flat row.

vital_features = vitals_24h.group_by("admission_id").agg([
    pl.col("heart_rate").mean().alias("hr_mean"),
    pl.col("heart_rate").std().alias("hr_std"),
    pl.col("heart_rate").max().alias("hr_max"),
    pl.col("systolic_bp").mean().alias("sbp_mean"),
    pl.col("systolic_bp").min().alias("sbp_min"),
    pl.col("spo2").min().alias("spo2_min"),
    pl.col("heart_rate").count().alias("n_vital_readings"),
])

Note the subtle feature n_vital_readings. In clinical data, the number of times a patient was measured is itself a signal — sicker patients get monitored more frequently. This is an informative missingness pattern, and tree-based models love it.

Step 4.

Engineer a domain-derived feature: the shock index, defined clinically as heart rate divided by systolic BP. A value above 1 is a red flag for early sepsis. This single derived feature often outperforms the two raw inputs.

vital_features = vital_features.with_columns(
    shock_index=pl.col("hr_mean") / pl.col("sbp_mean")
)
Step 5.

Join everything onto the admission table to produce the final feature matrix X and the label vector y.

features = (
    admissions
      .join(patients,       on="patient_id",   how="left")
      .join(vital_features, on="admission_id", how="left")
)

y = features["died"].to_numpy()
X = features.drop(["admission_id", "patient_id", "died",
                   "admit_time", "discharge_time"])  # drop leaky cols
Step 6.

Apply all three feature-selection families and compare the rankings.

from sklearn.feature_selection import mutual_info_classif, RFE
from sklearn.linear_model import LogisticRegression
from sklearn.ensemble import RandomForestClassifier

# 1) Filter: mutual information with the target
mi = mutual_info_classif(X.to_numpy(), y, random_state=42)
filter_top = sorted(zip(X.columns, mi), key=lambda t: -t[1])[:10]

# 2) Wrapper: recursive feature elimination with logistic regression
rfe = RFE(LogisticRegression(max_iter=2000), n_features_to_select=10)
rfe.fit(X.to_numpy(), y)
wrapper_top = [c for c, keep in zip(X.columns, rfe.support_) if keep]

# 3) Embedded: random forest feature importance
rf = RandomForestClassifier(n_estimators=200, random_state=42)
rf.fit(X.to_numpy(), y)
embedded_top = sorted(zip(X.columns, rf.feature_importances_),
                      key=lambda t: -t[1])[:10]

The three rankings will not agree perfectly — and that is the point. The filter method is fast but feature-blind to interactions. The wrapper method tunes to logistic regression specifically. The embedded random-forest importance captures non-linearities. In practice you use the intersection (features that every method ranks highly) as a robust core and the union (features any method ranks highly) as a candidate set for hyperparameter tuning.

Deeper Theory: Why Mutual Information Generalises Correlation

Pearson correlation captures linear dependence — that is, the extent to which $Y$ can be written as $aX + b + \varepsilon$. It is zero for perfect non-linear relationships like $Y = X^2$ when $X$ is symmetric about zero. Mutual information does not have this blind spot. From the definition:

$$I(X; Y) \;=\; H(X) + H(Y) - H(X, Y) \;=\; H(Y) - H(Y \mid X)$$

The last form is the cleanest: mutual information is how much the entropy of $Y$ drops once you know $X$. If knowing $X$ reduces your uncertainty about $Y$ by half, $I(X; Y) = H(Y)/2$. If $X$ is useless, $I(X; Y) = 0$. If $X$ determines $Y$ exactly, $I(X; Y) = H(Y)$. This holds for any functional form — linear, quadratic, threshold, whatever — which is why MI is the default filter statistic when you do not yet know what shape of relationship you are looking for.

🟣 Advanced — MI is invariant to monotonic reparameterisation

A subtle property: $I(X; Y) = I(g(X); Y)$ for any invertible $g$. Scaling, log-transforming, or rank-transforming a feature does not change its MI with the target. That is a property Pearson correlation does not have, which is why an MI-based filter is robust to the unit of measurement in a way that a correlation-based filter is not.

Common Mistakes

⚠ Mistake 1 — Fitting the scaler on the full dataset

A standard scaler learns the mean and std from its input. If you fit it on the combined train+test data before splitting, test-set statistics leak into the training features. Always fit the scaler on train only, then transform both. PreprocessingPipeline enforces this by default; sklearn's Pipeline does too. Doing it by hand is where leakage creeps in.

⚠ Mistake 2 — Using future-looking features

Any column whose value is only known after the prediction time is leakage. Classic offenders: length-of-stay, billed revenue, final diagnosis code, total medications prescribed over the admission. Draw the timeline explicitly and ask, for each feature, "would this number exist if I ran the model now?".

⚠ Mistake 3 — Selecting features before splitting

If you rank features by their correlation with the target and then split, you have already leaked information about the test set into the feature selection. The right order is: split first, then fit feature selection on the train split only, then apply the same selection to the test split.
