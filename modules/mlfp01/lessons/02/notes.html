Speaker Notes — Lesson 1.2 Filtering and Transforming Data

Total time: ~50 minutes  ·  15 slides  ·  ~3 min per slide

This is the first lesson where learners meet a real dataset. The jump from 12 rows to 500,000 is psychologically important — they need to feel that Polars has actually scaled. Run everything live, show the timings, let them see that filtering half a million rows takes less than a second.

Slide 1~2 minTitle

Filtering and Transforming Data

Open with the scale statement: "Yesterday we looked at 12 rows of weather. Today we jump to half a million rows of real HDB transactions. The three verbs you learn today — filter, with_columns, select — are the tools every professional reaches for first when they open a new dataset."

Ask how many learners have used Excel filter dropdowns. Most hands go up. Tell them: "Today we are going to do exactly that work, but in code that can be read, reviewed, and re-run. The Excel filter is a one-shot verbal instruction; the Polars filter is a recipe you can hand to anyone."

→ "Here's the big picture."
Slide 2~3 minNarrowing pipeline diagram

From 500,000 rows to the question

Walk the diagram left to right. "Start with everything. Filter to the rows that matter — BISHAN, 4-room. Then select only the columns you need. Then add any derived columns. At the end you have a tiny table that answers exactly the question you asked."

Emphasise that the diagram is universal. Every analysis they will do for the rest of their career follows this shape.

For beginners: compare to making a sandwich. "You don't use the whole fridge — you pull out the bread, the cheese, and the ham. That's filter. You don't serve everything on the plate — you arrange just the sandwich. That's select. Same idea with data."
→ "The building block is the boolean."
Slide 3~3 minBoolean operators

Booleans: True or False

Walk the six operators. For each, give a concrete example from HDB data. "Greater than 500,000 — is this a premium flat? Less than 50 square metres — is it a studio?"

Hammer the = vs == point. Tell them that Python will actually refuse to run if price = 500_000. This is the language being kind; a careless programmer would silently overwrite price.

If beginners worry about memorising: "You already know these from maths class. The only new ones are == (which is maths's equal sign) and != (which is 'not equal'). Every other operator is what you'd write on paper."
→ "Now the Polars version."
Slide 4~3 minExpression diagram

Polars expressions: recipes, not actions

This is the conceptual crux of the lesson. An expression is a recipe — a description of work — not the work itself. On its own it does nothing. It becomes real only when you hand it to a method that knows what to do with it.

Walk the diagram: expression box on the left, arrow to method, arrow down to the new DataFrame. Say: "pl.col gives you a handle on the column. The comparison gives you a boolean recipe. filter turns the recipe into a new DataFrame. Three steps, three roles."

For experts: this is the declarative / imperative split that makes lazy frames possible. Foreshadow 1.5 without diving in: "We will see in Lesson 1.5 that Polars can actually optimise the whole chain before running a single line — because every step is a description, not an action."
→ "Your first filter."
Slide 5~3 minSingle-town filter

Your first filter

Run it live. Load the HDB parquet. Filter to Ang Mo Kio. Print the count. Make a point of saying "we just scanned 500,000 rows and it took less than a second". This is the moment to make them feel Polars is fast.

Read the filter aloud in plain English: "Filter hdb to rows where the town column equals Ang Mo Kio." The aim is that by the end of today they can translate any Polars filter into an English sentence on sight.

→ "Real filters have more than one condition."
Slide 6~4 minCompound filters

Combining conditions with & and |

Walk both examples. Point out that the parentheses are not optional. Tell them the specific reason: Python's & binds tighter than ==, so an unparenthesised version gets parsed as gibberish.

Show a bad version on the board: pl.col("a") == 1 & pl.col("b") == 2. Ask them what it means. Then show the Python operator precedence and reveal that it becomes pl.col("a") == (1 & pl.col("b")) == 2. The look of horror on their faces is the teaching moment.

For beginners: "When in doubt, parenthesise. Too many parentheses never break a Polars filter; too few almost always do."
→ "For sets, there's a cleaner way."
Slide 7~2 minis_in

is_in: cleaner than chained ORs

Show the ugly version first — four ORs chained together. Then the clean is_in version. Ask which one they'd rather read in a code review. The answer is obvious.

Rule of thumb: "Three or more OR values, reach for is_in." It's shorter, readable, and less error-prone.

→ "Filtering picks rows — with_columns adds columns."
Slide 8~4 minwith_columns

with_columns: derive new features

Walk the two examples. Start with price_per_sqm — the most useful derived feature in HDB analysis. "A 3-room flat in Bishan may cost less than a 5-room in Jurong, but price per sqm tells you which neighbourhood is actually more expensive per unit area. That is a feature — a new quantity that helps answer the question."

Then the multi-column example. Point out that Polars plans all three together — more efficient than chaining three with_columns.

For experts: "This is your first taste of feature engineering. Module 2 is entirely about deciding which features to derive. Today you just learn the mechanics."
→ "What if the new column depends on a condition?"
Slide 9~4 minwhen / then / otherwise

Conditional columns: when / then

Walk the price-tier example line by line. "When price is below 350k, call it budget. When price is below 500k, call it mid-range. When price is below 700k, call it premium. Otherwise call it luxury." Emphasise that order matters — Polars checks each when in order, so the budget check catches everything under 350k before the mid_range check can even look at it.

The pl.lit gotcha is worth five seconds of explicit attention: "If you write .then("budget") Polars will search for a column called budget and throw an error. Wrap constants with pl.lit."

For beginners: compare to Excel's nested IF formulas. "Same logic, cleaner syntax. Here the indentation makes the chain readable; in Excel you'd be staring at IF(IF(IF(...)))."
→ "Now let's put it all together."
Slide 10~3 minChain diagram

Method chaining: the pipeline

Walk the diagram top to bottom. Point at each intermediate row count. "We start with 500,000. After filtering to 2020+ we have 180,000. After filtering to premium/luxury we have 32,000. After select we still have 32,000 but only four columns. After sort we have the same rows but in order by price."

Make the point that each box is a real DataFrame — but Polars doesn't materialise them unless you ask. In lazy mode (coming in 1.5) the entire chain runs in a single pass. For now, treat each step as independent.

→ "Same chain in code."
Slide 11~4 minChain code

The chain in code

Read the code aloud top to bottom. "Take hdb. Filter to recent years. Filter to premium tiers. Select five columns. Sort by price descending. Assign to recent_premium." Pause after each line.

Tell them the outer parentheses around the whole chain are what lets you put each step on its own line. Without them, Python would see a statement that ends after hdb.

Show them how to debug a chain: comment out a line, run, see what you get. Bisection works because each intermediate is itself a valid DataFrame.

For beginners learning fluent API: "You will see this shape hundreds of times. The first three chains feel awkward. By the fifth or sixth, you will start writing them naturally."
→ "A brief look at what's happening underneath."
Slide 12~3 minBoolean algebra

Boolean algebra under the hood

Walk the two definitions and the truth table. Don't dwell. The point is not to become a logician — it's to know that Polars is evaluating these tables across hundreds of thousands of rows simultaneously, and that the rules are entirely standard.

Make the throughput point: "A single vectorised AND across 500,000 rows is about 500,000 truth-table lookups. Polars does that in milliseconds using SIMD instructions on your CPU."

→ "One useful identity — De Morgan's laws."
Slide 13~3 minDe Morgan

🔵 Theory: De Morgan's laws

Introduce as "the useful trick for flipping filters". Walk the two identities. Then show the Polars version: ~pl.col("town").is_in([...]) is how you say "rows where the town is not in this set". The tilde is boolean negation.

Tell them: "This matters because sometimes it's easier to describe the rows you don't want. De Morgan lets you flip between the two descriptions without rewriting half the filter."

For experts: "These identities are also what the Polars query optimiser uses when it rewrites filters for pushdown. Not something you need today, but good to know exists."
→ "Traps to avoid."
Slide 14~3 minCommon mistakes

Three traps to avoid

Walk each one. The parentheses trap will burn every learner at least once — make sure they have seen it now so they recognise the error message when it happens for real.

The and/or trap is specific to pandas refugees. Tell them: "If you have used pandas before, you may be in the habit of using & anyway. Polars enforces it. If you slip up Polars will raise a very loud error."

The assign trap catches everyone once. Show it live: run hdb.filter(pl.col("town") == "BISHAN"), print hdb.height, watch them see that hdb still has 500,000 rows. The lightbulb moment is worth ten minutes of lecture.

→ "Reflect, then break."
Slide 15~3 minReflection + bridge

What you can now do

Run through the five bullets. Pause after each and ask them to silently confirm. If a bullet feels shaky, suggest they revisit that section of the textbook before the exercise.

Tease 1.3: "Today we filtered one town at a time. Tomorrow we will use group_by to summarise every town in a single call. If you understood filter, group_by is half a step away."

Take 5 minutes for questions. Then assign the exercise (ex_2.py) and circulate.
