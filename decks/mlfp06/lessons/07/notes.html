Speaker Notes — Lesson 6.7AI Governance Engineering

Total time: ~75 minutes  ·  9 slides  ·  ~8 min per slide

This lesson is engineering, not philosophy. No AI ethics frameworks. No position papers. The code IS the governance. Hammer this point from Slide 2 onward. The testing slide (Slide 7) is the most important — governance that is not tested does not exist. Singapore context: MAS, PDPA, and AI Verify all require demonstrable access controls and audit trails.

Slides 1–2~8 minTitle + principle

Governance is code

"This lesson is not a discussion about AI ethics. It is an engineering lesson. You will write access controls, operating envelopes, budget cascades, and governance tests. If the code does not deny unauthorised access, your system is ungoverned — no matter how many policy documents you publish." Singapore angle: "MAS requires financial institutions to demonstrate AI governance. 'We have a policy document' is not enough. 'Here is the code that denies unauthorised access, and here are the tests that prove it works' — that is governance."

→ "D/T/R addressing."
Slides 3–5~20 minD/T/R, envelopes, budgets

PACT implementation

Walk D/T/R code: "Domain, Team, Role — three coordinates that identify who is asking. The GovernanceEngine checks whether that address is authorised for the requested resource." Walk envelope diagram: "Nested boxes — outer boundary is never crossed, inner envelopes can only tighten." Walk budget cascade: "Parent allocates, children spend within their allocation, parent total is the sum."

→ "PactGovernedAgent wraps it all."
Slides 6–7~15 minAgent + testing

Governed agent and testing

Walk the PactGovernedAgent code. Then the governance tests — this is the climax. "Test #1: an analyst cannot deploy models. Test #2: an envelope that was tightened cannot be loosened. These are negative tests — they verify what does NOT happen. Governance tests are about proving that the system refuses to do what it should not."

→ "Mistakes."
Slides 8–9~8 minMistakes + recap

Mistakes and recap

Fail-open is the most dangerous: "If your governance check throws an exception and you catch it with a default-allow, every attacker knows to trigger that exception." Preview 6.8: "The capstone deploys everything via Nexus. Train a model, wrap it in an agent, govern it with PACT, deploy via API + CLI + MCP, and monitor with DriftMonitor. The complete stack."

→ End of lesson.
