Lesson 6.2 LoRA, Adapters, and the Fine-Tuning Landscape

You have squeezed every drop of behaviour out of a frozen model with prompting. Now you hit the wall. The model uses the wrong tone, invents Singapore regulations that do not exist, refuses to extract certain fields, or simply does not know your domain well enough. It is time to change the weights. But a 7B model has roughly 14 GB of parameters, and full fine-tuning requires 4× that in optimiser state — you do not want to retrain the whole thing. This lesson teaches you the two techniques that dominate modern fine-tuning (LoRA and adapters), implemented from scratch so you know what is happening, then maps the full landscape of ten fine-tuning techniques so you know when to reach for which.

Why This Matters

Prompting is cheap but limited. Full fine-tuning is powerful but expensive and destructive — it updates every weight, forgets general skills, and costs a small server farm for a serious run. Parameter-efficient fine-tuning (PEFT) is the compromise that made LLM customisation a democratic skill: you update less than 1% of the weights and the base model's knowledge stays intact. Every LLM customisation you see in the wild — from domain chatbots to code assistants to style-specific generators — is almost certainly built on LoRA or its cousins.

The LoRA Insight: Updates Live in a Tiny Subspace

A linear layer in a transformer is just a matrix multiply: $\mathbf{y} = W_0 \mathbf{x}$, where $W_0$ has shape $d \times k$ (say $d = 4096$ rows, $k = 4096$ columns, 16.8M parameters). Fine-tuning would normally add an update $\Delta W$ of the same shape and learn every one of those 16.8M values.

The LoRA paper (Hu et al. 2021) asked: "do we really need $\Delta W$ to be full rank?" Empirically, no. The update matrix that actually emerges from fine-tuning has very low intrinsic rank — the meaningful changes live in a tiny subspace. So you can factor the update as a product of two skinny matrices:

$$W \;=\; W_0 + \Delta W \;=\; W_0 + B A$$
$W_0 \in \mathbb{R}^{d \times k}$ — frozen base weights $B \in \mathbb{R}^{d \times r}$ — learned $A \in \mathbb{R}^{r \times k}$ — learned $r \ll \min(d, k)$

With $d = k = 4096$ and rank $r = 8$, the original $\Delta W$ has $4096 \cdot 4096 \approx 16.8\text{M}$ parameters. The factored version $BA$ has $4096 \cdot 8 + 8 \cdot 4096 \approx 65\text{k}$ parameters — a 250× reduction. And because you keep $W_0$ frozen, you can train on a single GPU what previously required a cluster.

The forward pass

The modified linear layer computes:

$$\mathbf{y} \;=\; W_0 \mathbf{x} + \frac{\alpha}{r}\, B A \mathbf{x}$$

The scaling factor $\alpha / r$ decouples the effective learning rate from the choice of rank: a larger $\alpha$ means the LoRA branch has more influence regardless of $r$. Typical values are $r \in \{4, 8, 16, 32\}$ and $\alpha$ equal to $r$ or $2r$.

Initialisation matters

$A$ is initialised with standard Kaiming uniform; $B$ is initialised to zero. That means at step 0 the LoRA branch contributes nothing ($BA\mathbf{x} = \mathbf{0}$) and the model behaves exactly like the base. Fine-tuning is then a strictly additive perturbation — you cannot accidentally make the model worse on the very first step.

LoRA From Scratch

Here is a minimal LoRA layer you can drop into any PyTorch linear layer. The Kailash AlignmentPipeline does this for you at scale, but implementing it once by hand is the only way to really understand it. 

import math
import torch
import torch.nn as nn

class LoRALayer(nn.Module):
    """Low-rank adapter wrapping a frozen linear layer."""

    def __init__(self, base_layer: nn.Linear, rank: int = 8, alpha: int = 16):
        super().__init__()
        self.base_layer = base_layer
        for p in self.base_layer.parameters():
            p.requires_grad = False        # freeze W_0

        in_features = base_layer.in_features
        out_features = base_layer.out_features
        self.rank = rank
        self.alpha = alpha
        self.scaling = alpha / rank

        # B ∈ R^{d × r}, A ∈ R^{r × k}
        self.lora_A = nn.Parameter(torch.empty(rank, in_features))
        self.lora_B = nn.Parameter(torch.zeros(out_features, rank))
        self.reset_parameters()

    def reset_parameters(self):
        # Kaiming init for A, zero for B
        nn.init.kaiming_uniform_(self.lora_A, a=math.sqrt(5))
        nn.init.zeros_(self.lora_B)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # W_0 x  +  (alpha/r) B A x
        base = self.base_layer(x)
        delta = (x @ self.lora_A.T) @ self.lora_B.T
        return base + delta * self.scaling

    def parameter_count(self) -> int:
        """Return number of *trainable* parameters (excludes frozen base)."""
        return self.lora_A.numel() + self.lora_B.numel()

Count the trainable parameters. lora_A is $r \cdot k$, lora_B is $d \cdot r$, for a total of $r(d + k)$. Compared to full fine-tuning of $d \cdot k$ parameters, the reduction is $r(d+k)/(dk)$. With $d = k = 4096$, $r = 8$, that is $8 \cdot 8192 / (4096 \cdot 4096) \approx 0.4\%$.

🔵 Theory — why zero init on $B$?

With $B$ initialised to zero, $BA\mathbf{x} = \mathbf{0}$ at step 0. The LoRA branch contributes nothing and the forward pass is numerically identical to the frozen base. This guarantees that the first gradient step is the only step that has to deal with the full complexity of the loss landscape at the base's solution point — previous work tried symmetric Gaussian init and found instability. Kaiming-init on $A$ with zero-init on $B$ is the cheapest way to get a clean starting point.

Adapter Layers: The Other PEFT Approach

Before LoRA, the most popular PEFT technique was adapter layers (Houlsby et al. 2019). Adapters insert small bottleneck modules between transformer layers rather than modifying the linear layers themselves. Each adapter projects the activation down to a lower dimension, applies a non-linearity, projects it back up, and adds a residual connection. 

class AdapterLayer(nn.Module):
    """Bottleneck adapter inserted between transformer sublayers."""

    def __init__(self, d_model: int, bottleneck: int = 16):
        super().__init__()
        self.down = nn.Linear(d_model, bottleneck)
        self.act = nn.GELU()
        self.up = nn.Linear(bottleneck, d_model)
        # Near-identity init: zero the up projection
        nn.init.zeros_(self.up.weight)
        nn.init.zeros_(self.up.bias)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        return x + self.up(self.act(self.down(x)))  # residual

    def parameter_count(self) -> int:
        return (
            self.down.weight.numel() + self.down.bias.numel()
            + self.up.weight.numel() + self.up.bias.numel()
        )

LoRA vs adapters — four dimensions

DimensionLoRAAdapters
Update mechanism Additive update to existing linear layer ($W_0 + BA$) New module inserted in the residual path
Parameter efficiency $r(d+k)$ per layer — ~0.4% for $r=8$ $2dm$ per module — comparable for similar $r, m$
Inference cost Zero — $BA$ merges into $W_0$ after training Extra forward pass per adapter — ~5–10% latency
Modularity Swap adapters by swapping $BA$ matrices Swap adapters by loading different modules
🟢 Rule of thumb

Default to LoRA. It has better parameter efficiency, zero inference overhead after merging, and is the de-facto standard backed by every major fine-tuning library. Use adapters only when you need multiple active adapters simultaneously in the same forward pass — because merged LoRA weights are, well, merged.

The Fine-Tuning Landscape: Ten Techniques

LoRA and adapters are the default but they are not the only options. Here is the full landscape, ordered from most parameter-efficient (left) to most expressive (right). Your job is not to memorise them — it is to know that they exist so you can reach for the right one when the default is not working.

Prefix tuning Learned vectors prepended to attention K/V at every layer Task-switching at inference via prefix swap
Prompt tuning Learned soft-prompt embeddings prepended to the input Ultra-low-parameter regime on huge models
Full SFT Backprop through every parameter with schedulers, clipping, mixed precision You have lots of compute and data, and LoRA plateaus
LLRD Layer-wise learning rate decay (lower LR for earlier layers) Full fine-tuning where earlier layers hold general features
Progressive freezing Unfreeze layers top-down as training progresses Small datasets prone to catastrophic forgetting
Knowledge distillation Student model matches teacher's soft-label distribution Compress a large model to a deployable size
Differential privacy (DPSGD) Per-example gradient clipping + noise injection Training data contains PII and must not leak
Elastic Weight Consolidation Fisher Information penalty on critical parameters Sequential multi-task learning; prevent forgetting

Model Merging: Combining Multiple Fine-Tuned Models

You trained one LoRA for legal Q&A and another for retail support. Can you combine them without retraining? Yes — this is model merging. The mathematics is surprisingly clean because LoRA updates are additive; you can linearly combine them.

TIES — Trim, Elect-sign, Merge

TIES (Yadav et al. 2023) addresses sign conflict when merging $n$ models. Different fine-tunings sometimes push a parameter in opposite directions — averaging cancels the signal. TIES fixes this in three steps:

  1. Trim: zero out the smallest magnitude updates in each model (keep the top $k\%$).
  2. Elect sign: for each parameter, the final sign is the one with the greater total magnitude across models.
  3. Merge: average only the updates whose sign agrees with the elected sign.

DARE — Drop And REscale

DARE (Yu et al. 2023) observes that most fine-tuning updates are redundant. Randomly drop a fraction $p$ of the delta parameters (set them to zero) and rescale the survivors by $1/(1-p)$. Surprisingly, you can often drop 90% of the updates and preserve quality. DARE composes with TIES: TIES-DARE is the de-facto merging recipe.

SLERP — Spherical Linear Interpolation

For two models, SLERP interpolates along the great circle on the parameter hypersphere rather than in Euclidean space. The formula for parameter vectors $\theta_1, \theta_2$ at interpolation factor $t \in [0, 1]$ is:

$$\theta_{\text{slerp}} \;=\; \frac{\sin((1-t)\phi)}{\sin \phi}\,\theta_1 + \frac{\sin(t\phi)}{\sin \phi}\,\theta_2$$

where $\phi$ is the angle between $\theta_1$ and $\theta_2$. SLERP is the default for merging exactly two checkpoints where you want smooth interpolation rather than linear averaging. At $t = 0.5$ you get the geodesic midpoint.

🟣 Advanced — task arithmetic

A fun consequence of additivity: task vectors $\Delta \theta_i = \theta_i - \theta_\text{base}$ behave like vectors in parameter space. You can subtract one ($\theta_\text{base} + \Delta\theta_\text{helpful} - \Delta\theta_\text{toxic}$) to remove a behaviour, or add several together to combine skills. Ilharco et al. 2022 call this "task arithmetic". It is hacky, sometimes works, sometimes does not, and is worth knowing about but not relying on.

Quantisation and QLoRA

A 7B parameter model in fp16 is ~14 GB — too much for a consumer GPU. Quantisation compresses the weights to lower precision (int8, int4, even 3-bit) so the model fits. Common formats:

  • GPTQ — post-training, per-layer calibration with a small dataset, typically 4-bit.
  • AWQ — activation-aware weight quantisation; protects salient channels.
  • GGUF — llama.cpp's container format; runs on CPU and Apple Silicon.
  • bitsandbytes — on-the-fly quantisation during PyTorch training.

QLoRA (Dettmers et al. 2023) combines 4-bit quantisation of the base model with LoRA fine-tuning on top. Only the LoRA matrices are in fp16; the frozen base is in 4-bit. Result: you can fine-tune a 65B model on a single 48 GB GPU. QLoRA is the standard recipe for serious LoRA fine-tuning on limited hardware.

Using AlignmentPipeline in Practice

⚙ Kailash — AlignmentPipeline + AdapterRegistry

The kailash-align package wraps all of this. You configure an AlignmentConfig — base model, dataset path, method (sft/dpo/grpo), LoRA hyperparameters — hand it to AlignmentPipeline, and call .train(). The trained adapter is automatically versioned in an AdapterRegistry so you can load, swap, and merge adapters without filesystem plumbing. 

import asyncio
from kailash_align import AlignmentConfig, AlignmentPipeline, AdapterRegistry
from shared.kailash_helpers import setup_environment

setup_environment()

cfg = AlignmentConfig(
    method="sft",
    base_model="TinyLlama/TinyLlama-1.1B-Chat-v1.0",
    train_data="data/mlfp06/sg_domain_qa_train.parquet",
    eval_data="data/mlfp06/sg_domain_qa_eval.parquet",
    # LoRA hyperparameters
    lora_r=8,
    lora_alpha=16,
    lora_dropout=0.05,
    target_modules=["q_proj", "v_proj"],
    # Training
    epochs=3,
    batch_size=4,
    learning_rate=2e-4,
    max_seq_length=512,
    output_dir="artifacts/mlfp06/sft_adapter",
)

async def run_sft():
    pipeline = AlignmentPipeline(cfg)
    result = await pipeline.train()

    registry = AdapterRegistry("artifacts/mlfp06/registry")
    registry.register(
        name="sg_domain_sft",
        version="v1",
        path=result.adapter_path,
        metadata={"base_model": cfg.base_model, "method": "sft", "r": cfg.lora_r},
    )

    print(f"Final train loss: {result.train_loss:.4f}")
    print(f"Final eval loss:  {result.eval_loss:.4f}")
    return result

asyncio.run(run_sft())

Reading the loss curves

During training you will see two numbers: train loss (how well the model fits the training data) and eval loss (how well it generalises to held-out data). Healthy LoRA runs show both curves decreasing smoothly. Warning signs:

  • Eval loss rising while train loss falls — overfitting. Reduce epochs or increase dropout.
  • Both losses flat — learning rate too low, or rank too small for the task.
  • Eval loss starts below 0.5 and never moves — task is too easy or there is train/eval leakage.

Mathematical Foundations: Parameter Count and Compute

Parameter reduction formula

For a single linear layer of shape $d \times k$, the reduction from full fine-tuning to LoRA rank $r$ is:

$$\text{reduction} \;=\; \frac{r(d+k)}{dk}$$

Plug in realistic numbers. For $d = k = 4096$ and $r = 8$: reduction $= 8 \cdot 8192 / (4096 \cdot 4096) \approx 0.004 = 0.4\%$. For $r = 32$: reduction $\approx 1.6\%$. Bumping the rank by a factor of 4 quadruples the parameter count — there is no magic.

Compute cost

LoRA's compute advantage is more modest than its parameter advantage. The forward pass still runs through all of $W_0$; the LoRA branch adds two small matrix multiplies. Gradient computation now only needs to flow through $A$ and $B$, which is where the big savings come from — the optimiser state for Adam (which stores $m$ and $v$ per parameter, doubling memory) is now negligible. In practice LoRA training uses ~30% of the GPU memory of full fine-tuning.

Common Mistakes

⚠ Mistake 1: Targeting the wrong modules

The target_modules argument controls which linear layers get LoRA. Hitting only q_proj and v_proj (the query and value projections in attention) is the safest starting point. Applying LoRA to every linear layer in the model wastes parameters; applying it to only the output projection often underfits.

⚠ Mistake 2: Rank too small for the task

A rank of 4 is enough for style adjustment. Domain knowledge absorption often needs rank 16 or 32. When eval loss plateaus early, the first thing to try is doubling the rank.

⚠ Mistake 3: Forgetting to set the adapter on inference

After training, the base model is still loaded without the LoRA weights unless you explicitly load the adapter. Always check that your inference path calls registry.load("adapter_name", "version") before running predictions.

⚠ Mistake 4: Catastrophic forgetting on small datasets

Training on 50 examples for 20 epochs at a high learning rate can destroy the base model's general skills. Use fewer epochs, lower learning rate, or switch to Progressive Freezing if you see quality drop on tasks you did not explicitly train for.
