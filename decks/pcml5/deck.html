5.1 — What Is a Language Model?

Foundations

A language model does exactly one thing: predict the next word.

Input: "The capital of Singapore is"

Model predicts: "Singapore" (77%), "a" (8%), "the" (3%), ...

That is the entire trick. Every chatbot, every code generator, every AI agent — it is next-word prediction at scale.

Analogy: Autocomplete on your phone, but trained on billions of pages and capable of reasoning about what comes next across paragraphs, not just one word.

What Is a Token?

Foundations

A token is not a word. It is a piece of text — sometimes a word, sometimes a sub-word, sometimes a single character.

"unbelievable" → ["un", "believ", "able"] (3 tokens)

"Singapore" → ["Sing", "apore"] (2 tokens)

"HDB" → ["H", "DB"] (2 tokens)

Why it matters: You pay per token. A 1,000-word document is roughly 1,300 tokens. Cost = input tokens + output tokens.

BPE (Byte Pair Encoding): The algorithm that decides how to split text into tokens. It learns frequent character pairs from training data and merges them iteratively.

BPE: How Tokenization Works

Theory

Byte Pair Encoding builds a vocabulary by iteratively merging the most frequent character pairs:

Start: vocabulary = all individual characters

Step 1: Count all adjacent pairs in corpus. Most frequent: ("t","h") → merge to "th"
Step 2: Re-count. Most frequent: ("th","e") → merge to "the"
Step 3: Re-count. Most frequent: ("i","n") → merge to "in"
...
Stop: when vocabulary reaches target size (e.g., 50,000 tokens)
Key property: Common words become single tokens ("the", "and"). Rare words are split into sub-words. This handles any word — even novel ones — without an unknown-token problem.

The Transformer — Bird's Eye

Foundations

The Transformer (Vaswani et al., 2017) is the architecture behind every modern LLM.

Two key ideas:

  1. Self-attention — every word "looks at" every other word to understand context
  2. Parallel processing — unlike RNNs, all words are processed simultaneously
Encoder-Decoder

Encoder: reads input → builds understanding
Decoder: generates output word by word

BERT = encoder only | GPT = decoder only | T5 = both

Self-Attention — Step by Step

Theory

Given input matrix X \in \mathbb{R}^{n \times d} (n tokens, d dimensions):

Q = XW^Q, \quad K = XW^K, \quad V = XW^V
Q = queries ("what am I looking for?")   K = keys ("what do I contain?")   V = values ("what information do I carry?")

Each token generates three vectors by multiplying with learned weight matrices W^Q, W^K, W^V \in \mathbb{R}^{d \times d_k}.

Intuition: Q is a question each token asks. K is the label each token advertises. V is the actual content. Attention = match questions to labels, then read the content.

Computing Attention Scores

Theory
\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^\top}{\sqrt{d_k}}\right) V

Step by step for a single query token q_i:

1. Score: compute s_{ij} = q_i \cdot k_j for every key k_j (dot product = similarity)
2. Scale: divide by \sqrt{d_k} to prevent extreme values
3. Normalize: apply softmax to get weights \alpha_{ij} = \frac{e^{s_{ij}/\sqrt{d_k}}}{\sum_m e^{s_{im}/\sqrt{d_k}}}
4. Aggregate: output = \sum_j \alpha_{ij} \, v_j (weighted sum of values)

Why Scale by \sqrt{d_k}?

Theory

Without scaling, dot products grow with dimension d_k. This pushes softmax into saturation.

The math: If q_i, k_j \sim \mathcal{N}(0, 1) independently, then:

q \cdot k = \sum_{l=1}^{d_k} q_l k_l
E[q \cdot k] = 0, \quad \text{Var}(q \cdot k) = d_k

Numerically (d_k = 64):

d_kStd(q·k)Max softmax
11.00.73
164.00.98
648.0≈1.00
51222.6≈1.00

At d_k=64, softmax becomes one-hot → no gradient flows → training stalls.

Dividing by \sqrt{d_k} restores \text{Var}(q \cdot k / \sqrt{d_k}) = 1, keeping softmax in its smooth regime where gradients are healthy.

Scaling — Numerical Proof

Theory

Consider d_k = 64, two tokens with random Q, K vectors:

Without scaling

Raw scores: [12.3, -8.7, 15.1, 2.4]

softmax → [0.06, 0.00, 0.94, 0.00]

Nearly one-hot. Gradient ≈ 0.

With /\sqrt{64} = /8

Scaled scores: [1.54, -1.09, 1.89, 0.30]

softmax → [0.24, 0.02, 0.34, 0.07]

Smooth distribution. Healthy gradients.
Why this matters: Without this one division, Transformers cannot train. Self-attention without scaling = a broken architecture.

Self-Attention: Complete Derivation

Theory

Putting it all together for input sequence X = [x_1, x_2, \ldots, x_n]^\top \in \mathbb{R}^{n \times d}:

Q = XW^Q \in \mathbb{R}^{n \times d_k}, \quad K = XW^K \in \mathbb{R}^{n \times d_k}, \quad V = XW^V \in \mathbb{R}^{n \times d_v}
A = \text{softmax}\!\left(\frac{QK^\top}{\sqrt{d_k}}\right) \in \mathbb{R}^{n \times n}
A_ij = how much token i attends to token j (each row sums to 1)
\text{Output} = AV \in \mathbb{R}^{n \times d_v}
Each output row is a weighted sum of value vectors, weighted by attention

Total learnable parameters per head: W^Q, W^K \in \mathbb{R}^{d \times d_k} and W^V \in \mathbb{R}^{d \times d_v} = d(2d_k + d_v) parameters.

Multi-Head Attention

Theory

Instead of one attention function, run h attention "heads" in parallel, each with its own learned projections:

\text{MultiHead}(Q,K,V) = \text{Concat}(\text{head}_1, \ldots, \text{head}_h) \, W^O
\text{where head}_i = \text{Attention}(XW_i^Q, XW_i^K, XW_i^V)
Intuition: Each head learns a different relationship. Head 1 might track syntax ("subject-verb"). Head 2 might track co-reference ("he = John"). Head 3 might track sentiment.

Typical: d_{model} = 512, h = 8, d_k = d_{model}/h = 64. Same total compute as single-head, but richer representation.

Positional Encoding

Theory

Attention is permutation-invariant — it has no notion of word order. We must inject position information.

Sinusoidal (original)

PE_{(pos, 2i)} = \sin\!\left(\frac{pos}{10000^{2i/d}}\right)
PE_{(pos, 2i+1)} = \cos\!\left(\frac{pos}{10000^{2i/d}}\right)

Each dimension oscillates at a different frequency → unique fingerprint per position.

Modern alternatives

RoPE (Rotary Position Embedding):
Rotate Q and K vectors by position-dependent angle. Encodes relative position in the attention score itself.

ALiBi (Attention with Linear Biases):
No embedding — subtract a linear penalty m \cdot |i-j| from attention scores. Simpler, better extrapolation.

RoPE: Rotary Position Embeddings

Advanced

RoPE encodes position by rotating query and key vectors in 2D subspaces:

f_q(x_m, m) = R_{\Theta,m} W^Q x_m, \quad f_k(x_n, n) = R_{\Theta,n} W^K x_n

where the rotation matrix for position m applies to each pair of dimensions:

R_{\Theta,m} = \begin{pmatrix} \cos m\theta_1 & -\sin m\theta_1 \\ \sin m\theta_1 & \cos m\theta_1 \end{pmatrix} \oplus \cdots \oplus \begin{pmatrix} \cos m\theta_{d/2} & -\sin m\theta_{d/2} \\ \sin m\theta_{d/2} & \cos m\theta_{d/2} \end{pmatrix}
Key property: f_q(x_m,m)^\top f_k(x_n,n) depends on m-n (relative position), not absolute positions. This gives length generalization — the model can extrapolate to longer sequences than it was trained on.

Flash Attention

Advanced

Problem: Standard attention materializes the full n \times n attention matrix in GPU HBM. For n=8192, that is 512 MB per head per layer.

Standard Attention

1. Compute S = QK^\top → write to HBM
2. Compute P = \text{softmax}(S) → write to HBM
3. Compute O = PV

IO: O(n^2) reads/writes to slow HBM

Flash Attention (Dao, 2022)

1. Tile Q, K, V into blocks that fit in SRAM
2. Compute attention per block in fast SRAM
3. Accumulate output with online softmax

IO: O(n^2 d / M) where M = SRAM size
Key insight: IO-aware tiling. Never materialize the full attention matrix. 2-4x wall-clock speedup, exact (not approximate).

GQA and MQA

Advanced

Reduce KV-cache memory by sharing key-value heads across query heads:

Multi-Head (MHA)32321xGPT-3, BERT

Why it matters for agents: Smaller KV-cache = longer context windows = agents that can read more documents before acting. Llama 3 uses GQA-8 to support 128k context.

Speculative Decoding

Advanced

LLM inference is memory-bandwidth-bound, not compute-bound. One token at a time wastes GPU compute.

Idea: Use a small draft model to generate \gamma candidate tokens cheaply, then verify all of them in one pass through the large model.

1. Draft model generates: ["the", "cat", "sat", "on"]
2. Target model verifies all 4 in parallel
3. Accept first 3 (match), reject "on" (target says "upon"), generate from "upon"

Result: 2-3x speedup with exact same output distribution as the large model alone.
Formally: acceptance probability via modified rejection sampling ensures the output distribution is identical to running the target model alone (Leviathan et al., 2023).

5.2 — Chain-of-Thought Reasoning

Foundations

LLMs are better at hard problems when they "show their work".

Direct prompting

Q: "If a model has 85% accuracy on 70/30 balanced data, what is its F1 score?"

A: "0.85" (wrong — assumes balanced predictions)

Chain-of-Thought

Q: "...Let's think step by step."

A: "Accuracy = (TP+TN)/(TP+TN+FP+FN). For F1 we need precision and recall separately. Without knowing the confusion matrix, F1 cannot be derived from accuracy alone."
(correct reasoning)
Why it works: Each reasoning step constrains the next step. The LLM builds up an intermediate representation in its own output that guides subsequent predictions.

CoT — Formal View

Theory

Without CoT: P(a \mid q) — directly predict answer from question.

With CoT: P(a \mid q) = \sum_{r} P(a \mid q, r) \cdot P(r \mid q)

P(\text{answer} \mid \text{question}) = \sum_{\text{reasoning}} P(\text{answer} \mid \text{question}, \text{reasoning}) \cdot P(\text{reasoning} \mid \text{question})
Marginalize over all possible reasoning chains

In practice, we sample one chain (greedy or beam search), not sum over all.

Computational interpretation: CoT gives the model intermediate "scratch space." Without it, the model must solve the problem in a single forward pass. With it, each step is a separate forward pass that can condition on prior steps.

Beyond CoT: Tree and Graph of Thought

Advanced

Tree-of-Thought (Yao et al., 2023)

Branch into multiple reasoning paths at each step. Evaluate each branch with a value function. Prune bad branches, expand promising ones. BFS or DFS over the thought tree.

Graph-of-Thought (Besta et al., 2024)

Allow reasoning steps to merge, creating a DAG. Combine partial solutions from different branches. More flexible than tree structure — models real problem-solving better.

Production trade-off: CoT adds ~2x token cost. ToT adds ~5-20x. For most agent tasks, CoT is sufficient. ToT shines on complex planning problems (e.g., multi-step data pipeline design).

5.3 — What Is an Agent?

Foundations

An agent is an LLM that can take actions, not just generate text.

Chatbot (no tools)

User: "What is the average HDB price?"
Bot: "Based on my training data, the average HDB resale price was approximately $500,000 in 2023."

Might be outdated or wrong.

Agent (with tools)

User: "What is the average HDB price?"
Agent: Calls DataExplorer on live HDB data
Agent: "The current average HDB resale price is $582,341 based on Q3 2025 data."

Verified from source.

The ReAct Loop

Foundations

ReAct (Yao et al., 2023) = Reasoning + Acting, interleaved.

Thought
"I need to check the latest prices"
→
Action
Call DataExplorer.profile()
→
Observation
"Mean price: $582K, std: $185K"
→
Thought
"Let me check the trend..."

Loop continues until the agent has enough information to answer.

ReAct — Formal Definition

Theory

At each step t, the agent generates:

a_t \sim \pi_\theta(\cdot \mid s_t), \quad s_t = (q, \, c_1, a_1, o_1, \ldots, c_{t-1}, a_{t-1}, o_{t-1})
q = user query   c_t = thought (chain-of-thought reasoning)   a_t = action (tool call)   o_t = observation (tool result)

The policy \pi_\theta is the LLM itself. The state s_t is the full conversation history. Actions are tool invocations with structured arguments.

Key insight: The thought step is not decorative — it forces the LLM to plan before acting. Ablation studies show removing thoughts degrades tool-selection accuracy by 15-30%.

Tool Use Formalization

Theory

Each tool is a function with a typed signature:

\text{tool}_i : \mathcal{X}_i \rightarrow \mathcal{Y}_i
Input space X_i (parameters) → output space Y_i (results)

The agent must solve a tool selection problem at each step:

Given tools \{t_1, \ldots, t_k\} and current state s_t:

1. Select tool: i^* = \arg\max_i P(t_i \mid s_t)
2. Generate arguments: x^* \sim P(\cdot \mid s_t, t_{i^*})
3. Execute: o_t = t_{i^*}(x^*)
4. Update state: s_{t+1} = s_t \oplus (c_t, a_t, o_t)

Chunking Strategies

Theory

Fixed-size chunking

Split every N tokens with M token overlap.

Pros: Simple, predictable
Cons: Cuts mid-sentence, loses context

Semantic chunking

Split on topic boundaries using embedding similarity.

Pros: Preserves meaning
Cons: Slower, variable size

Document-structure chunking

Split on headers, paragraphs, sections.

Pros: Respects document hierarchy
Cons: Requires structured input

Recursive chunking

Try paragraph → sentence → token splits in order.

Pros: Adaptive, maintains coherence
Cons: Implementation complexity

Theory

Re-Ranking with Cross-Encoders

Theory

Bi-encoder (retrieval)

Encode query and document separately

\text{score} = \cos(E_q, E_d)

Fast: encode once, compare with millions
But: no cross-attention between q and d

Cross-encoder (re-ranking)

Encode query and document together

\text{score} = f([q; d])

Slow: must run for each (q, d) pair
But: full cross-attention = much more accurate
Production pattern: Bi-encoder retrieves top-100 candidates cheaply. Cross-encoder re-ranks to top-5 accurately. Two-stage pipeline gives the best quality-latency trade-off.

Theory

HyDE: Hypothetical Document Embeddings

Theory

Problem: User queries are short and vague. Documents are long and specific. The embedding space gap hurts retrieval.

HyDE approach (Gao et al., 2023):

1. User asks: "What causes HDB price drops?"
2. LLM generates a hypothetical answer (may be wrong, but has the right vocabulary)
3. Embed the hypothetical answer instead of the query
4. Retrieve documents similar to the hypothetical answer
5. Generate final answer from retrieved real documents
Why it works: The hypothetical document is in the same "language" as real documents — long-form, technical, domain-specific. This bridges the query-document gap. Retrieval precision improves 10-30% on domain-specific tasks.

Self-RAG and Corrective RAG

Advanced

Self-RAG (Asai et al., 2023)

Model decides when to retrieve. Generates a "retrieve" token, fetches context, then generates a "relevance" critique token. Only uses retrieval when needed — reduces latency for simple questions.

Corrective RAG (Yan et al., 2024)

Evaluates retrieved documents for relevance. If confidence is low, performs web search as fallback. Decomposes then recomposes knowledge. Self-correcting retrieval pipeline.

Key insight: Standard RAG always retrieves, even for questions the model can answer directly. Self-RAG/CRAG add a retrieval gate — retrieve only when uncertain. This reduces latency by 30-50% on average.

5.7 — Multi-Agent Patterns

Foundations

Complex tasks require multiple agents working together. Four production patterns:

Supervisor-Worker

One agent assigns tasks, workers execute. Supervisor decides next step based on results.

Sequential

Agents run in order, each output feeds the next. Pipeline pattern.

Parallel

Multiple agents run simultaneously on independent sub-tasks. Results are aggregated.

Handoff

One agent works until it cannot continue, then hands off to a specialist.

Handoff Pattern

Theory

An agent works until it reaches the boundary of its expertise, then transfers control to a specialist:

Example: Customer support pipeline

1. Triage Agent receives query: "My model predictions are worse this week"
2. Triage identifies: drift-related → hands off to DriftAnalyst
3. DriftAnalyst finds PSI > 0.2 on 3 features → hands off to RetrainingDecision
4. RetrainingDecision recommends: "Retrain with last 90 days of data" 
pipeline = HandoffPattern(
    entry_agent=triage_agent,
    handoff_map={
        "drift": drift_analyst,
        "training": model_selector,
        "data_quality": data_scientist,
    },
)
result = await pipeline.run(user_query)

Debate Pattern

Theory

Two or more agents argue opposing positions. A judge agent evaluates and decides.

Use case: Model selection with conflicting requirements

Agent A (accuracy advocate): "XGBoost achieves 0.92 AUC — highest accuracy."
Agent B (interpretability advocate): "Logistic regression achieves 0.87 AUC but is fully interpretable for regulatory compliance."
Judge: "For a regulated financial application, interpretability at 0.87 AUC is preferable to black-box 0.92 AUC. Recommend logistic regression with SHAP overlay."
Research connection: Debate is inspired by AI safety work (Irving et al., 2018). The idea: if two AI agents debate, a human judge can identify the truthful one — even when neither agent alone is fully trustworthy.

Discussion: Cost and Scale

Theory
Your multi-agent pipeline runs 6 LLM calls per request. Each call costs $0.002. You expect 10,000 requests per day.

Daily cost: 6 × $0.002 × 10,000 = $120/day = $3,600/month

Questions:
  1. Is this reasonable? How does it compare to a human analyst?
  2. Which agents could be cached? Which must run fresh every time?
  3. How would you reduce costs without reducing quality?

Synthesis — Key Takeaways

Everyone should remember:
  • An LLM predicts the next token — agents add tools, memory, and structure
  • RAG = look up, then answer. No hallucination when grounded in sources
  • Cost budgets are enforced by the framework, not by trust
  • Nexus deploys as API + CLI + MCP from one codebase
If you followed the math:
  • Self-attention is Q, K, V with sqrt(d_k) scaling to prevent softmax saturation
  • ReAct formalizes the thought-action-observation loop as a policy over states
  • Hybrid retrieval (BM25 + dense + re-ranking) dominates pure approaches
  • RAGAS gives you four metrics to evaluate RAG pipeline quality

Synthesis — Expert Insights

For the experts:
  • Flash Attention is IO-aware tiling — exact, not approximate — 2-4x speedup
  • GQA reduces KV-cache memory, enabling longer context windows
  • Speculative decoding gives 2-3x inference speedup with identical output distribution
  • Self-RAG and CRAG add retrieval gating — retrieve only when uncertain
  • Tree/Graph-of-Thought generalize CoT but at 5-20x token cost
