Pre-training optimises prediction, not helpfulness

ZERO-SHOT

Task description only. No examples.

Works well for simple, well-defined tasks. Fails on

prompt = """Classify sentiment as 'positive' or 'neg

Review: "The food was terrible and the service slow.
Sentiment:"""

Few-shot is in-context learning: the model infers the task
from examples without weight updates.

import torch
import torch.nn as nn
class LoRALayer(nn.Module):
"""Low-Rank Adaptation layer."""
def __init__(self, in_features: int, out_features: int, rank: int = 8):
super().__init__()
self.rank = rank
# Frozen original weights
self.linear = nn.Linear(in_features, out_features, bias=False)
self.linear.weight.requires_grad = False
# Low-rank matrices
self.A = nn.Parameter(torch.randn(rank, in_features) * 0.01)
self.B = nn.Parameter(torch.zeros(out_features, rank))
# A is initialised with small random values; B with zeros
# So BA = 0 at init: adaptation starts from the original model
def forward(self, x: torch.Tensor) -> torch.Tensor:
original = self.linear(x)
# W_0 @ x
adaptation = (x @ self.A.T) @ self.B.T # x @ A^T @ B^T = (BA)x
return original + adaptation

Why B=0? At initialisation, BA = 0, so the model starts as the original pre-trained model. Training gradually learns the
adaptation. This is why LoRA is safe: you cannot make the model worse at the start.

import torch
import torch.nn.functional as F
def dpo_loss(
policy_chosen_logps: torch.Tensor,
# log pi(y_w|x)
policy_rejected_logps: torch.Tensor, # log pi(y_l|x)
ref_chosen_logps: torch.Tensor,
# log pi_ref(y_w|x)
ref_rejected_logps: torch.Tensor,
# log pi_ref(y_l|x)
beta: float = 0.1,
) -> torch.Tensor:
"""Direct Preference Optimization loss."""
chosen_rewards = beta * (policy_chosen_logps - ref_chosen_logps)
rejected_rewards = beta * (policy_rejected_logps - ref_rejected_logps)
loss = -F.logsigmoid(chosen_rewards - rejected_rewards).mean()
return loss

OPERATING ENVELOPES
WHAT ARE OPERATING ENVELOPES?

A ConstraintEnvelopeConfig defines the
boundaries of what a role can do
Five canonical dimensions: Financial,
Operational, Temporal, Data Access,
Communication — plus a confidentiality clearance
and a delegation-depth cap
Monotonic tightening: child envelopes must be
≤ parent on every dimension; the framework
catches violations via
RoleEnvelope.validate_tightening

Fail-closed: If the governance check fails (error, timeout),
access is denied. Never fail-open. A broken guard is not a
free pass.

KEY TAKEAWAYS

1. Prompting is an engineering skill. Start simple,
escalate by measured need.
2. LoRA democratised fine-tuning. 256x fewer
parameters, same quality.
3. DPO eliminated the reward model. Alignment is
now accessible.
4. RAG is the production default. Ground LLMs in
facts, not memory.

FOUNDATIONS

5. Agents need tools, not just prompts. ReAct =
reasoning + action.
6. Multi-agent systems need coordination
protocols. MCP for tools, A2A for agents.
7. Governance is engineering, not philosophy.
Code it. Test it. Ship it.
8. Deploy with monitoring. A model without
DriftMonitor is a liability.
