A1 — Decoder-Only Transformer

Foundations

Modern LLMs use decoder-only transformers — the right half of the original transformer from M8.

Encoder-Decoder (M8)

  • Reads full input, then generates output
  • Used for: translation, summarization
  • Example: T5, BART

Decoder-Only (LLMs)

  • Generates tokens left-to-right
  • Each token attends only to previous tokens
  • Used for: chatbots, code gen, reasoning
  • Example: GPT, Claude, Llama
Key insight: Decoder-only is simpler. One architecture, one objective (predict next token), scales to trillions of tokens of training data.
Time: ~2 min. Recap quickly from M8: "Remember the transformer had two halves — encoder and decoder. Modern LLMs threw away the encoder. Why? Because predicting the next token is all you need if you have enough data." Beginners: "Think of it as autocomplete on steroids."

Causal Masking

Theory

In decoder-only models, each token can only "see" tokens that came before it. This is called causal masking.

\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^T}{\sqrt{d_k}} + M\right)V
M is the causal mask: upper-triangle = -infinity, lower-triangle = 0
Token 1 sees: [Token 1]
Token 2 sees: [Token 1, Token 2]
Token 3 sees: [Token 1, Token 2, Token 3]
Token N sees: [Token 1, Token 2, ..., Token N]
Time: ~2 min. Explain the mask visually: "Imagine a grid. The model can only look down and to the left — never forward. This prevents it from 'cheating' by reading the answer before generating it." For experts: "This is why decoder-only models are autoregressive — each output depends only on past outputs."

RoPE — Rotary Position Encoding

Theory

RoPE encodes position by rotating query and key vectors in 2D subspaces:

f(x_m, m) = x_m e^{im\theta} = \begin{pmatrix} \cos m\theta & -\sin m\theta \\ \sin m\theta & \cos m\theta \end{pmatrix} \begin{pmatrix} x_m^{(1)} \\ x_m^{(2)} \end{pmatrix}
m = position index, theta = frequency, x = embedding pair
  • Relative position: attention between tokens depends on distance, not absolute position
  • Extrapolation: models can handle longer sequences than seen during training
  • No learned parameters: computed directly from position
Time: ~2 min. Intuition: "Imagine each token is a clock hand. Its position is encoded by how far the hand has rotated. Two tokens that are close together have similar angles — so their attention is naturally high." For beginners: "Don't worry about the matrix. The key idea: position is encoded as rotation, and rotation is easy for the model to compute."

GQA — Grouped-Query Attention

Advanced

Multi-Head Attention (MHA)

  • Each head has its own Q, K, V
  • Maximum expressiveness
  • KV cache grows linearly with heads

Multi-Query Attention (MQA)

  • All heads share one K, V
  • Minimum KV cache
  • Potential quality loss

Grouped-Query (GQA)

  • Groups of heads share K, V
  • Sweet spot: 4-8 groups
  • Near-MHA quality, near-MQA speed
DeepSeek MLA: Multi-Head Latent Attention compresses KV cache by 93.3% via learned low-rank projection — even more efficient than GQA.
Time: ~3 min. For experts: "GQA was the key innovation enabling Llama 2's 70B model to run efficiently on consumer hardware. DeepSeek went further with MLA, which projects keys and values into a much smaller latent space." Beginners: "The takeaway — newer models are more memory-efficient. A model that needed 4 GPUs in 2023 can run on 1 GPU in 2025."

KV Cache — Why Inference Memory Grows

Advanced

During generation, the model reuses key-value pairs from all previous tokens. This KV cache grows with context length:

\text{KV cache} = 2 \times n_{\text{layers}} \times n_{\text{heads}} \times d_{\text{head}} \times \text{seq\_len} \times \text{precision}
Factor of 2 for K and V; grows linearly with sequence length
Example — Llama 3 70B at FP16, 128K context:
2 x 80 layers x 8 KV heads x 128 dim x 128,000 tokens x 2 bytes = ~42 GB just for KV cache
Why context length is expensive: Doubling the context doubles the KV cache memory. This is why 1M-token models require significant engineering (sliding window, paged attention, KV cache compression).
Time: ~2 min. Analogy: "Imagine you're writing an essay and you need to remember every word you've already written. The longer the essay, the more memory you need. That's the KV cache." For experts: "Paged attention (vLLM) treats KV cache like virtual memory — pages in/out based on demand. Flash Attention reduces memory by computing attention in tiles."

A2 — Pre-Training: Next Token Prediction

Foundations

Pre-training teaches the model language by predicting the next word, billions of times:

\mathcal{L} = -\sum_{t=1}^{T} \log P(x_t \mid x_1, \ldots, x_{t-1}; \theta)
Minimize negative log-likelihood of next token prediction across entire corpus
Training data (typical): Common Crawl (web), books, Wikipedia, code (GitHub), scientific papers — trillions of tokens total
Analogy: A child learns language by exposure — hearing millions of sentences. An LLM learns by reading trillions of sentences and trying to predict what comes next.
Time: ~2 min. Emphasize the simplicity: "The entire training objective is: predict the next word. That's it. Everything else — understanding, reasoning, coding — emerges from doing this prediction at massive scale." Beginners: "It's like the world's most intense autocomplete training."

Scaling Laws

Theory

The Chinchilla scaling law tells us how to allocate a fixed compute budget optimally:

C \approx 6ND
C = compute (FLOPs), N = parameters, D = data tokens. Optimal: scale N and D equally.

Compute-Optimal

  • For fixed budget C, optimal N and D are equal
  • Chinchilla (70B, 1.4T tokens) beat Gopher (280B, 300B tokens)

Inference-Optimal (2024+)

  • Over-train smaller models beyond Chinchilla optimal
  • Llama 3 8B trained on 15T tokens (18x "optimal")
  • Cheaper to deploy: small model, high quality
Time: ~2 min. Key insight: "Chinchilla told us to train models on more data than anyone expected. But Llama 3 showed that if you care about deployment cost, you should over-train even further — making a small model punch above its weight by feeding it way more data."

Test-Time Compute Scaling

Advanced

A paradigm shift: instead of making models bigger, let them think longer.

Train-Time Scaling (traditional)

More parameters, more data, more compute during training. Fixed cost at inference.

Test-Time Scaling (reasoning models)

Generate extended chain-of-thought at inference. Variable cost per query. Harder queries get more compute.

The insight: DeepSeek-R1 matches OpenAI o1 by generating 10-100x more tokens per query. Instead of training a bigger model, it thinks harder about each problem. Cost per easy question stays low; cost per hard question goes up.
Time: ~3 min. This is one of the biggest shifts in 2024-2025. "Traditional scaling: make the model bigger before deployment, fixed cost per query. Test-time scaling: let the model generate more reasoning tokens at inference, variable cost per query." For experts: "GRPO — Group Relative Policy Optimization — is how DeepSeek trained R1 to reason, cutting RL costs by ~50% compared to PPO."

SwiGLU Activation Function

Theory

SwiGLU replaced ReLU/GELU in modern LLMs for better gradient flow:

\text{SwiGLU}(x, W, V, b, c) = \text{Swish}(xW + b) \otimes (xV + c)
Swish(x) = x * sigmoid(x). The gating mechanism (element-wise multiply) enables selective information flow.
  • Gating: The second linear transformation gates the first — not all information passes through
  • Trade-off: 50% more parameters in FFN layer, but better quality per FLOP
  • Adopted by: Llama 2+, Mistral, Qwen, DeepSeek — essentially all post-2023 models
Time: ~2 min. Intuition: "SwiGLU is like having two filters instead of one. The first filter processes the information. The second filter decides how much of that processed information to let through." For beginners: "The key takeaway is that modern models use smarter activation functions that learn better representations."

RMSNorm vs LayerNorm

Theory

LayerNorm (original)

\text{LN}(x) = \frac{x - \mu}{\sigma} \cdot \gamma + \beta

Subtracts mean, divides by std. Two learned parameters per dimension.

RMSNorm (modern)

\text{RMSNorm}(x) = \frac{x}{\text{RMS}(x)} \cdot \gamma

Divides by root-mean-square only. One learned parameter. ~10% faster.

Why RMSNorm won: Zhang & Sennrich (2019) showed the mean subtraction in LayerNorm is unnecessary — RMS normalization alone stabilizes training equally well, with fewer parameters and less compute.
Time: ~2 min. "RMSNorm is simpler and faster than LayerNorm. It skips the mean-centering step, which turns out not to matter much for training stability. Every modern LLM uses RMSNorm." For beginners: "Normalization keeps the numbers in the model from getting too large or too small. RMSNorm does this efficiently."

Mixture-of-Experts (MoE) Architecture

Theory

MoE activates only a subset of parameters for each token:

Dense model: Every token uses ALL 70B parameters → 70B compute
MoE model: Router selects 2 of 16 experts per token → only 37B compute
Total parameters: 671B (large capacity) | Active parameters: 37B (fast inference)
y = \sum_{i=1}^{N} g(x)_i \cdot E_i(x), \quad g(x) = \text{TopK}(\text{softmax}(Wx))
g(x) = router/gating function, E_i = expert networks, TopK selects active experts
Time: ~3 min. "MoE is the breakthrough that lets models have huge capacity without huge inference cost. Think of it like a hospital: you have 100 specialist doctors, but each patient only sees 2-3 of them. The router decides which specialists are needed." Examples: DeepSeek V3, Qwen 3.5, Mistral Large 3, Llama 4 Maverick.

MoE: Router Design and Load Balancing

Advanced

Router Challenges

  • Expert collapse: All tokens routed to same few experts
  • Load imbalance: Some experts overloaded, others idle
  • Training instability: Router gradients are noisy

Solutions

  • Load balancing loss: Penalty for uneven expert usage
  • Capacity factor: Limit tokens per expert per batch
  • Shared experts: Some experts always active (DeepSeek V3)
  • Fine-grained routing: DeepSeek uses 256 small experts (vs 8-16 large)
DeepSeek V3 innovation: 256 routed experts with 8 active per token, plus 1 always-active shared expert. Fine-grained routing improves specialization — each expert handles a narrower domain.
Time: ~2 min. For experts: "The router is the hardest part of MoE to get right. If the router collapses (sends everything to the same expert), you've wasted all those parameters. DeepSeek's fine-grained approach with 256 small experts seems to solve this — each expert naturally specializes."

Speculative Decoding

Advanced

Speed up large model inference by using a small "draft" model:

Draft model
(fast, small)
→
Generate K tokens
(quickly)
→
Target model
(large, slow)

Key insight: Verifying K tokens in parallel is the same cost as generating 1 token. If the draft model is accurate 80% of the time, you get ~3-4x speedup with zero quality loss.

Flash Attention

Advanced

Flash Attention makes long-context models practical by optimizing GPU memory access:

Standard Attention

  • Compute full N x N attention matrix
  • Store entire matrix in GPU memory (HBM)
  • Memory: O(N^2) — quadratic in sequence length
  • 128K context = 128K x 128K matrix = impossible

Flash Attention (IO-aware)

  • Compute attention in tiles (blocks)
  • Keep tiles in fast SRAM, not slow HBM
  • Memory: O(N) — linear in sequence length
  • 128K context = practical on single GPU
FlashAttention-3 (2024): Adds asynchronous computation, overlapping memory transfers with compute. 2x faster than FlashAttention-2 on Hopper GPUs (H100). This is what enables 1M+ token context windows.
Time: ~2 min. Analogy: "Standard attention is like reading an entire library to find one fact. Flash Attention brings relevant books to your desk in batches, reads them quickly, and puts them back. Same result, dramatically less memory."

GRPO: How DeepSeek Trained R1

Advanced

Group Relative Policy Optimization — train reasoning without a reward model:

\hat{A}_i = \frac{r_i - \text{mean}(r_1, \ldots, r_G)}{\text{std}(r_1, \ldots, r_G)}
Advantage = how much better this output is compared to others in its group
Process:
1. Generate G outputs for the same prompt
2. Score each output with a simple verifier (e.g., "is the math correct?")
3. Compute relative advantage within the group
4. Reinforce outputs that scored above average
No reward model needed: Traditional RLHF trains a separate reward model (expensive). GRPO uses group-relative scoring — no separate model, ~50% cheaper than PPO.
Time: ~2 min. For experts: "GRPO is elegant: instead of learning a complex reward function, you just compare outputs within a batch. If one solution is better than the group average, reinforce it. If worse, penalize it. This is enough to train frontier-quality reasoning."

Parallel Attention + FFN

Theory

Traditional transformers run attention and FFN sequentially. Some models run them in parallel:

Sequential (standard)

x = x + \text{Attn}(\text{Norm}(x))
x = x + \text{FFN}(\text{Norm}(x))

Two serial operations per layer. Used by: Llama, Mistral, Qwen.

Parallel (PaLM-style)

x = x + \text{Attn}(\text{Norm}(x)) + \text{FFN}(\text{Norm}(x))

Attention and FFN run simultaneously. ~15% faster training. Used by: PaLM, GPT-J.

Time: ~2 min. "Parallel attention + FFN is a simple optimization: instead of waiting for attention to finish before starting FFN, run both at the same time and add the results. Slightly worse quality in theory, but the speed benefit is worth it at scale."

Positional Encoding: ALiBi

Theory

ALiBi (Attention with Linear Biases) adds a linear distance penalty to attention scores:

\text{softmax}\!\left(\frac{QK^T}{\sqrt{d_k}} - m \cdot |i - j|\right)
m = head-specific slope, |i-j| = distance between positions. Farther tokens get lower attention.

RoPE

Rotates embeddings. Relative position encoding. Dominant choice. Requires NTK-aware scaling for context extension.

ALiBi

Linear bias on attention. No learned parameters. Better extrapolation to unseen lengths. Used by MPT, BLOOM.

Time: ~2 min. "RoPE won the popularity contest — it's used by Llama, Qwen, Mistral, DeepSeek. ALiBi is simpler and extrapolates better, but is less common. Both solve the same problem: encoding position information without absolute position embeddings."

Chain-of-Thought (CoT) Prompting

Foundations

Ask the model to show its reasoning step by step before answering:

Without CoT

Q: A shop has 23 apples. 6 are sold, then 12 more arrive. How many?
A: 29

With CoT

Q: ...Let's think step by step.
A: Start with 23. Sell 6: 23-6=17. Add 12: 17+12=29. Answer: 29
Why it works: Generating intermediate steps extends the model's effective compute — each reasoning token is an additional computation step. The model essentially uses its own output as a scratchpad.
Time: ~3 min. Live demo opportunity: show a math problem where the model fails without CoT but succeeds with "Let's think step by step." Key point: "CoT doesn't teach the model anything new — it gives it more time to think by generating intermediate tokens."

CoT as Implicit Computation

Advanced

Each token generated in a chain-of-thought is an additional forward pass — additional compute:

\text{Effective compute} = \text{FLOPs/token} \times \text{num\_reasoning\_tokens}
More reasoning tokens = more compute = better answers (up to a point)

Zero-shot CoT

"Let's think step by step" — simple trigger, significant improvement on reasoning tasks

Few-shot CoT

Provide examples WITH reasoning chains — teaches the model your preferred reasoning style

Reasoning models (o1, R1): These are models trained via RL to generate long CoT automatically. You don't need to prompt "think step by step" — the model does it internally, sometimes generating thousands of reasoning tokens before answering.
Time: ~2 min. For experts: "This is the theoretical basis for test-time compute scaling. Each reasoning token adds ~2N FLOPs of compute. A model that generates 1000 reasoning tokens before answering effectively does 1000x more computation than one that answers directly." C1 — Why RAG?

Foundations

LLMs have three fundamental problems that RAG solves:

Knowledge Cutoff

Models stop learning at training time. They don't know what happened yesterday.

Hallucination

Models generate plausible but incorrect information. No way to verify without external sources.

No Source Attribution

Even when correct, models can't point to where they learned something. No citations, no audit trail.

ColBERT: Late Interaction

Advanced

ColBERT is a middle ground between bi-encoders and cross-encoders:

Bi-encoder: One vector per document. Fast but loses detail.
Cross-encoder: Full attention per pair. Accurate but slow.
ColBERT: One vector per TOKEN. Compute token-level similarity, then aggregate. Fast AND accurate.

Incremental Indexing

# Track document versions
class IncrementalIndexer:
    def update(self, new_docs: list, updated_docs: list, deleted_ids: list):
        # Add new documents
        for doc in new_docs:
            chunks = self.chunk(doc)
            embeddings = self.embed(chunks)
            self.vector_db.upsert(chunks, embeddings)

        # Update changed documents
        for doc in updated_docs:
            self.vector_db.delete(doc.id)  # Remove old chunks
            chunks = self.chunk(doc)
            embeddings = self.embed(chunks)
            self.vector_db.upsert(chunks, embeddings)

        # Remove deleted documents
        self.vector_db.delete(deleted_ids)

D2 — ReAct Architecture

Foundations

ReAct (Reasoning + Acting) is the most widely used agent architecture:

Thought: "The user asked about Singapore's Q3 2025 GDP. I need to search for this data."
Action: search("Singapore GDP Q3 2025 growth rate")
Observation: "MAS reports: GDP grew 3.2% YoY in Q3 2025..."
Thought: "I have the data. I should also check the sector breakdown."
Action: search("Singapore GDP sector breakdown Q3 2025")
Observation: "Manufacturing grew 5.1%, services grew 2.8%..."
Thought: "I have enough information to answer comprehensively."
Final Answer: "Singapore's GDP grew 3.2% in Q3 2025, driven by manufacturing (5.1%)..."
Time: ~3 min. Walk through each step. "The key insight: the model explicitly writes out its reasoning before each action. This makes it debuggable — you can see WHY it made each decision." Ask: "What would happen if the first search returned no results?" Expected: "The model would try a different search query."

Plan-and-Execute Architecture

Theory

ReAct (interleaved)

  • Think → Act → Observe → Think
  • Plan emerges step by step
  • Flexible: adapts to surprises
  • Expensive: frontier model at every step

Plan-and-Execute (separated)

  • Phase 1: Planner creates full plan
  • Phase 2: Executor runs each step
  • Planner: expensive frontier model
  • Executor: cheap, fast model
  • Cost reduction: up to 90%
Heterogeneous architectures: Use expensive frontier models for reasoning and planning, mid-tier models for standard tasks, and small models for high-frequency execution. This is how enterprises manage agent costs in 2026.
Time: ~2 min. "Plan-and-Execute is how you make agents affordable. The planner thinks hard once (using Claude Opus or GPT-5), then the executor follows the plan using a cheap model (Llama 8B or Claude Haiku). You get frontier-quality planning with budget-friendly execution."

Reflexion Architecture

Theory

Reflexion adds self-critique to the agent loop:

Attempt
→
Evaluate
→
Reflect
→
Retry
Attempt: Agent tries to solve the problem
Evaluate: Check if the solution is correct (test, verify, validate)
Reflect: "My approach failed because I forgot to handle edge case X."
Retry: Agent tries again with the reflection in memory
Key insight: The reflection is stored in episodic memory. The agent doesn't repeat mistakes because it remembers what went wrong. No gradient updates needed — just verbal self-critique.
Time: ~2 min. "Reflexion is like a student who gets feedback on a test, writes down what they got wrong and why, and uses those notes on the next attempt." For experts: "Reflexion achieves significant improvement on HumanEval coding benchmarks by storing failed attempts and their reflections."

LATS: Language Agent Tree Search

Advanced

LATS combines Monte Carlo Tree Search (MCTS) with LLM agents:

Selection: Choose the most promising unexplored action path
Expansion: Generate possible next actions
Evaluation: Score each path's quality (LLM evaluator or environment feedback)
Backpropagation: Update path quality estimates based on outcomes
Think AlphaGo for agents: Instead of exploring one path (ReAct) or planning one path (Plan-Execute), LATS explores a TREE of possible action sequences, using the same search algorithm that powers game-playing AI.
Time: ~2 min. For experts: "LATS brings the exploration-exploitation trade-off from RL to agent systems. It's particularly powerful for tasks where there are many possible action sequences and the cost of failure is high." Beginners: "Think of it as the agent considering multiple strategies simultaneously, like a chess player thinking several moves ahead."

Attention Mechanism: Step-by-Step Review

Theory

Quick review of self-attention from M8 — the core of every LLM:

\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^T}{\sqrt{d_k}}\right)V
Q = queries (what am I looking for?), K = keys (what do I contain?), V = values (what do I contribute?)
1. Each token generates Q, K, V vectors from its embedding
2. Compute attention scores: Q dot K for all token pairs
3. Scale by sqrt(d_k) to prevent softmax saturation
4. Apply softmax to get attention weights (sum to 1)
5. Multiply weights by V to get the output for each token
Time: ~2 min. Quick M8 review. "This is the engine inside every LLM. When the model decides that 'capital' is relevant to 'Singapore,' it's because the Q vector of 'capital' has high dot product with the K vector of 'Singapore.'"

Multi-Head Attention: Why Multiple Heads?

Theory
\text{MultiHead}(Q, K, V) = \text{Concat}(\text{head}_1, \ldots, \text{head}_h)W^O
h = number of heads (typically 32-128). Each head learns a different attention pattern.

Head 1

Might learn syntax: which words modify which

Head 2

Might learn coreference: "it" refers to "Singapore"

Head 3

Might learn proximity: nearby words matter more

Key insight: Multiple heads let the model attend to different types of relationships simultaneously. One head for syntax, another for semantics, another for position. This is why multi-head attention outperforms single-head.
Time: ~2 min. "Think of each attention head as a different lens. One lens looks for grammatical relationships. Another looks for semantic similarity. Another looks for long-range dependencies. Together, they give the model a rich understanding of the text."

Why Scale by sqrt(d_k)?

Theory

Without scaling, large dot products saturate the softmax and kill gradients:

Without scaling (d_k = 128):
Dot products range: [-100, 100]
softmax([100, -100]) = [1.0, 0.0] ← saturated, gradients vanish

With scaling (divide by sqrt(128) = 11.3):
Scaled range: [-8.8, 8.8]
softmax([8.8, -8.8]) = [0.9999, 0.0001] ← still peaked but gradient flows
The math: If Q and K entries are independent with mean 0 and variance 1, then the dot product QK has variance d_k. Dividing by sqrt(d_k) normalizes the variance back to 1, keeping softmax in its useful range.
Time: ~2 min. "This is a detail from M8 worth revisiting. Without scaling, attention becomes binary — either full attention or zero attention. With scaling, attention can be graduated — some tokens get 70% attention, others 20%. This graded attention is what makes transformers powerful."

BPE Tokenization: Detailed Example

Foundations

Watch BPE build a vocabulary step by step:

Corpus: "low lower lowest"

Step 0 (characters): l, o, w, e, r, s, t
Step 1: Merge (l,o) → "lo" — most frequent pair
Step 2: Merge (lo,w) → "low" — "low" appears in all three words
Step 3: Merge (e,r) → "er" — appears in "lower"
Step 4: Merge (e,s) → "es" — appears in "lowest"
Step 5: Merge (es,t) → "est" — appears in "lowest"

Final tokens: "low" | "low" + "er" | "low" + "est"
Why BPE matters: It creates tokens for common words ("the", "and") and sub-word units for rare words. No word is "unknown" — just broken into pieces. Vocab sizes of 32K-128K cover virtually all text.
Time: ~2 min. "BPE is how every modern LLM tokenizer works. The key idea: start with characters, iteratively merge the most common pairs, until you reach your target vocabulary size." Live demo: "Use tiktoken or the Hugging Face tokenizer to tokenize a sentence and count tokens."

The Chinchilla Paper: Impact on LLM Design

Theory

Before Chinchilla (2022)

  • GPT-3: 175B params, 300B tokens

After Chinchilla

Embedding Similarity: Visual Intuition

Foundations

Embeddings create a semantic space where distance = meaning:

Close together (high similarity):
"Singapore GDP growth" ↔ "Economic expansion in SG" (cosine sim: 0.92)
"HDB flat resale" ↔ "Public housing secondary market" (cosine sim: 0.88)

Far apart (low similarity):
"Singapore GDP growth" ↔ "Recipe for chicken rice" (cosine sim: 0.12)
"MAS monetary policy" ↔ "Football match results" (cosine sim: 0.08)
Cosine similarity range: 1.0 = identical meaning, 0.0 = completely unrelated, -1.0 = opposite meaning. In practice, most similar documents score 0.7-0.9 and irrelevant ones score 0.0-0.3.
Time: ~2 min. "Think of embeddings as GPS coordinates for meaning. Sentences about economics are clustered together. Sentences about cooking are clustered elsewhere. When you search 'Singapore GDP,' the system finds the nearest points in meaning-space."

Vector Search: k-Nearest Neighbors

Foundations

Given a query embedding, find the k most similar document embeddings:

\text{top-k} = \text{argmax}_{k} \frac{q \cdot d_i}{\|q\| \|d_i\|}
Find k documents with highest cosine similarity to the query
Query: "What is the max LTV for HDB loans?" → embed → [0.12, -0.34, ...]
Search: Find top-5 chunks nearest to this vector
Result 1: "MAS Notice 645 stipulates a maximum LTV of 80%..." (sim: 0.91)
Result 2: "Borrowers must maintain adequate CPF..." (sim: 0.84)
Result 3: "The loan-to-value ratio is determined by..." (sim: 0.79)
Time: ~2 min. "This is the core retrieval operation. Your query becomes a vector, and we find the most similar document vectors. With HNSW indexing, this takes milliseconds even across millions of documents."
