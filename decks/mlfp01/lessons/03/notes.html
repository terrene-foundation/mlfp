Speaker Notes — Lesson 1.3 Functions and Aggregation

Total time: ~55 minutes  ·  14 slides  ·  ~4 min per slide

This lesson combines two ideas: functions (Python abstraction) and group_by (Polars aggregation). The learners are at their peak confidence from Lesson 1.2 — don't let them drift. The group_by pattern is the most important single idea in the entire module.

Slide 1~2 minTitle

Functions and Aggregation

Open with: "Yesterday you filtered one town at a time. Today we make that obsolete. By the end of this lesson you will produce a 26-row summary of every HDB town with one call, and you'll have written your first reusable Python functions."

→ "The productivity picture."
Slide 2~3 minBefore/after diagram

The productivity jump

Walk the left side first — 26 filter calls in a red block. Make them feel the pain. Then the right side: one group_by call. Ask them which one they'd want to maintain.

Tell them: "If you finish this module and only remember one pattern, it should be group_by plus agg. It's the single most common Polars pattern in real analysis code."

For beginners: "You don't need to understand how the aggregation works internally. Just know the shape: give it a group column, give it what to compute, get back a table."
→ "First tool: functions."
Slide 3~4 minFunction basics

Functions: reusable logic

Walk the example. Run it live. Call format_sgd a few times with different values. Make the point that the function doesn't know which number you'll pass — it's a tiny machine that works on any float.

The type hints: explain they're documentation, not enforcement. Python will not raise an error if you pass a string to a function annotated with : float. They're for humans and IDEs.

For experts: "mypy and pyright enforce these statically — a whole parallel universe of type-safe Python. We won't cover that here, but know it exists."
→ "Let's label every part."
Slide 4~3 mindef anatomy diagram

Anatomy of def

Walk each labelled part. Point at def — "the keyword that tells Python 'a function definition follows'". Point at the name — "how you call the function later". Point at the parameter — "the input". Point at the type hints — "documentation". Point at the return type — "what kind of value comes back".

Then the body: "indented under the signature, run when the function is called, ends with return to hand a value back to the caller."

→ "Python collections you'll use constantly."
Slide 5~3 minLists, dicts, for

Lists, dicts, for loops

Walk all three quickly. Lists are ordered — access by integer index. Dicts are keyed — access by the key you wrote. For loops walk any sequence.

Make this point: "We'll use for loops for printing and reporting, not for filtering. Polars has group_by for filtering; loops are for doing things that don't fit into a DataFrame."

For beginners: "Lists are like numbered lockers; dicts are like named lockers. Both hold multiple items — but you open them differently."
→ "The main event."
Slide 6~5 mingroup_by diagram

group_by + agg: the pattern

Walk the diagram left to right. Input DataFrame with mixed towns on the left. Groups in the middle — each one a bucket of rows sharing a town value. Output table on the right — one row per group.

Say: "Polars scans the input once, hashes each town, drops the row into the right bucket. For every bucket it then computes the aggregations you asked for. The output has one row per unique town — 26 rows for 26 towns."

For experts: "Same algorithm as SQL's GROUP BY — hash aggregation. Polars runs it in parallel across cores over columnar memory, which is why it's 10–100× faster than the equivalent pandas."
→ "The code."
Slide 7~4 mingroup_by code

group_by in code

Run it live. Read it aloud: "Group hdb by town. For each group, compute four aggregations: count, mean, median, std. Alias each one. Sort by median descending."

Print district_stats.head(5) and walk the output. "Look — 26 rows, five columns (town plus four stats), already sorted. The most expensive towns rise to the top."

Emphasise the .alias calls. Without them you'd see columns named by guesswork — and with multiple aggregations on the same column, Polars would number them resale_price, resale_price_2, which nobody wants to read.

→ "Multi-key grouping."
Slide 8~3 minMulti-key group_by

Multi-key grouping

Show the example. "Group by town and flat type. Every unique combination becomes one row. If a town has all 7 flat types, it gets 7 rows in the output."

Tell them to expect roughly 180 rows — 26 towns × 7 flat types, minus some combinations that don't appear.

For beginners: "Think pivot table — a spreadsheet summary with rows and columns for two categories. Polars does the same thing with one method call."
→ "Now the statistics behind the scenes."
Slide 9~3 minMean vs median

Measures of centre: mean vs median

Walk the two formulas. Mean is arithmetic average. Median is the middle value of sorted data. Easy to compute, critical to distinguish.

The rule of thumb: "Mean is pulled by outliers; median is not." Give the concrete Singapore example: one billion-dollar HDB in a district of 300 flats moves the mean by several thousand dollars but leaves the median unchanged.

For beginners: "If Jeff Bezos walks into your coffee shop, the average net worth of the customers becomes a billion dollars. The median stays what it was. That's why median is the 'typical' value."
→ "And spread — IQR."
Slide 10~3 minIQR

Measure of spread: IQR

Walk the definition. Q1 at 25%, Q3 at 75%, IQR = Q3 - Q1. The spread of the middle half — robust to outliers.

Show the Polars code. Point out that quantile(0.25) gives you Q1 directly — no need to sort or index manually.

→ "Comparing spread across groups."
Slide 11~3 minCoefficient of variation

🔵 Theory: Coefficient of variation

Walk the formula. "Standard deviation divided by mean, times 100. Expressed as a percentage. Unitless, so you can compare across groups with very different means."

Concrete example: "Bishan and Jurong both have std of 80k. Bishan's mean is 700k, so CV ≈ 11%. Jurong's mean is 400k, so CV ≈ 20%. Jurong's prices are relatively more spread out — more mixed housing stock."

For experts: "CV is also the inverse of signal-to-noise ratio. High CV = noisy relative to signal = harder to model. You'll see this again in Module 3 when we discuss heteroscedasticity."
→ "Derived aggregates."
Slide 12~3 minDerived aggregates

Derived aggregates with with_columns

Show the pattern: first aggregate, then derive. Point out that after agg, the result is just a DataFrame — you can chain with_columns, filter, sort on it the same way.

This is an important conceptual point: "There's no 'aggregated DataFrame' special type. It's still a DataFrame. Everything you learned in Lesson 1.2 still applies."

→ "Putting it all together — a for loop report."
Slide 13~3 minfor + function

Iterating with for + functions

Walk the code. The helper function takes a dict (one row) and produces a formatted string. The for loop walks the top 10 rows and prints each.

Point out iter_rows(named=True) — "this yields each row as a dict, so you can access fields by name. Without named=True you'd get tuples and have to remember the column order."

Run it live. Admire the aligned output. Point out the format specifiers: :<20 means "left-aligned in a 20-char column", :>12 means "right-aligned in a 12-char column".

→ "Mistakes to avoid."
Slide 14~3 minMistakes + reflection

Three mistakes to avoid

Walk each one. The hardcoding mistake is about turning scripts into libraries — functions without parameters are just namespaces.

The alias mistake is about output readability — Polars will let you skip alias but the result will be ugly. Make them see it live: run an aggregation without alias, show the auto-names.

The loop-vs-group_by mistake is the big one. "If you ever find yourself writing for town in towns: subset = df.filter(...), stop. There's a group_by that does the same thing 50× faster."

Close by teasing 1.4: "Today we aggregated within one table. Tomorrow we combine data from three tables — HDB transactions, MRT stations, schools. You'll meet join, and you'll meet Python's if/elif/else for code-level decisions."

→ End of Lesson 1.3.
