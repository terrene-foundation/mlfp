Gini Importance (MDI) — Why It Lies

Theory

Mean Decrease in Impurity sums the weighted impurity reduction at every node that splits on a given feature:

\text{Importance}(x_j) = \sum_{t \in T_j} \frac{n_t}{n} \left[ \text{Impurity}(t) - \frac{n_{tL}}{n_t}\text{Impurity}(t_L) - \frac{n_{tR}}{n_t}\text{Impurity}(t_R) \right]
Sum over all nodes t that split on feature j; nt = samples at node t
The bias: Features with many unique values (high cardinality) create more potential split points. More split points = more chances to reduce impurity = artificially inflated importance. This is a mathematical artefact, not a real signal.
Time: ~2 min. Show the formula but focus on the intuition. A feature like "customer_id" with 10,000 unique values will appear important because it can perfectly split any subset — but it is pure noise.

Permutation Importance

Theory

Randomly shuffle one feature’s values. If the model’s accuracy drops significantly, that feature matters.

\text{PI}(x_j) = \text{Score}_{\text{original}} - \frac{1}{K}\sum_{k=1}^{K}\text{Score}_{\text{permuted}}^{(k)}
K = number of permutations; Score = accuracy, AUC, MSE, etc.
  • Model-agnostic: works with any model (trees, neural nets, linear models)
  • Unbiased: does not favour high-cardinality features
  • P-value: repeat K times, compute standard deviation, test whether drop is statistically significant
Caveat: With correlated features, permuting one feature may not drop accuracy because the correlated feature compensates. Result: both features appear unimportant when both are important.
Time: ~2 min. This is the recommended default method. Mention that K=10-30 permutations is typical. The p-value interpretation: under the null hypothesis that feature j is irrelevant, the score drop is zero.

Drop-Column Importance

Theory

The most honest approach: retrain the model without each feature and measure the performance change.

\text{DCI}(x_j) = \text{Score}_{\text{full model}} - \text{Score}_{\text{model without } x_j}

Advantages

  • Handles correlated features correctly
  • Measures true causal importance
  • No bias from any source

Disadvantages

  • Requires retraining p+1 times
  • Computationally prohibitive for large models
  • Can be noisy for small datasets
Time: ~2 min. Gold standard but expensive. In practice, use permutation importance for screening, then drop-column for the top 10-20 features to confirm. This is the recommended workflow.

SHAP Values — The Intuition

Foundations

Imagine a team of players wins a game. How do you fairly divide the prize money?

  • Each player contributed differently
  • Some players are only useful when paired with others
  • The fair share depends on all possible team combinations
SHAP values answer: "How much did each feature contribute to moving this prediction away from the average?" Every feature gets a fair share of the prediction, like every player gets a fair share of the prize.
Time: ~3 min. The cooperative game theory analogy is powerful for beginners. Players = features. Prize = prediction minus average prediction. SHAP distributes the prize fairly. Do NOT jump to math yet.

Shapley Values — From Game Theory

Theory

Lloyd Shapley (1953, Nobel Prize 2012) defined a unique fair allocation in cooperative games:

\phi_j = \sum_{S \subseteq \{1,\ldots,p\} \setminus \{j\}} \frac{|S|!\;(p - |S| - 1)!}{p!} \left[ v(S \cup \{j\}) - v(S) \right]
φj = Shapley value for feature j; v(S) = model prediction using only features in S; p = total features

For each feature j, consider every possible subset of other features. Measure the marginal contribution of adding feature j to that subset. Weight by the number of ways this ordering could arise.

Time: ~3 min. Walk through each piece of the formula. S = a "coalition" of features already present. v(S union j) - v(S) = the value feature j adds to coalition S. The weighting factor counts the orderings.

Why Shapley? — The Four Axioms

Theory
1. Efficiency: All feature contributions sum to the total prediction minus the average prediction. \sum_{j=1}^{p} \phi_j = f(x) - E[f(x)]
2. Symmetry: If two features contribute equally in every coalition, they receive the same Shapley value.
3. Dummy (Null Player): If a feature never changes the prediction in any coalition, its Shapley value is zero.
4. Linearity (Additivity): For a combined model f = f1 + f2, the SHAP values are the sum of the individual SHAP values.
Uniqueness theorem: Shapley values are the only allocation that satisfies all four axioms simultaneously. No other method is as theoretically justified.
Time: ~3 min. The uniqueness is the key selling point. Other feature importance methods satisfy some axioms but not all. For experts: mention that this is a characterisation theorem, not just a definition.

The Computational Problem

Theory

Exact Shapley values require summing over all subsets of features:

2^p \text{ subsets for } p \text{ features}
10 features = 1,024 subsets. 20 features = 1,048,576. 100 features = 1.27 × 1030.

This is intractable for any real-world model. We need approximations:

Time: ~2 min. The exponential complexity is why we need TreeSHAP and KernelSHAP. TreeSHAP exploits tree structure for polynomial time. KernelSHAP uses sampling.

TreeSHAP — Exact SHAP for Trees

Theory

TreeSHAP (Lundberg et al., 2020) exploits the tree structure to compute exact Shapley values in polynomial time.

Key Insight

In a tree, the prediction path depends on a subset of features. We can recursively track which features are "in" or "out" of each coalition as we traverse the tree.

Two Variants

  • Interventional: break dependencies between features (what if we set xj independently?)
  • Observational: respect feature correlations (what if we condition on xj?)
Complexity: O(TLD2) where T = number of trees, L = max leaves, D = max depth. For a 500-tree XGBoost with depth 6, this is fast enough for real-time explanations.
Time: ~2 min. TreeSHAP is why SHAP became practical. Before it, SHAP was theoretically beautiful but too slow. The interventional vs observational distinction matters when features are correlated — use interventional by default.

KernelSHAP — Model-Agnostic Approximation

Theory

KernelSHAP frames SHAP estimation as weighted linear regression:

Step 1: Sample coalitions

Randomly sample subsets S of features. For each subset, create a "masked" input where missing features are replaced by background values.

Step 2: Weight by Shapley kernel
\pi(S) = \frac{p - 1}{\binom{p}{|S|} \cdot |S| \cdot (p - |S|)}

Coalitions with very few or very many features get the highest weight — they are most informative about marginal contributions.

Step 3: Solve weighted least squares

Fit a linear model from coalition indicators to predictions, weighted by π(S). The coefficients are the SHAP values.

Time: ~3 min. KernelSHAP is slower than TreeSHAP (~12.5s/instance vs 3.1s) but works with ANY model — neural nets, SVMs, anything. Accuracy is ~94% fidelity with enough samples.

SHAP Interaction Values

Advanced

SHAP can decompose each feature’s effect into main effect + interaction effects:

\phi_{ij} = \sum_{S \subseteq \{1,\ldots,p\} \setminus \{i,j\}} \frac{|S|!\;(p - |S| - 2)!}{2(p-1)!} \nabla_{ij}(S)
∇ij(S) = discrete second-order derivative: the effect of adding feature j depends on whether feature i is present

The interaction matrix is symmetric: \phi_{ij} = \phi_{ji}. Diagonal entries are main effects.

Use case: Discover that "age" and "income" interact: high income reduces risk only for applicants over 30. Without interaction values, you would miss this conditional relationship.
Time: ~2 min. Advanced topic. TreeSHAP can compute interaction values efficiently. For experts: the interaction matrix has p^2 entries, but TreeSHAP computes them in O(TLMD^2) time.

From Local to Global — Aggregating SHAP

Theory

SHAP values are local (per prediction). To get global importance, aggregate:

\text{Global importance}(x_j) = \frac{1}{n}\sum_{i=1}^{n} |\phi_j^{(i)}|
Mean absolute SHAP value across all samples
Advantage over permutation importance: SHAP-based global importance is consistent (a feature with higher SHAP values always ranks higher) and additive (global values sum to mean |prediction - average|). Permutation importance has neither property.
Time: ~2 min. This is why SHAP is preferred over permutation importance when you can afford the computation. The consistency property was proved by Lundberg and Lee (2017).

Multioutput SHAP

Advanced

For multi-class classification or multi-output regression, SHAP values become a 3D tensor:

\Phi \in \mathbb{R}^{n \times p \times c}
n samples × p features × c classes/outputs

Each feature has a separate SHAP value for each output class. This reveals:

  • A feature may push toward class A while pushing away from class B
  • The same feature can have opposite effects on different outputs
  • Interaction effects can be class-specific
Time: ~2 min. Practical application: in a credit model with outcomes {approve, review, deny}, a feature might push toward "review" specifically while having no effect on "deny." This nuance is lost with single-output importance methods.

LIME — Local Interpretable Model-agnostic Explanations

Foundations

LIME answers the same question as SHAP — "why this prediction?" — but uses a completely different approach.

The idea: Even if the model is a complex black box, its behaviour near this particular point might be simple enough to approximate with a straight line. LIME fits a simple model (like linear regression) to the model’s behaviour in the neighbourhood of the prediction.
Original model f(x)
→
Perturb x to get neighbours
→
Fit simple model g(x) locally
→
Read coefficients of g
Time: ~3 min. LIME is the main alternative to SHAP for local explanations. The metaphor: if you zoom into any curve closely enough, it looks like a straight line. LIME zooms into the prediction and draws that straight line.

LIME — The Algorithm

Theory
Step 1: Perturb

Generate perturbed samples around x by randomly toggling features on/off or adding noise.

Step 2: Predict

Use the black-box model to predict on all perturbed samples.

Step 3: Weight

Weight each perturbed sample by proximity to x using an exponential kernel:

\pi_x(z) = \exp\left(-\frac{d(x, z)^2}{\sigma^2}\right)
Step 4: Fit

Solve weighted linear regression: \min_{g \in G} \sum_z \pi_x(z) (f(z) - g(z))^2 + \Omega(g)

Time: ~3 min. The kernel width sigma is crucial — too small and you only see noise, too large and you approximate the global model, not the local behaviour. This is LIME's main weakness: results can change with sigma.

LIME — Limitations

Theory

Strengths

  • Model-agnostic (any model)
  • Works for text, images, tabular
  • Intuitive output (linear coefficients)
  • Fast (no need to enumerate subsets)

Weaknesses

  • Instability: different runs can give different explanations
  • Kernel width: choice of σ is arbitrary, changes results
  • Not additive: does not satisfy the efficiency axiom
  • Sampling: may miss important regions of feature space
SHAP vs LIME: SHAP has stronger theoretical guarantees (4 axioms, uniqueness). LIME is faster and more flexible. For regulated applications, prefer SHAP. For quick debugging, LIME is acceptable.
Time: ~2 min. The instability point is critical. If you run LIME twice on the same prediction and get different explanations, how can you trust it? SHAP does not have this problem because it computes a deterministic value (with TreeSHAP) or converges with more samples (KernelSHAP).

Partial Dependence Plots (PDP)

Foundations

PDP shows the average effect of one feature on the prediction, holding all other features constant.

Analogy: "If I change only the temperature setting on the thermostat, what happens to the electricity bill on average?" The PDP plots thermostat setting (x-axis) vs average bill (y-axis), averaging over all the other factors (insulation, household size, time of year).
\hat{f}_S(x_S) = E_{x_C}\!\left[\hat{f}(x_S, x_C)\right] = \frac{1}{n}\sum_{i=1}^{n}\hat{f}(x_S, x_C^{(i)})
xS = feature of interest; xC = all other features; average over all observed values of xC
Time: ~3 min. PDP is the oldest and most widely used global explanation method. Start with the thermostat analogy for beginners. The formula is just an average: plug in a specific value for the feature of interest, then average the model's predictions over all rows in the dataset.

PDP — 2D Plots and Limitations

Theory

2D PDPs

Show the joint effect of two features as a heatmap:

\hat{f}_{S}(x_{S_1}, x_{S_2}) = \frac{1}{n}\sum_{i=1}^{n}\hat{f}(x_{S_1}, x_{S_2}, x_C^{(i)})

Useful for discovering interaction effects visually.

Critical Limitation

PDP assumes features are independent.

When computing the average, PDP creates impossible combinations: e.g., a 20-year-old with 30 years of work experience.

This extrapolation into impossible regions distorts the marginal effect.

Rule: If your features are correlated, PDP can be misleading. Use ALE plots instead (Slide 39).
Time: ~2 min. The independence assumption is the single biggest criticism of PDP. In real datasets, features are almost always correlated. This is why ALE was invented.

Individual Conditional Expectation (ICE)

Foundations

PDP shows the average. But the average can hide important differences between individuals.

ICE plots draw one line per observation. Instead of averaging, you see every individual’s response to the feature change.

PDP

One thick line: the average effect

Tells you: "on average, increasing age reduces risk"

ICE

Many thin lines: one per data point

Reveals: "for most people increasing age reduces risk, but for a subgroup with low income, the effect reverses"

Time: ~2 min. ICE reveals heterogeneity. The PDP is just the average of all ICE curves. If ICE curves cross or diverge, the PDP is hiding important structure. Always plot ICE alongside PDP.

Centered ICE (c-ICE)

Theory

Raw ICE curves start at different baseline predictions, making visual comparison difficult. Centered ICE anchors all curves to start at zero:

\hat{f}_{\text{c-ICE}}^{(i)}(x_S) = \hat{f}^{(i)}(x_S) - \hat{f}^{(i)}(x_S^{*})
Subtract each curve’s value at a reference point xS*
  • All curves start at (xS*, 0)
  • Differences in shape become visible
  • If all c-ICE curves are parallel, there are no interactions
  • If c-ICE curves diverge, there are interactions with other features
Time: ~2 min. c-ICE is a simple but powerful diagnostic. Parallel curves = the feature effect is the same for everyone. Diverging curves = the effect depends on other features (interaction). This is a quick way to discover interactions without computing SHAP interaction values.

Accumulated Local Effects (ALE)

Foundations

ALE solves PDP’s correlation problem by computing effects locally instead of globally.

The difference:
• PDP asks: "What if I set age to 30 for everyone?" (creates impossible combos)
• ALE asks: "Among people whose age is around 30, how does a tiny increase in age change the prediction?" (stays in observed data)

ALE never extrapolates into impossible regions of feature space because it uses differences rather than absolute predictions.

Time: ~2 min. The key insight for beginners: PDP teleports people into impossible situations (age 20 with 30 years experience). ALE only looks at people who actually exist near each value. Friedman (2001) introduced PDP; Apley and Zhu (2020) introduced ALE to fix it.

ALE — How It Works

Theory
Step 1: Divide the feature range into bins

Create intervals [z0, z1], [z1, z2], ... along feature j.

Step 2: Compute local effects within each bin
\hat{f}_{j,\text{ALE}}(x) = \sum_{k=1}^{k_j(x)} \frac{1}{n_j(k)} \sum_{i: x_j^{(i)} \in N_j(k)} \left[\hat{f}(z_k, x_{-j}^{(i)}) - \hat{f}(z_{k-1}, x_{-j}^{(i)})\right]
Step 3: Center the curve

Subtract the mean so the ALE plot is centred at zero: \hat{f}_{j,\text{ALE}}(x) - E[\hat{f}_{j,\text{ALE}}]

Key: The difference-based approach (zk vs zk-1) means ALE never needs to evaluate the model at unrealistic feature combinations. This is why ALE is unbiased for correlated features.
Time: ~3 min. The difference-based approach is the key innovation. PDP uses absolute values (creating impossible combos). ALE uses differences within small bins (staying near observed data). The trade-off: ALE needs enough data in each bin.

Second-Order ALE

Advanced

ALE can be extended to show interaction effects between two features:

\hat{f}_{jk,\text{ALE}}(x_j, x_k) = \sum_{l_j=1}^{K_j}\sum_{l_k=1}^{K_k} \frac{1}{n_{jk}(l_j, l_k)} \sum_{i \in N_{jk}} \Delta_{jk}^{(i)}
Δjk = the mixed partial effect: does the effect of feature j change when feature k changes?

Second-order ALE reveals pure interactions not captured by the main effects. Unlike 2D PDP, it handles correlated features correctly.

Time: ~2 min. Advanced topic. The mixed partial difference is like a numerical second derivative. If it is zero, the features do not interact. If non-zero, the effect of one feature depends on the value of the other. This is the gold standard for interaction detection in correlated data.

Counterfactual Explanations

Foundations

"Your loan was denied. What would need to change for it to be approved?"

Counterfactual: "If your income were $5,000 higher and your credit utilisation were below 30%, the model would approve your loan."

Unlike SHAP (which says "income contributed -0.15 to the decision"), counterfactuals give actionable advice.

Wachter et al. (2017) formalised this as an optimisation problem: find the smallest change to the input that flips the decision.

Time: ~3 min. Counterfactuals are unique because they answer "what should I DO?" not "what did the model SEE?" This matters for compliance (EU AI Act right to explanation) and customer experience. Ask: "Would you rather hear why you were denied, or what you can do to be approved?"

Counterfactual — The Optimisation Problem

Theory
\arg\min_{x'}\; \lambda \cdot (f(x') - y')^2 + d(x, x')
y' = desired outcome; d(x, x') = distance between original and counterfactual; λ = trade-off

Two competing objectives:

  • Validity: the counterfactual must actually flip the model’s decision — f(x') \approx y'
  • Proximity: the change should be as small as possible — minimise d(x, x')
Distance function matters: L1 distance encourages sparse changes (change few features). L2 encourages many small changes. Weighted distances can encode feature mutability (you can change income but not age).
Time: ~2 min. The distance function is where domain knowledge enters. Age is immutable, race is immutable, but income and credit utilisation are actionable. The distance function should reflect what the person can actually change.

Actionability Constraints

Theory

Not all changes are possible. Counterfactual explanations must respect actionability:

FeatureConstraintExample
AgeImmutable (can only increase)Cannot become younger
Race, genderImmutable (protected attribute)Must not suggest changing
EducationMonotone (can only increase)Cannot un-learn a degree
IncomeBounded rangeRealistic increase only
Credit scoreDependent on other featuresChanges as debt changes
Without actionability constraints, a counterfactual might suggest: "If you were 10 years younger and changed your ethnicity..." — which is useless and offensive.
Time: ~2 min. Actionability is what separates useful counterfactuals from mathematical curiosities. In production, every counterfactual should be reviewed against domain constraints before showing to users.

DiCE — Diverse Counterfactual Explanations

Theory

A single counterfactual might not be helpful. DiCE (Mothilal et al., 2020) generates multiple diverse counterfactuals:

\arg\min_{\{x'_1,\ldots,x'_k\}}\; \sum_{i=1}^{k}\left[\lambda_1 \cdot \ell(f(x'_i), y') + \lambda_2 \cdot d(x, x'_i)\right] - \lambda_3 \sum_{i \neq j} d(x'_i, x'_j)
Last term encourages diversity: counterfactuals should differ from each other

Instead of "change X", the user gets a menu of options:

  • Option A: increase income by $5,000
  • Option B: reduce credit utilisation to 25% and increase credit history by 1 year
  • Option C: pay off personal loan
Time: ~2 min. DiCE is the state-of-the-art for counterfactual generation. The diversity term is crucial — without it, all counterfactuals converge to the same suggestion. DiCE-Extended (2025) adds robustness via multi-objective optimisation.

Anchors — If-Then Rules as Explanations

Advanced

Anchors (Ribeiro et al., 2018) produce explanations as if-then rules with coverage guarantees:

Example anchor: "IF income > $60K AND employment = full-time, THEN loan approved with 95% precision."

This rule holds for all similar applicants (high coverage). The precision is the proportion of the rule's domain where the prediction matches.
E_{D(z|A)}[\mathbb{1}_{f(z) = f(x)}] \geq \tau
Anchor A covers a region D(z|A) where the prediction matches with probability ≥ τ

Anchors are more stable than LIME (same input always gives same rule) and more interpretable than SHAP for non-technical audiences.

Time: ~2 min. Anchors are particularly useful for compliance because they produce human-readable rules. The trade-off: anchors can oversimplify — the "if-then" format cannot capture continuous effects like SHAP values.

Concept-Based Explanations (TCAV)

Advanced

Testing with Concept Activation Vectors (Kim et al., 2018) explains models in terms of human-understandable concepts rather than raw features.

Example: Instead of "pixel 234 at position (12, 45) contributed +0.03", TCAV says: "The presence of stripes in the image increased the model's confidence in 'zebra' by 15%."

How it works:

  1. Define a concept (e.g., "striped texture") with example images
  2. Train a linear classifier to separate the concept in the model’s internal activations
  3. The direction (Concept Activation Vector) represents the concept in latent space
  4. Measure how sensitive the model’s output is to movement along this direction
Time: ~2 min. Advanced topic, mostly for deep learning models. TCAV is powerful for image and text models where raw feature explanations are meaningless. The key innovation: explanations in human concepts, not model internals.

Why Fairness Matters

Foundations

A model trained on historical data inherits the biases of that history.

Amazon (2018): An automated hiring tool penalised resumes that mentioned "women's" (e.g., "women's chess club") because the model learned from 10 years of male-dominated hiring decisions. The model was accurate — it predicted who would have been hired in the past. But it was not fair.

Fairness is not about making the model less accurate. It is about making sure accuracy is equally distributed across groups.

Time: ~3 min. The Amazon example is powerful because the model worked exactly as designed — it predicted historical hiring decisions. The problem was the data, not the algorithm. This is why fairness must be an explicit goal, not a side effect of accuracy.

Types of Bias

Foundations
Bias TypeSourceExample
HistoricalPast discrimination in training dataLoan approvals reflecting decades of redlining
MeasurementProxy variables for protected attributesPostcode as proxy for race
RepresentationUnder-representation of minorities in dataFacial recognition trained mostly on light-skinned faces
AggregationSingle model for heterogeneous groupsOne diabetes model for all ethnicities (ignoring HbA1c differences)
Key insight: Removing the protected attribute (race, gender) does not fix bias. The model learns proxies (postcode, name, school) that correlate with the attribute. This is called redundant encoding.
Time: ~3 min. Walk through each type. Singapore context: measurement bias is common when postcode correlates with ethnicity. Aggregation bias matters for healthcare models that ignore ethnic differences in baseline health metrics.

Fairness Definition 1: Demographic Parity

Theory
Demographic Parity (Statistical Parity): The probability of a positive prediction should be the same across groups.
P(\hat{Y} = 1 \mid A = 0) = P(\hat{Y} = 1 \mid A = 1)
A = protected attribute (e.g., gender); &hat;Y = model prediction

Advantages

  • Simple to measure and enforce
  • Ensures equal representation in outcomes

Problems

  • Does not consider actual qualification
  • May require approving unqualified applicants or rejecting qualified ones
  • Incompatible with other fairness criteria
Time: ~2 min. Demographic parity is the simplest fairness metric but also the most controversial. It says: "the same fraction of men and women should be approved." But if qualification rates genuinely differ, this forces the model to ignore real differences.

Fairness Definition 2: Equalized Odds

Theory
Equalized Odds (Hardt et al., 2016): The model should have the same true positive rate AND false positive rate across groups.
P(\hat{Y} = 1 \mid Y = y, A = 0) = P(\hat{Y} = 1 \mid Y = y, A = 1) \quad \forall\, y \in \{0, 1\}
For both Y=1 (true positive rate) and Y=0 (false positive rate), the rates must be equal across groups
Intuition: "Among people who actually repay their loans, the approval rate should be the same for both groups. Among people who actually default, the denial rate should be the same for both groups." This is about equal accuracy, not equal outcomes.
Time: ~2 min. Equalized odds is generally considered stronger than demographic parity because it conditions on the true label. It says: "if you deserve a positive outcome, you should get it regardless of group membership."

Fairness Definition 3: Calibration

Theory
Calibration (Predictive Parity): Among people who receive the same predicted probability, the actual outcome rate should be the same across groups.
P(Y = 1 \mid \hat{Y} = p, A = a) = p \quad \forall\, a, p
If the model says "70% chance of default", then 70% of people in BOTH groups should actually default
Intuition: "The model’s confidence should mean the same thing for everyone." A 70% risk score should mean 70% risk regardless of whether you are male or female.
Time: ~2 min. Calibration is especially important in healthcare and finance where the predicted probability is used for decision-making. A well-calibrated model is honest about its uncertainty for all groups.

Fairness Definition 4: Individual Fairness

Theory
Individual Fairness (Dwork et al., 2012): Similar individuals should receive similar predictions.
d_Y(\hat{f}(x), \hat{f}(x')) \leq L \cdot d_X(x, x')
The distance between predictions should be bounded by L times the distance between individuals

The challenge: defining "similar" requires a fair metric d_X that captures the relevant notion of similarity. This is often harder than the ML problem itself.

Time: ~2 min. Individual fairness is the strongest definition but the hardest to implement. The fair metric must encode what counts as "similar" from a fairness perspective — and that is a moral/legal question, not a technical one.

The Impossibility Theorem

Theory
Chouldechova (2017) / Kleinberg et al. (2016): You CANNOT simultaneously satisfy calibration, equalized odds, and demographic parity — except in two degenerate cases: (1) the model is perfect, or (2) the base rates are identical across groups.

When the base rate differs between groups (e.g., default rate differs by age group), any calibrated classifier must violate equalized odds, and vice versa.

\text{If } P(Y=1 | A=0) \neq P(Y=1 | A=1) \text{, then calibration } \implies \neg\text{equalized odds}
Practical implication: You must choose which fairness definition matters most for your application. There is no universally fair model. This is a policy decision, not a technical one.
Time: ~3 min. This is one of the most important results in ML fairness. It means that perfect fairness is mathematically impossible in most real-world settings. The choice of fairness criterion must be guided by the application, legal requirements, and ethical considerations — not by the data scientist alone.

Disparate Impact — The 80% Rule

Foundations

A practical legal test used in US employment law:

\text{Disparate Impact Ratio} = \frac{P(\hat{Y}=1 \mid A=\text{minority})}{P(\hat{Y}=1 \mid A=\text{majority})}
If this ratio is below 0.8 (80%), the model has adverse impact
Example: If 50% of male applicants are approved but only 35% of female applicants, the ratio is 35/50 = 0.70. This is below 0.80 and triggers a disparate impact finding.

The 80% rule is not a fairness definition — it is a screening tool. A model can pass the 80% rule while still being unfair by other criteria.

Time: ~2 min. The 80% rule comes from US EEOC guidelines (1978). It is widely used as a first-pass check. Singapore does not have an equivalent threshold, but MAS guidelines encourage similar monitoring for financial services models.

Bias Mitigation — Three Stages

Foundations
StageWhenMethods
Pre-processingBefore trainingReweighting, disparate impact remover, fair representation learning
In-processingDuring trainingAdversarial debiasing, prejudice remover, constrained optimisation
Post-processingAfter trainingEqualized odds post-processing, threshold adjustment
Practical guidance: Pre-processing is simplest but may not be effective. In-processing gives the best results but requires modifying the training algorithm. Post-processing is model-agnostic but can only fix what the model already learned.
Time: ~2 min. Overview slide. We will go into each stage briefly. The key message: there is no silver bullet — you often need to combine methods from multiple stages.

Pre-processing: Reweighting

Theory

Assign weights to training samples so that the weighted distribution satisfies demographic parity:

w(x) = \frac{P_{\text{expected}}(A, Y)}{P_{\text{observed}}(A, Y)}
Up-weight under-represented (group, outcome) combinations; down-weight over-represented ones

If women with positive outcomes are under-represented, their samples get higher weight. This guides the model to pay equal attention to all groups.

Time: ~2 min. Reweighting is the simplest mitigation technique. It does not modify the data, just how much each sample counts during training. Works with any weighted learning algorithm (most tree-based models, logistic regression).

In-processing: Adversarial Debiasing

Theory

Train a model that cannot predict the protected attribute from its own predictions:

Predictor

Learns to predict Y from X

Tries to maximise accuracy

Adversary

Tries to predict A from the predictor’s output

If it succeeds, the predictor is leaking group information

\min_\theta \max_\phi \;\mathcal{L}_{\text{task}}(\theta) - \lambda \cdot \mathcal{L}_{\text{adversary}}(\phi, \theta)
Minimax: the predictor tries to make the adversary’s job impossible
Time: ~2 min. This is a GAN-style training approach. The adversary acts as a fairness critic — if it can guess the group from the prediction, the prediction encodes group information. The predictor learns to be accurate while hiding group identity.

Post-processing: Equalized Odds Adjustment

Theory

After training, adjust the decision threshold per group to equalize TPR and FPR:

  • Group A: threshold = 0.45 (lower, to increase TPR)
  • Group B: threshold = 0.55 (higher, to decrease FPR)
Hardt et al. (2016): The optimal post-processing can be found by linear programming. It minimises the loss in accuracy while satisfying the equalized odds constraint. The solution is a randomised threshold at the group level.
Controversy: Some argue that different thresholds per group is itself discriminatory. Others argue it compensates for historical bias. This is a policy debate, not a technical one.
Time: ~2 min. Post-processing is the easiest to implement and the most controversial. It literally treats groups differently to achieve equal outcomes. The legal permissibility varies by jurisdiction.

Fairness Auditing Workflow

Foundations
  1. Define protected attributes and fairness criteria (legal + ethical)
  2. Measure baseline metrics: disparate impact ratio, TPR/FPR per group, calibration per group
  3. Identify failure modes: which groups are disadvantaged? By which metric?
  4. Apply mitigation: reweighting, adversarial debiasing, or threshold adjustment
  5. Re-measure: did the intervention improve fairness without destroying accuracy?
  6. Document: model card with fairness analysis, mitigation applied, residual risks
  7. Monitor: continuous fairness monitoring in production (fairness can degrade like accuracy)
Time: ~2 min. This is the end-to-end workflow. Steps 1-3 are analysis, 4-5 are mitigation, 6-7 are governance. In production, fairness monitoring is as important as accuracy monitoring — demographic drift can change fairness metrics even when accuracy stays stable.

Connections — Data Flow Between Nodes

Theory

Connections define how data moves between nodes. Each node has input ports and output ports:

# Simple connection: output of node A feeds into input of node B
wf.connect("load_data", "preprocess")

# Named ports: specify which output goes to which input
wf.connect("preprocess", "train",
           source_port="features",
           target_port="X_train")
wf.connect("preprocess", "train",
           source_port="labels",
           target_port="y_train")
Under the hood: Connections form a Directed Acyclic Graph (DAG). The runtime topologically sorts the graph and executes nodes in dependency order, parallelising independent branches.
Time: ~2 min. Named ports matter when a node produces multiple outputs. For example, a preprocessing node might output both features and labels. Without named ports, the runtime would not know which output goes where.

Custom Nodes

Theory 
from kailash import Node, register_node

@register_node("fairness_audit")
class FairnessAuditNode(Node):
    """Compute fairness metrics and flag violations."""

    async def process(self, inputs: dict) -> dict:
        model = inputs["model"]
        X_test = inputs["X_test"]
        y_test = inputs["y_test"]
        protected = inputs["protected_attribute"]

        predictions = model.predict(X_test)

        # Compute disparate impact
        group_0 = predictions[protected == 0].mean()
        group_1 = predictions[protected == 1].mean()
        di_ratio = min(group_0, group_1) / max(group_0, group_1)

        return {
            "disparate_impact": di_ratio,
            "passes_80_rule": di_ratio >= 0.8,
            "predictions": predictions,
        }
Time: ~3 min. Custom nodes let you encode domain logic into reusable pipeline components. The @register_node decorator makes the node discoverable by WorkflowBuilder. The process() method is async — it receives inputs and returns outputs. Error handling is built into the Node base class.

Async/Await — Why It Matters for ML

Foundations

Problem: Database and API calls are I/O-bound — the CPU waits while data travels over the network.

Analogy: You are cooking dinner. While the pasta boils (I/O wait), you can chop vegetables (other work). Async means: "start the task, do something else, come back when it is ready." Without async, you stare at the pot. 
# Synchronous (blocking) — stares at the pot

result2 = db.express.get(Experiment, id=2)  # Wait...

# Asynchronous (non-blocking) — chops vegetables while waiting

    db.express.get(Experiment, id=1),
    db.express.get(Experiment, id=2),

Grid Search — Exhaustive but Exponential

Foundations

Try every combination of hyperparameters on a predefined grid:

\text{Grid size} = \prod_{i=1}^{d} |G_i| = |G_1| \times |G_2| \times \cdots \times |G_d|
d = number of hyperparameters; |Gi| = number of values tried for parameter i
The curse of dimensionality: 5 parameters × 5 values each = 3,125 combinations. With 5-fold CV, that is 15,625 model trainings. For a model that takes 1 minute to train: 10.9 days.

Grid search is only practical for small grids (2-3 parameters, few values each).

Time: ~2 min. The exponential blowup is the key takeaway. Grid search is the default in many tutorials but it scales terribly. Use it only when you have very few hyperparameters to tune.

Random Search — Surprisingly Effective

Theory

Bergstra & Bengio (2012) showed that random search outperforms grid search for most problems.

Why Random Beats Grid

Most hyperparameters have different importance. Learning rate matters much more than subsample ratio.

Grid search wastes trials exploring values of unimportant parameters. Random search allocates trials uniformly across the important dimensions.

Visual Intuition

Imagine a 2D grid with 9 points (3×3). If only the x-axis matters, grid search tests only 3 unique x-values. Random search with 9 points tests 9 unique x-values.

Each random point explores a new slice of the important dimension.

Time: ~3 min. The Bergstra-Bengio result is one of the most impactful findings in hyperparameter optimization. Draw the 3x3 grid vs 9 random points on the board. With grid, you project onto 3 x-values. With random, you project onto 9. If x is the important parameter, random gives 3x more coverage.

Bayesian Optimisation — Learning from Past Trials

Foundations

Instead of trying random combinations, learn from previous results to make smarter choices.

Analogy: You are looking for the best restaurant in a new city. Random search: try random restaurants. Bayesian optimisation: try a few, notice that the Thai restaurants in the east side are consistently good, and focus your search there — while occasionally exploring other areas in case you are missing something.
Try a few configs
→
Build a model of "config → score"
→
Use model to pick the next config
→
Repeat
Time: ~3 min. Bayesian optimization is the state-of-the-art for hyperparameter tuning. The key insight: instead of treating each trial independently (like grid/random search), we build a probabilistic model of the objective function and use it to make informed decisions about where to search next.

Bayesian Optimisation — The Surrogate Model

Theory

We model the objective function f(\theta) with a Gaussian Process (GP) that provides both a prediction and uncertainty:

f(\theta) \sim \mathcal{GP}(\mu(\theta), k(\theta, \theta'))
μ(θ) = predicted score; k(θ, θ') = kernel encoding similarity between configs

After few trials

Wide uncertainty bands. The GP has seen little data. Many regions are unexplored.

After many trials

Narrow uncertainty near observed points. The GP confidently predicts good regions and identifies unexplored areas.

Time: ~2 min. The GP is the surrogate model — a cheap approximation of the expensive objective function. Instead of training a model to get the AUC (expensive), we query the GP to predict the AUC (cheap). The GP also tells us how uncertain it is, which is critical for the acquisition function.

Acquisition Functions — Explore vs Exploit

Theory

The acquisition function decides where to search next by balancing:

Exploit

Search near the best known result. Likely to find a slightly better config. May miss a much better region.

Explore

Search where uncertainty is high. May discover a great new region. May waste a trial on a bad area.

FunctionFormulaBias
Expected Improvement (EI)\text{EI}(\theta) = E[\max(0, f(\theta) - f^+)]Balanced
Upper Confidence Bound (UCB)\text{UCB}(\theta) = \mu(\theta) + \kappa \cdot \sigma(\theta)Tunable via κ
Probability of Improvement (PI)\text{PI}(\theta) = P(f(\theta) > f^+)Exploitative
Time: ~3 min. EI is the most commonly used acquisition function and the default in most libraries. It answers: "how much improvement can I expect from trying this config?" UCB with high kappa is more exploratory. PI is greedy and tends to converge too quickly.

Expected Improvement — Derivation

Theory
Step 1: Define improvement
I(\theta) = \max(0,\; f(\theta) - f^+)
f+ = best score observed so far
Step 2: Take expectation under the GP posterior
\text{EI}(\theta) = (\mu(\theta) - f^+)\,\Phi(Z) + \sigma(\theta)\,\phi(Z)

Where Z = \frac{\mu(\theta) - f^+}{\sigma(\theta)}, Φ is the CDF and φ is the PDF of the standard normal.

Two terms: The first term favours high predicted value (exploitation). The second term favours high uncertainty (exploration). When σ is large, the exploration term dominates.
Time: ~2 min. This is the closed-form EI formula under a Gaussian posterior. The two terms make the explore-exploit trade-off explicit. Show how setting sigma to zero recovers pure exploitation (first term only).

Hyperband — Adaptive Resource Allocation

Theory

Instead of training every configuration fully, start cheap and cut losers early.

Successive Halving (SHA)
  1. Start with N configurations, each with budget B/N (e.g., 10 epochs)
  2. Evaluate all configurations
  3. Keep the best half; double their budget
  4. Repeat until one configuration remains with full budget B
Hyperband: Run SHA at multiple aggressiveness levels. Some brackets start with many configs and little budget (aggressive). Others start with fewer configs and more budget (conservative). This hedges against the risk that early stopping misses slow starters.
Time: ~3 min. Hyperband is the most practical tuning method for expensive models (deep learning, large ensembles). The key insight: if a configuration performs badly after 10 epochs, it is unlikely to become the best after 100 epochs. So we can stop it early and reallocate resources.

ASHA — Asynchronous Successive Halving

Advanced

Standard SHA is synchronous: all configurations at a given rung must complete before promotion. ASHA makes this asynchronous:

  • Configurations are promoted as soon as they complete a rung
  • No waiting for slower configurations
  • Better GPU utilisation — no idle resources
BOHB (Falkner et al., 2018) combines Bayesian optimisation with Hyperband: the configurations to try are chosen by a surrogate model (TPE, not GP), and resource allocation follows Hyperband’s successive halving. This gives the best of both worlds: smart configuration selection + early stopping.
Time: ~2 min. Advanced topic. ASHA is the distributed-friendly version of Hyperband. BOHB is the current state-of-the-art for hyperparameter tuning when you have access to parallel compute. For most practical purposes, students should use Bayesian optimization for small search spaces and Hyperband for large/expensive ones.

Why Ensembles?

Foundations

Wisdom of crowds: A group of imperfect predictors, combined wisely, outperforms any individual predictor.

Analogy: If you ask 100 people to guess the number of jellybeans in a jar, the average guess is almost always closer to the truth than any individual guess. Why? Individual errors cancel out. The same principle applies to ML models.

Reduce Variance

Average out the noise from individual models

Method: Bagging (Random Forest)

Reduce Bias

Correct for systematic errors

Method: Boosting (XGBoost)

Combine Strengths

Different models excel on different parts of the data

Method: Stacking, Blending

Time: ~2 min. The jellybean jar is the classic example of the wisdom of crowds (Galton, 1907). Each person has a biased, noisy estimate. But averaging removes the noise and the biases (partially) cancel. The same math applies to models.

Bagging — Bootstrap + Aggregate

Foundations

Bagging (Breiman, 1996) reduces variance by training multiple models on bootstrap samples and averaging:

\hat{f}_{\text{bag}}(x) = \frac{1}{B}\sum_{b=1}^{B}\hat{f}_b(x)
B models, each trained on a bootstrap sample (sampling with replacement)
Variance reduction: If models are independent with variance \sigma^2, averaging B models gives variance \sigma^2/B. In practice, models are correlated (same data, same algorithm), so the reduction is less. Random Forest adds feature subsampling to reduce correlation.
Time: ~2 min. Quick recap from M3/M4. Random Forest = bagging of decision trees + random feature subsampling. The feature subsampling decorrelates the trees, making the variance reduction more effective.

Boosting — Sequential Error Correction

Foundations

Boosting (covered in depth in M3/M4) trains models sequentially, each correcting the previous model’s errors:

\hat{f}_M(x) = \sum_{m=1}^{M}\eta \cdot h_m(x), \qquad h_m \text{ fits the residuals of } \hat{f}_{m-1}
η = learning rate; hm = weak learner (shallow tree); residuals = mistakes of previous round

Key methods: XGBoost (2nd-order Taylor), LightGBM (histogram splitting, GOSS), CatBoost (ordered boosting for categoricals).

Time: ~2 min. Quick recap. The point is not to re-teach boosting (M3 did that) but to position it as one ensemble strategy among several. Students should understand: bagging reduces variance, boosting reduces bias.

Stacking — Meta-Learning

Theory

Stacking uses a meta-learner to combine predictions from diverse base models:

Level 0: Base Models

Train diverse models on the training data: XGBoost, Random Forest, Logistic Regression, Neural Network, etc.

Level 1: Meta-Learner

Use base model predictions as features for a simple meta-model (often logistic regression or ridge regression).

\hat{y}_{\text{stack}} = g\!\left(\hat{y}_1(x),\; \hat{y}_2(x),\; \ldots,\; \hat{y}_K(x)\right)
g = meta-learner; &hat;yk = prediction from base model k
Time: ~3 min. Stacking is the most powerful ensemble method but also the most complex. The meta-learner learns which base model to trust in different regions of the input space. The key requirement: base models must be diverse (different algorithms, not just different hyperparameters).

K-Fold Stacking — Preventing Leakage

Theory

Naive stacking creates base model predictions on the same data used for training, causing leakage. K-fold stacking fixes this:

  1. Split training data into K folds
  2. For each fold k:
    • Train each base model on the other K-1 folds
    • Predict on fold k (out-of-fold predictions)
  3. Combine all out-of-fold predictions into a new feature matrix
  4. Train the meta-learner on this out-of-fold matrix
  5. For test data: train all base models on full training data, predict, pass to meta-learner
Without K-fold stacking, base models see their own training data at level 1, leading to overfitting. The out-of-fold approach ensures the meta-learner sees predictions that the base models made on unseen data — mimicking production conditions.
Time: ~3 min. This is the most important detail of stacking. Without K-fold, stacking overfits badly because base model predictions on training data are artificially good. With K-fold, predictions are honest because each base model predicts on data it did not train on.

Blending — Simple Alternative to Stacking

Theory

Blending is a simplified version of stacking that uses a holdout set instead of K-fold:

  1. Split training data into train (80%) and blend (20%) sets
  2. Train base models on train set
  3. Generate predictions on blend set
  4. Train meta-learner on blend predictions

Advantages over Stacking

  • Simpler to implement
  • Faster (no K-fold loop)
  • Easier to debug

Disadvantages

  • Less data for training base models
  • Meta-learner trained on less data
  • Higher variance in meta-learner
Time: ~2 min. Blending is the pragmatic choice when K-fold stacking is too expensive. In Kaggle competitions, stacking usually wins. In production, blending is often preferred because it is simpler and faster.

Voting — Simple and Effective

Foundations

Hard Voting

Each model votes for a class. The class with the most votes wins.

\hat{y} = \text{mode}(\hat{y}_1, \hat{y}_2, \ldots, \hat{y}_K)

Soft Voting

Average the predicted probabilities. Choose the class with highest average probability.

\hat{p}(c) = \frac{1}{K}\sum_{k=1}^{K}\hat{p}_k(c)
Soft voting is usually better because it uses more information (probabilities vs binary votes). But it requires all models to output well-calibrated probabilities — otherwise, an overconfident model dominates the vote.
Time: ~2 min. Voting is the simplest ensemble. Hard voting is majority rule. Soft voting averages probabilities. Soft voting works better when models are well-calibrated (M3 taught calibration). An uncalibrated model can hijack soft voting by producing extreme probabilities.

Diversity — The Key to Good Ensembles

Theory

Ensembles benefit from diversity. If all models make the same mistakes, combining them adds no value.

Source of DiversityExample
Different algorithmsXGBoost + Neural Net + SVM + Logistic Regression
Different featuresOne model uses text features, another uses numeric only
Different data subsetsBagging (bootstrap samples)
Different hyperparametersSame algorithm with different depth/learning rate
Different preprocessingOne model uses PCA, another uses raw features
The diversity-accuracy trade-off: Maximising diversity (using very different models) may include weak models that hurt overall performance. The ideal ensemble has models that are individually accurate AND collectively diverse.
Time: ~2 min. Diversity is the secret ingredient. Five XGBoost models with slightly different hyperparameters add little value. One XGBoost + one neural net + one linear model adds significant value because they make different kinds of errors.

Monitoring — Data Drift

Foundations

Data drift = the input data distribution changes after deployment.

Analogy: You trained a weather model on Singapore data. Suddenly, you start getting inputs from Norway. The model has never seen temperatures below 10°C. It will make terrible predictions — but it will not know its predictions are terrible.

Population Stability Index (PSI)

\text{PSI} = \sum_i (p_i - q_i) \ln\frac{p_i}{q_i}

p = training distribution, q = production distribution.

Interpretation

PSI < 0.1No significant drift
0.1 ≤ PSI < 0.25Moderate drift — investigate
PSI ≥ 0.25Significant drift — retrain
Time: ~3 min. PSI is the most widely used drift metric. It measures the divergence between training and production feature distributions. The formula is a symmetrized KL divergence. PSI is computed per feature — check all features, not just the overall distribution.

Kolmogorov-Smirnov Test for Drift

Theory

The KS test measures the maximum distance between two cumulative distribution functions:

D = \sup_x |F_{\text{train}}(x) - F_{\text{prod}}(x)|
D = maximum vertical distance between the two CDFs; sup = supremum (maximum)

Advantages

  • Non-parametric (no distribution assumptions)
  • Sensitive to any kind of distributional change
  • Well-known p-value tables

Limitations

  • Most sensitive in the middle of the distribution
  • Less sensitive to tail changes
  • Requires continuous features
Time: ~2 min. KS test complements PSI. PSI is better for binned distributions; KS is better for continuous features. Use both: PSI for a quick overview, KS for statistical significance. DriftMonitor computes both automatically.

Performance Degradation

Theory

Drift in inputs eventually causes drift in outputs. Monitor the full chain:

SignalMetricDetection Window

The ground truth delay: In credit scoring, you do not know if a loan defaults until 90+ days after approval. You cannot wait 90 days to detect model failure. Use input and prediction drift as early warnings before ground truth is available.
Time: ~2 min. The four monitoring layers are arranged from early (input drift) to late (business impact). The key insight: by the time you see business impact, the damage is done. Monitor input drift as a leading indicator. This is why DriftMonitor exists.

Alerting and Retraining Triggers

Theory

When to Alert

  • PSI ≥ 0.25 on any critical feature
  • KS test p-value < 0.01
  • Prediction distribution shift > 2σ
  • Fairness metric breach

When to Retrain

  • Sustained drift for > 7 days
  • Performance drop > 5% on labelled subset
  • Scheduled (weekly/monthly regardless)
  • After significant data distribution change (e.g., product launch)

Behavioural Testing — Beyond Accuracy

Theory

Ribeiro et al. (2020) proposed three types of behavioural tests:

Invariance Tests

Changing a feature that should not matter must NOT change the prediction. Example: swapping "John" for "Aisha" in a loan application.

Directional Tests

Changing a feature in a known direction must change the prediction accordingly. Example: increasing credit score should never decrease approval probability.

Minimum Functionality Tests

The model must handle specific known cases correctly. Example: an applicant with 0 income and no collateral must be denied.

Data Validation Tests

Input data must match expected schema, ranges, and distributions. Example: no negative ages, no future dates.

Time: ~2 min. These tests come from the CheckList paper (Ribeiro et al., 2020). They are model-agnostic and can catch issues that aggregate metrics miss. In production, run these as part of CI/CD before any model promotion.

Conformal Prediction — Honest Uncertainty

Theory

How do you know when your model is unsure? Conformal prediction provides guaranteed prediction intervals:

P(Y \in C(X)) \geq 1 - \alpha
C(X) = prediction set/interval; α = error rate (e.g., 0.05 for 95% coverage)

Key Properties

  • Distribution-free: no assumptions about data distribution
  • Model-agnostic: works with any model
  • Finite-sample guarantee: valid for any sample size
  • No retraining: post-hoc calibration

How It Works

  1. Split data into training + calibration
  2. Train model on training data
  3. Compute non-conformity scores on calibration
  4. Use quantile of scores as threshold
  5. For new predictions, include values with score below threshold
Time: ~3 min. Conformal prediction is becoming standard in production ML (2025-2026). Unlike model confidence scores, conformal prediction intervals have formal statistical guarantees. The guarantee holds even if the model is wrong, because it comes from the calibration data, not the model itself.

Conformal Prediction — Practical Example

Theory

Classification: instead of "approved" (0.73 confidence), conformal prediction says:

Prediction set at 95% coverage: {approved, review}
Prediction set at 90% coverage: {approved}

When the set contains only one class, the model is confident. When it contains multiple classes, the model is uncertain — and a human should review.

Regression: instead of "$450,000" (point estimate), conformal prediction says:

Prediction interval at 95% coverage: [$410,000, $495,000]
Guarantee: The true value falls within this interval at least 95% of the time.
Time: ~2 min. Conformal prediction is transformative for production because it tells you WHEN the model is uncertain. A wide prediction interval means "do not trust this prediction" — route it to a human. A narrow interval means "this prediction is reliable." This is exactly what Zillow was missing (opening case from M3).

Conformal Prediction Under Distribution Shift

Advanced

Standard conformal prediction assumes exchangeability (i.i.d. calibration and test data). In production, distributions shift.

Adaptive conformal prediction (Gibbs & Candes, 2021) adjusts the threshold over time:

\alpha_{t+1} = \alpha_t + \gamma \cdot (\text{err}_t - \alpha)

If the coverage falls below the target, the threshold widens. If it exceeds the target, the threshold tightens. This provides approximate coverage guarantees even under gradual distribution shift.
Time: ~2 min. Advanced topic. The adaptive method is like a feedback controller for coverage. When the model encounters new data distributions, the intervals widen automatically. Recent work (2025-2026) handles subpopulation shifts where the test environment contains a different mix of subgroups than calibration.

Discussion Prompt
