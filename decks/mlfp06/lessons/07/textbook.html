Lesson 6.7 AI Governance Engineering

Governance is not a committee discussion or an ethics position paper. It is code — access controls that deny unauthorised access, operating envelopes that restrict agent behaviour, budget cascading that prevents runaway costs, and audit trails that prove compliance. Today you implement all four with PACT: D/T/R addressing, GovernanceEngine, PactGovernedAgent, and CostTracker. The result: a multi-agent system where every tool call is checked, every budget is enforced, and every decision is logged.

Why This Matters

Singapore's Monetary Authority (MAS) requires financial institutions to explain every AI-driven decision that affects customers. A recommendation engine that denies a loan must log why. An agent that executes a trade must record who authorised it, what envelope it operated under, and how much it was allowed to spend. "The AI decided" is not an acceptable audit response.

Without governance engineering, you get one of two failure modes. Either the system is wide open — any agent can access any data and spend any amount — which is a security and compliance disaster. Or governance is bolted on as an afterthought — a brittle layer of if-statements that breaks every time the system changes. PACT makes governance a first-class component: declared, compiled, enforced, and tested.

Core Concepts

D/T/R Addressing

PACT's addressing grammar breaks identity into three dimensions:

DimensionMeaningExample
Domain (D)Business unit or functional areafinance, operations, hr
Team (T)Functional group within a domainrisk, compliance, trading
Role (R)Individual capability levelanalyst, manager, director

A D/T/R address like finance/risk/analyst uniquely identifies what access a requester should have. The GovernanceEngine compiles an organisational structure from domain, team, and role definitions, then resolves access by walking the hierarchy: a manager inherits all analyst permissions plus any additional manager-specific grants.

Foundations

Think of D/T/R as a postal address for permissions. The domain is the country, the team is the city, the role is the street address. When a letter (access request) arrives, the system checks: does this address have a mailbox (permission) for this type of mail (resource)? The address system is hierarchical — a domain-level permission applies to all teams within it.

Operating Envelopes

An operating envelope defines the boundaries within which an agent can operate. If D/T/R addressing answers "who can access what", envelopes answer "what can this agent do, even with access". Two types:

Task Envelopes

Restrict the agent to specific task types. A data analysis agent may read data and compute statistics, but it cannot write to the database or send emails. The task envelope is the list of permitted actions. 

task_envelope = {
    "allowed_tools": [
        "explore_data",
        "compute_statistics",
        "generate_report"
    ],
    "blocked_tools": [
        "write_database",
        "send_email",
        "deploy_model"
    ]
}

Role Envelopes

Restrict based on the agent's organisational role. An analyst agent gets a smaller budget, fewer tool options, and read-only data access. A manager agent gets a larger budget and write access. 

role_envelopes = {
    "analyst": {
        "max_cost": 5.00,
        "data_access": "read",
        "model_access": "predict"
    },
    "manager": {
        "max_cost": 50.00,
        "data_access": "read_write",
        "model_access": "predict_deploy"
    }
}

Monotonic Tightening

The critical safety property of envelopes: they can only get stricter, never looser. A supervisor agent that allocates a $10 budget to a child agent cannot later increase it to $20. A task envelope that excludes deploy_model cannot later add it. This is monotonic tightening — trust flows downward and narrows at each level.

Theory

Monotonic tightening is a lattice property. If we define a partial order on envelopes where $E_1 \leq E_2$ means "$E_1$ is at least as restrictive as $E_2$", then the set of all envelopes forms a lattice with the most restrictive envelope (deny everything) at the bottom and the most permissive (allow everything) at the top. Child envelopes must always be below or equal to their parent in this lattice — they can only move downward.

Enforcement Modes

PACT supports three enforcement modes for governance checks:

ModeBehaviourUse case
blockDeny access immediately. Fail-closed.Production systems. This is the default.
warnLog violation, allow access.Development and testing. See what would be blocked.
auditAllow access, log for review.Shadow mode before enabling block. Compliance review.
Advanced

The standard deployment progression is: audit for one week (collect baseline), review audit logs for false positives, adjust rules, switch to warn for another week (verify no critical workflows break), then block for production. This gradual rollout prevents governance rules from accidentally blocking legitimate operations.

Mathematical Foundations: Budget Cascading

Budget cascading is a constraint propagation problem. A parent agent with budget $B$ allocates portions to $n$ child agents:

$$ \sum_{i=1}^{n} b_i \leq B \quad \text{where each } b_i \geq 0 $$
$B$ — parent budget $b_i$ — allocation to child $i$ $n$ — number of children

The sum of all child budgets cannot exceed the parent's total. Each child can further subdivide its allocation, but the total at every level is bounded. When a child exhausts its budget, it stops gracefully and returns partial results. The parent can then decide whether to reallocate remaining budget from other children.

For a three-level hierarchy (supervisor, team leads, workers) with supervisor budget $B = \$50$:

$$ B_{\text{supervisor}} = \$50 \geq \underbrace{b_{\text{lead}_1} + b_{\text{lead}_2}}_{= \$20 + \$20} + b_{\text{reserve}} $$ $$ b_{\text{lead}_1} = \$20 \geq \underbrace{b_{\text{worker}_1} + b_{\text{worker}_2} + b_{\text{worker}_3}}_{= \$6 + \$6 + \$6} + b_{\text{lead reserve}} $$

Each allocation is a hard ceiling. Workers cannot request additional budget from the supervisor — they can only use what their team lead allocated. This prevents any single branch from consuming the entire budget.

Foundations

Budget cascading works like a company's departmental budgets. The CEO allocates to VPs; VPs allocate to directors; directors allocate to teams. Each level can only spend what it was given. No one can reach up to grab more money. If your team runs out, you stop and report back.

Worked Example: Governed Multi-Agent System

Goal

Build a governed multi-agent system for HDB price analysis in Singapore. A supervisor agent delegates to three specialists: DataScientist, ModelSelector, and ReportWriter. Each agent has a defined role, budget, and task envelope. Unauthorised access is blocked. Budget overruns are prevented. Every decision is logged.

Step 1 — Define the organisational structure. 
from kailash_pact import GovernanceEngine, Address, CostTracker

engine = GovernanceEngine()

# Define the D/T/R hierarchy
org = engine.compile_org({
    "domains": ["mortgage-risk"],
    "teams": {
        "mortgage-risk": ["assessment", "audit"]
    },
    "roles": {
        "assessment": ["analyst", "senior-analyst", "manager"],
        "audit": ["reviewer", "compliance-officer"]
    },
    "permissions": {
        "analyst": ["read_data", "run_predictions"],
        "senior-analyst": ["read_data", "run_predictions", "train_model"],
        "manager": ["read_data", "run_predictions", "train_model",
                     "deploy_model", "manage_budget"],
        "reviewer": ["read_data", "read_audit_log"],
        "compliance-officer": ["read_data", "read_audit_log",
                                "export_audit_report"]
    }
})
Step 2 — Define operating envelopes for each agent. 
# Supervisor envelope: full control within domain
supervisor_envelope = {
    "allowed_tools": ["delegate", "allocate_budget", "aggregate_results"],
    "max_cost": 10.00,
    "enforcement": "block"
}

# DataScientist envelope: data exploration only
data_scientist_envelope = {
    "allowed_tools": ["explore_data", "compute_statistics",
                       "feature_engineering"],
    "blocked_tools": ["train_model", "deploy_model"],
    "max_cost": 3.00,
    "data_access": "read"
}

# ModelSelector envelope: model training, no deployment
model_selector_envelope = {
    "allowed_tools": ["train_model", "evaluate_model",
                       "hyperparameter_search"],
    "blocked_tools": ["deploy_model"],
    "max_cost": 4.00
}

# ReportWriter envelope: read-only, reporting tools only
report_writer_envelope = {
    "allowed_tools": ["generate_report", "format_output"],
    "max_cost": 3.00,
    "data_access": "read"
}
Step 3 — Set up budget cascading. 
tracker = CostTracker(total_budget=10.00)

# Allocate to children — sum must not exceed total
tracker.allocate("data_scientist", 3.00)
tracker.allocate("model_selector", 4.00)
tracker.allocate("report_writer", 3.00)

# Verify constraint
assert tracker.remaining() >= 0  # no overallocation
print(f"Total allocated: ${tracker.total_allocated():.2f}")
print(f"Remaining reserve: ${tracker.remaining():.2f}")
Step 4 — Wrap agents with governance. 
from kailash_pact import PactGovernedAgent

# Each agent gets its envelope and budget
governed_data_scientist = PactGovernedAgent(
    agent=data_scientist_agent,
    governance=engine,
    address=Address("mortgage-risk", "assessment", "analyst"),
    envelope=data_scientist_envelope,
    cost_tracker=tracker.child("data_scientist")
)

governed_model_selector = PactGovernedAgent(
    agent=model_selector_agent,
    governance=engine,
    address=Address("mortgage-risk", "assessment", "senior-analyst"),
    envelope=model_selector_envelope,
    cost_tracker=tracker.child("model_selector")
)

governed_report_writer = PactGovernedAgent(
    agent=report_writer_agent,
    governance=engine,
    address=Address("mortgage-risk", "assessment", "analyst"),
    envelope=report_writer_envelope,
    cost_tracker=tracker.child("report_writer")
)
Step 5 — Test access controls. 
# Verify access: analyst can read data
addr = Address("mortgage-risk", "assessment", "analyst")
assert engine.can_access(addr, "read_data")
print(engine.explain_access(addr, "read_data"))
# → "Granted: role 'analyst' has permission 'read_data'"

# Verify denial: analyst cannot deploy models
assert not engine.can_access(addr, "deploy_model")
print(engine.explain_access(addr, "deploy_model"))
# → "Denied: role 'analyst' lacks permission 'deploy_model'.
#    Required role: 'manager'"

# Verify envelope enforcement
try:
    governed_data_scientist.call_tool("deploy_model", {})
except GovernanceViolation as e:
    print(f"Blocked: {e}")
# → "Blocked: tool 'deploy_model' not in allowed_tools for envelope"
Step 6 — Generate audit trail. 
# Retrieve audit log
audit_log = engine.get_audit_trail()
for entry in audit_log:
    print(f"[{entry.timestamp}] {entry.address} → {entry.resource}: "
          f"{entry.decision} ({entry.reason})")

# Output:
# [2026-04-13T10:01:02] mortgage-risk/assessment/analyst → read_data:
#     GRANTED (role 'analyst' has permission 'read_data')
# [2026-04-13T10:01:03] mortgage-risk/assessment/analyst → deploy_model:
#     DENIED (role 'analyst' lacks permission 'deploy_model')

Every governance decision — granted and denied — is logged with a timestamp, the D/T/R address, the resource, the decision, and the reason. This audit trail is what MAS (or any regulator) will review.

Clearance Levels

PACT's ClearanceManager adds graduated access based on trust levels. Rather than binary allow/deny, clearance levels enable progressive trust: a new agent starts at level 1 (read-only, low budget) and earns higher clearance through demonstrated reliability.

LevelNameAccessTypical budget
1ObserverRead-only data access$1
2ContributorRead + compute (statistics, predictions)$5
3PractitionerRead + compute + train models$20
4AuthorityFull access including deploy$50
 
from kailash_pact import ClearanceManager

clearance = ClearanceManager()

# New agent starts at level 1
clearance.set_level("new_agent", level=1)

# After 100 successful operations with no violations
clearance.promote("new_agent", to_level=2,
    reason="100 successful ops, 0 violations")

# Cannot skip levels
clearance.promote("new_agent", to_level=4)  # raises: max +1 per promotion
Advanced

Clearance promotion can be automated with a verification gradient. Define a scoring function that measures agent reliability (successful operations / total operations), governance compliance (violations / total checks), and cost efficiency (actual spend / allocated budget). When the score exceeds a threshold over a defined period, the agent is eligible for promotion. Demotion is immediate on any critical violation — trust is earned slowly and lost quickly.

Governance Testing Patterns

Governance testing is not optional — it is the most important testing you do. Functional tests verify that the system works. Governance tests verify that the system fails safely. Here are the four patterns: 

import pytest
from kailash_pact import GovernanceViolation

# Pattern 1: Verify access denial
def test_analyst_cannot_deploy():
    addr = Address("mortgage-risk", "assessment", "analyst")
    assert not engine.can_access(addr, "deploy_model")

# Pattern 2: Verify envelope enforcement
def test_tool_blocked_by_envelope():
    with pytest.raises(GovernanceViolation):
        governed_data_scientist.call_tool("deploy_model", {})

# Pattern 3: Verify monotonic tightening
def test_cannot_loosen_envelope():
    agent = PactGovernedAgent(agent=base_agent, envelope={"max_cost": 5.00})
    agent.tighten_envelope(max_cost=3.00)  # OK
    with pytest.raises(GovernanceViolation):
        agent.loosen_envelope(max_cost=10.00)  # BLOCKED

# Pattern 4: Verify budget exhaustion
def test_budget_stops_agent():
    tracker = CostTracker(total_budget=0.01)  # tiny budget
    agent = PactGovernedAgent(agent=base_agent, cost_tracker=tracker)
    result = agent.run("Complex analysis")
    assert result.stopped_reason == "budget_exhausted"
    assert tracker.total_spent() <= 0.01

# Pattern 5: Verify audit trail completeness
def test_all_checks_logged():
    engine.clear_audit_trail()
    addr = Address("mortgage-risk", "assessment", "analyst")
    engine.can_access(addr, "read_data")
    engine.can_access(addr, "deploy_model")
    trail = engine.get_audit_trail()
    assert len(trail) == 2
    assert trail[0].decision == "GRANTED"
    assert trail[1].decision == "DENIED"
Critical principle

Governance tests are negative tests. They test what does NOT happen. Every assert not and every pytest.raises(GovernanceViolation) is a guardrail that prevents the system from doing something dangerous. If these tests fail, it means your governance has a hole. Treat governance test failures as critical — they are security vulnerabilities.

Common Mistakes

Mistake 1: Fail-open governance

If the governance check fails (network error, config issue, missing rules), deny access. Never default to allow. A system that allows everything when governance breaks is worse than a system with no governance at all, because it creates a false sense of security. PACT's default is enforcement="block" with fail-closed semantics.

Mistake 2: Testing only the happy path

"The analyst can read data" is a happy path test. The governance tests that matter are: "The analyst cannot deploy models", "The analyst cannot exceed budget", "The analyst cannot loosen their envelope". Test what is denied, not just what is granted.

Mistake 3: No audit trail

Compliance requires proving every access decision was checked. Log every can_access() call — both grants and denials. The audit trail is not for debugging; it is for the regulator. MAS, PDPA, and GDPR all require demonstrable access control.

Mistake 4: Governance as philosophy instead of code

"We have an AI ethics framework" is not governance. Governance is engine.can_access(addr, resource) returning False and the tool call being blocked. If it is not in code, it is not enforced. If it is not tested, it does not work. This lesson is engineering, not philosophy.

Reflection

Governance is the hardest lesson in the programme to appreciate — until you deploy without it and face the consequences. An ungoverned agent with access to a production database can delete records, leak personal data, or spend thousands of dollars in minutes. An ungoverned multi-agent system can cascade errors across agents with no way to trace what happened.

What you built today — D/T/R addressing, operating envelopes, budget cascading, audit trails, governance tests — is the difference between a prototype and a production system. Prototypes work when everything goes right. Production systems work when everything goes wrong. PACT ensures that "wrong" is bounded, logged, and recoverable.
