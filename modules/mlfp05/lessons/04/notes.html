Speaker Notes — Lesson 5.4 Transformers and Self-Attention

Total time: ~75 minutes · 17 slides · ~4 min per slide

This is the most important lesson in the module. If the class understands nothing else at the end of Module 5, they need to understand self-attention — it is the beating heart of every LLM, every modern translation system, every image generator that uses conditioning, every Whisper-style speech model, and everything in Module 6. Allocate more time here than any other lesson. Budget ~20 min on the attention derivation (slides 3–5), ~15 min on multi-head and positional encoding (6–9), and ~15 min on the encoder assembly and worked example (10–14).

Unlike earlier lessons in this module, do not rush the math. The ELBO in 5.1 was optional; the attention math is not. Every student must be able to write down the formula and explain each piece.

Slide 1~2 minTitle

Transformers and Self-Attention

Open with the stakes. "This is probably the most influential architecture of the last decade. Every large language model — GPT, Claude, Gemini, Llama — is a Transformer. Every modern translation, summarisation, QA system is a Transformer. Vision Transformers are catching CNNs on standard benchmarks. Whisper is a Transformer. AlphaFold 2 is a Transformer. If you leave this course understanding only one architecture deeply, this should be the one."

Warn them about the pace: "we are going to slow down on the math. I want every single person in this room to be able to write down the attention equation from memory by the end of the hour. No exceptions."

→ "Why did attention replace recurrence?"
Slide 2~4 minMotivation

Why attention replaced recurrence

Start with the parallelism argument. "Yesterday we saw LSTMs. They carry a hidden state forward one step at a time. To compute the state at time 5, you need the state at time 4, which needs time 3, and so on. The computation is inherently sequential. Modern GPUs have thousands of cores. An LSTM uses essentially one of them for the time-step direction — enormous waste."

Then the observation: "Vaswani et al. asked: what if every token could directly look at every other token, in parallel, via an attention mechanism? No hidden state pipelined through time. Every position computed at once. The paper was titled 'Attention is all you need'. The title is correct — you don't need recurrence at all."

→ "The database metaphor."
Slide 3~4 minQKV

Query, Key, Value — database metaphor

Walk the metaphor slowly. Use the database analogy explicitly. "Imagine a key-value store. You submit a query — a description of what you want. The store has a bunch of keys, each describing what it contains. Your query is compared against every key. The best match returns its value."

Extend the analogy: "In attention, we soften that. Instead of picking the single best key, we compute a similarity to every key, softmax into weights, and return a weighted combination of the values. Now instead of one answer, we get a smooth mixture — and that mixture is differentiable, so we can train it."

The key insight: "every token produces a Q, K, and V simultaneously. So every token can query every other token. It's an all-to-all lookup, in parallel, in one matrix multiplication."

→ "The formula."
Slide 4~5 minAttention formula

Scaled dot-product attention

Put the formula on the board. Walk it right to left, slowly. "Q and K are L-by-d_k matrices — one row per token. QKᵀ is L-by-L, containing every pairwise dot product. Entry (i,j) is how much token i's query matches token j's key."

"Divide by √d_k — we come back to why in the next slide. Softmax each row — so row i becomes a probability distribution over all keys: 'how much should token i attend to each other token?' Then multiply by V — each token's output is the weighted sum of value vectors according to its attention distribution."

Concrete: "if token 3 wants information from token 7, it learns to produce a query whose dot product with token 7's key is large. After softmax, most of the attention weight lands on token 7, and the output for position 3 is dominated by V_7."

For beginners: "This formula will not make sense on first read. That's normal. Read it five times over the next week. Each time one more piece clicks."
→ "But why √d_k specifically?"
Slide 5~4 min√d_k reasoning

Why divide by √d_k?

Walk the variance calculation. "q and k are d_k-dimensional. Assume unit variance per entry. Their dot product is a sum of d_k products of unit-variance numbers. Independent sum gives variance d_k. Standard deviation is √d_k. That's the typical magnitude of a raw score."

"Feed √d_k-magnitude scores into a softmax and the softmax saturates — one entry gets essentially all the weight, the rest are zero, gradients are zero for everything except the winner, training stalls."

"Dividing by √d_k brings scores back to unit variance. Softmax stays in its useful region. Gradients flow. Three characters of code, entire paper does not work without them."

For experts: this is why Gaussian initialisation of the Q/K/V projection matrices is standard — it keeps the variance assumption approximately valid. If you ever see a Transformer stuck at high loss, the first thing to check is the scale factor.
→ "Now the code."
Slide 6~4 mineinsum impl

From scratch with einsum

Walk the six lines. Pause at each einsum: "einsum lets us multiply tensors along named axes. 'bld,bmd->blm' means: for each batch b, take the outer product of the length-l and length-m axes, summing over d. That gives us the L×L score matrix."

"Divide by √d_k. Softmax over the last axis (the key positions). Second einsum: for each batch b and each query position l, weight the value vectors by the attention row. Output: (B, L, d_k)."

"Two einsum calls, one softmax, one math.sqrt. That's the entire op. Everything else is just book-keeping and projection matrices."

→ "Multi-head."
Slide 7~4 minMulti-head

Multi-head attention

Motivate: "one attention head gives every token one 'viewpoint'. But different linguistic phenomena live on different subspaces — syntactic agreement cares about verb-subject pairs, coreference cares about noun-pronoun links, topic-relevance cares about content words. We want the model to attend to all of these simultaneously."

"Multi-head runs h parallel attentions on smaller subspaces and concatenates. With d_model = 512 and 8 heads, each head gets d_k = 64. Same total parameter count as one big d_k=512 attention, but eight different attention patterns."

Show the formula. Explain Concat and W^O briefly — "concatenate the h head outputs, then a final linear projection back to d_model."

→ "Diagram."
Slide 8~2 minMulti-head diagram

Multi-head diagram

Walk the picture. "Input x. Four parallel projections into Q, K, V subspaces — one per head. Each head runs its own scaled dot-product attention. Outputs concatenated along the feature axis. Final W^O projection back to d_model."

Emphasise: "this is all done in a single matrix multiplication in practice. PyTorch's nn.MultiheadAttention fuses the Q/K/V projections for speed."

→ "Position."
Slide 9~4 minPositional encoding

Positional encoding

Set up the problem. "Self-attention is permutation invariant. If I shuffle the tokens, the output is the same shuffled. But language is ordered: 'dog bites man' vs 'man bites dog' have completely different meanings. We need to inject position information."

Explain the sinusoidal choice. "The original paper uses sin and cos at different frequencies. Each position gets a unique fingerprint — a vector of sin and cos values at geometrically spaced wavelengths. The key property: PE(pos + k) is a linear function of PE(pos), so the model can learn relative position via trigonometric identities."

"We add the position vector to the token embedding, element by element. It's not concatenated — it's added. Modern variants (RoPE, ALiBi) use different schemes, but the sinusoidal encoding is still the textbook one."

→ "Code."
Slide 10~3 minPE code

PositionalEncoding in PyTorch

Walk the code. "Precompute a (max_len, d_model) matrix of sin/cos values. Register it as a buffer (not a parameter — it's constant). In forward, just add the first L rows to the input. Broadcast handles the batch dimension."

Point out register_buffer: "this tells PyTorch the tensor is part of the module state (moved to GPU when you call .to(device)) but not trained. Use it for any constant tensor inside a module."

→ "Encoder block structure."
Slide 11~3 minEncoder block

The encoder block

Walk the six components. "Multi-head attention. Residual (add input). LayerNorm. Feed-forward (two linear layers with ReLU). Residual. LayerNorm. That's one encoder block. Stack N of them to get the full encoder."

"Pre-norm vs post-norm: the original paper used LayerNorm after the residual (post-norm). Modern practice is to apply LayerNorm before the attention (pre-norm), which trains much more stably. PyTorch has a norm_first=True flag."

→ "The code we'll use."
Slide 12~3 minTinyTransformer

TinyTransformer in PyTorch

Walk the code. "Embedding layer — maps token ids to vectors. PositionalEncoding — adds position info. nn.TransformerEncoderLayer — one encoder block with 4 heads. nn.TransformerEncoder — stack 2 of them. Linear head."

"In forward: embed, add position, run encoder, mean-pool over sequence, project to classes. Twenty lines total, full working Transformer. PyTorch does all the heavy lifting once you understand what's inside."

→ "Why layer norm not batch norm?"
Slide 13~2 minLayer norm

Layer norm, not batch norm

Quick explanation. "BatchNorm normalises across the batch — for each feature, subtract the batch mean and divide by the batch std. Works great for CNNs, fails for Transformers because sequence lengths vary and the effective batch size at each position is not constant."

"LayerNorm normalises across features — for each token independently, subtract its mean and divide by its std. No batch dependency. Works the same way at train and inference. Always LayerNorm in Transformers."

→ "The variants zoo."
Slide 14~3 minVariants

Transformer variants

Walk the table. Do not dwell — just orient the class. "BERT uses only the encoder, trained bidirectionally with masked language modelling. GPT uses only the decoder with causal masking and next-token prediction. T5 uses both with a text-to-text objective. ViT applies the encoder to image patches. Whisper applies encoder-decoder to audio."

"The underlying block is the same. What changes is the mask, the training objective, and the data. We'll fine-tune BERT in 5.7."

→ "Worked example."
Slide 15~4 minWorked example

Majority-bit task worked example

Set up the problem: "given a length-20 bit string, output 1 if the majority are 1, else 0. Trivial for a human, non-trivial for a model because it requires counting across positions. A positional-agnostic bag-of-words classifier can solve it, but any sequence model that looks only at the last position will fail."

Run the training if you have the time. 20 epochs, ~10 seconds on CPU. Should get 100% accuracy.

"Optional extension: visualise attention weights. PyTorch's attention returns them if you hook it correctly. Show a heat map and you'll see the model is essentially summing across positions uniformly — exactly what the task requires."

→ "Traps."
Slide 16~3 minTraps

Four traps

Walk each. The missing PE trap is the worst — "your model gets stuck at 50% on any order-dependent task and you stare at the code for an hour before noticing you forgot to add the positional encoding."

Missing √d_k: same class of bug — "loss drops for one epoch then flatlines at high value".

Mask types: "Pytorch's attn_mask and key_padding_mask are different arguments. Confusing them is a rite of passage."

Quadratic memory: "OOM killer strikes at seq_len 4096 on a consumer GPU. Use FlashAttention or Longformer for longer contexts."

Reflection — the four paradigms so far
