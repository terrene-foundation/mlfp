Your first job is to throw away the 99% of rows you don't need.

Booleans: True or False

Six comparison operators that produce a boolean:

OperatorMeaningExample

Watch: = assigns. == compares. Different operators.

Polars expressions: recipes, not actions

An expression on its own does nothing. Handed to filter, with_columns, or select, it becomes work.

What filter does to rows

filter evaluates a boolean expression on every row, then keeps only the True rows.

Your first filter


import polars as pl
from shared import MLFPDataLoader

loader = MLFPDataLoader()
hdb = loader.load("mlfp01", "hdb_resale.parquet")

# All Ang Mo Kio transactions
amk = hdb.filter(pl.col("town") == "ANG MO KIO")
print(f"AMK transactions: {amk.height:,}")

Read it aloud: filter hdb to rows where the town column equals "ANG MO KIO".

Combining conditions with & and |


# AND — both must hold
affordable = hdb.filter(
    (pl.col("resale_price") >= 300_000)
    & (pl.col("resale_price") <= 500_000)
)

# OR — either holds
central = hdb.filter(
    (pl.col("town") == "BISHAN")
    | (pl.col("town") == "TOA PAYOH")
)
Parenthesise every condition. & binds tighter than ==, so unparenthesised combinations parse wrong.

is_in: cleaner than chained ORs


# Instead of:
#   (town == "BISHAN") | (town == "TOA PAYOH")
#   | (town == "QUEENSTOWN") | (town == "BUKIT MERAH")

central_towns = ["BISHAN", "TOA PAYOH", "QUEENSTOWN", "BUKIT MERAH"]
central = hdb.filter(pl.col("town").is_in(central_towns))

print(f"Central towns: {central.height:,}")

Rule of thumb: three or more OR values → reach for is_in.

with_columns: derive new features


hdb = hdb.with_columns(
    # price per sqm — normalised measure
    (pl.col("resale_price") / pl.col("floor_area_sqm"))
        .alias("price_per_sqm"),
    # year as integer
    pl.col("month").str.slice(0, 4).cast(pl.Int32)
        .alias("year"),
)

print(hdb.select("month", "year", "price_per_sqm").head(3))

Add multiple columns in one call — more efficient than chaining.

Conditional columns: when / then


hdb = hdb.with_columns(
    pl.when(pl.col("resale_price") < 350_000)
      .then(pl.lit("budget"))
    .when(pl.col("resale_price") < 500_000)
      .then(pl.lit("mid_range"))
    .when(pl.col("resale_price") < 700_000)
      .then(pl.lit("premium"))
    .otherwise(pl.lit("luxury"))
    .alias("price_tier")
)

pl.lit(...) wraps a constant. Without it, Polars would look for a column named "budget".

Method chaining: the pipeline

The chain in code


recent_premium = (
    hdb
    .filter(pl.col("year") >= 2020)
    .filter(pl.col("price_tier").is_in(["premium", "luxury"]))
    .select(
        "transaction_date", "town", "flat_type",
        "price_per_sqm", "resale_price",
    )
    .sort("resale_price", descending=True)
)

Each line consumes the DataFrame from the line above and produces the DataFrame for the line below. Reads top-to-bottom like a recipe.

Boolean algebra under the hood

\[ A \land B \iff A \text{ and } B \text{ are both True} \] \[ A \lor B \iff \text{at least one of } A, B \text{ is True} \]
$A$$B$$A \land B$$A \lor B$
TTTT
TFFT
FTFT
FFFF

Polars evaluates these truth tables across hundreds of thousands of rows at once.

🔵 Theory: De Morgan's laws

\[ \lnot(A \land B) \equiv (\lnot A) \lor (\lnot B) \] \[ \lnot(A \lor B) \equiv (\lnot A) \land (\lnot B) \]

"Not (A and B)" = "not A or not B". Useful when you flip a filter.

In Polars: ~pl.col("town").is_in([...]) is the clean way to say "not in this set". The tilde ~ is boolean negation.

Four traps to avoid

1. Missing parentheses. & binds tighter than ==. Always wrap each condition.
2. Python and/or. Do not use them on Polars expressions. Use & and |.
3. Forgetting to assign. hdb.filter(...) alone throws the result away. Write hdb = hdb.filter(...).
4. Bare strings in then. Wrap literals with pl.lit(...) or Polars will look for a column with that name.
