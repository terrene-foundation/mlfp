What Went Wrong?

FOUNDATIONS

Dashboard Blindness

  • Aggregation hides outliers — a mean over 50,000 transactions masks 200 anomalous ones
  • Pre-built charts answer yesterday's questions — they cannot ask new ones
  • No distribution view — the dashboard showed a single number, not a histogram

What EDA Would Have Caught

  • A bimodal distribution in the price column
  • Temporal clustering — low prices all in a 2-week window
  • Flat type correlation — only 3-room and 4-room affected
The Fix: 
df.filter(pl.col("town") == "QUEENSTOWN")
  .group_by("month")
  .agg(pl.col("resale_price").median())

Creating Computed Columns

FOUNDATIONS 
# with_columns adds new columns (or overwrites existing ones)
df = df.with_columns(
    # Price per square metre
    (pl.col("resale_price") / pl.col("floor_area_sqm"))
    .alias("price_per_sqm"),

    # Remaining lease in years (from months)
    (pl.col("remaining_lease_months") / 12)
    .round(1)
    .alias("remaining_lease_years"),

    # High-value flag
    (pl.col("resale_price") > 600_000)
    .alias("is_high_value")
)

print(df.select("town", "price_per_sqm", "remaining_lease_years", "is_high_value").head())
Why expressions, not loops? Polars expressions run on the entire column at once (vectorised). Processing 15 million rows takes milliseconds, not minutes. This is why Polars beats pandas for large datasets.

Method Chaining

FOUNDATIONS

Chaining = Readability

# Instead of 5 separate variables...
result = (
    df
    .filter(pl.col("town") == "TAMPINES")
    .with_columns(
        (pl.col("resale_price") /
         pl.col("floor_area_sqm"))
        .alias("price_per_sqm")
    )
    .select(
        "month", "flat_type",
        "resale_price", "price_per_sqm"
    )
    .sort("price_per_sqm",
          descending=True)
    .head(10)
)
print(result)

The Pattern

Each method returns a new DataFrame, so you can chain the next method directly:

  1. filter — select rows
  2. with_columns — compute new columns
  3. select — choose columns
  4. sort — order results
  5. head — limit output
Tip: Wrap the chain in parentheses ( ) so you can split across lines without backslashes.

Vectorised Conditionals in Polars

THEORY

Python if/else vs Polars when/then

# Python if/else: works on ONE value
def classify(price):
    if price > 1_000_000:
        return "Million"
    else:
        return "Standard"

# Polars when/then: works on ENTIRE column
df = df.with_columns(
    pl.when(pl.col("resale_price") > 1_000_000)
      .then(pl.lit("Million"))
      .when(pl.col("resale_price") > 700_000)
      .then(pl.lit("Premium"))
      .otherwise(pl.lit("Standard"))
      .alias("price_category")
)

Why when/then?

  • Speed: Processes millions of rows in milliseconds
  • No loops: The entire column is categorised at once
  • Chainable: Multiple conditions without nested if/else
Rule: Use Python if/else for code-level decisions (which function to call). Use Polars when/then for data-level decisions (categorise rows).
