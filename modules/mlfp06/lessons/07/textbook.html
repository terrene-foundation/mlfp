Lesson 6.7 AI Governance Engineering

Governance is not a committee discussion or an ethics position paper. It is code — access controls that deny unauthorised access, operating envelopes that restrict agent behaviour, budget cascading that prevents runaway costs, and audit trails that prove compliance. Today you implement all four with PACT: D/T/R addressing, GovernanceEngine, GovernedSupervisor, and the five-dimension ConstraintEnvelopeConfig. The result: a multi-agent system where every tool call is checked, every budget is enforced, and every decision is logged.

Why This Matters

Singapore's Monetary Authority (MAS) requires financial institutions to explain every AI-driven decision that affects customers. A recommendation engine that denies a loan must log why. An agent that executes a trade must record who authorised it, what envelope it operated under, and how much it was allowed to spend. "The AI decided" is not an acceptable audit response.

Without governance engineering, you get one of two failure modes. Either the system is wide open — any agent can access any data and spend any amount — which is a security and compliance disaster. Or governance is bolted on as an afterthought — a brittle layer of if-statements that breaks every time the system changes. PACT makes governance a first-class component: declared, compiled, enforced, and tested.

Core Concepts

D/T/R Addressing

PACT's addressing grammar breaks identity into three dimensions:

DimensionMeaningExample
Department (D)Top-level organisational unitD1 = mortgage-risk
Task (T)A task nested inside a department or teamT1 = assessment
Role (R)The responsible role that owns a D or T leafR1 = the head or responsible

A role address is a dash-delimited path like D1-R1-T1-R1. The grammar rule: every D or T MUST be immediately followed by exactly one R. The GovernanceEngine is constructed from a loaded org YAML (load_org_yaml) and resolves access by walking the tree: the responsible at a deeper level inherits the envelope of its parent role, tightened by its own envelope.

Foundations

Think of D/T/R as a postal address for permissions. The department is the street, the task is the building, the role is the apartment. When a request arrives, the system checks whether that address has the mailbox (envelope) for this type of mail (action). The grammar is strict: every D or T is immediately followed by exactly one R.

Operating Envelopes

An operating envelope defines the boundaries within which a role can operate. If D/T/R addressing answers "who is asking", envelopes answer "what may they do". A modern ConstraintEnvelopeConfig has five canonical dimensions, plus a confidentiality clearance and a delegation-depth cap:

Monotonic Tightening

The critical safety property of envelopes: they can only get stricter, never looser. A supervisor agent that allocates a $10 budget to a child agent cannot later increase it to $20. A task envelope that excludes deploy_model cannot later add it. This is monotonic tightening — trust flows downward and narrows at each level.

Theory

Monotonic tightening is a lattice property. If we define a partial order on envelopes where $E_1 \leq E_2$ means "$E_1$ is at least as restrictive as $E_2$", then the set of all envelopes forms a lattice with the most restrictive envelope (deny everything) at the bottom and the most permissive (allow everything) at the top. Child envelopes must always be below or equal to their parent in this lattice — they can only move downward.

Enforcement Modes

PACT supports three enforcement modes for governance checks:

ModeBehaviourUse case
blockDeny access immediately. Fail-closed.Production systems. This is the default.
warnLog violation, allow access.Development and testing. See what would be blocked.
auditAllow access, log for review.Shadow mode before enabling block. Compliance review.
Advanced

The standard deployment progression is: audit for one week (collect baseline), review audit logs for false positives, adjust rules, switch to warn for another week (verify no critical workflows break), then block for production. This gradual rollout prevents governance rules from accidentally blocking legitimate operations.

Mathematical Foundations: Budget Cascading

Budget cascading is a constraint propagation problem. A parent agent with budget $B$ allocates portions to $n$ child agents:

$$ \sum_{i=1}^{n} b_i \leq B \quad \text{where each } b_i \geq 0 $$
$B$ — parent budget $b_i$ — allocation to child $i$ $n$ — number of children

The sum of all child budgets cannot exceed the parent's total. Each child can further subdivide its allocation, but the total at every level is bounded. When a child exhausts its budget, it stops gracefully and returns partial results. The parent can then decide whether to reallocate remaining budget from other children.

For a three-level hierarchy (supervisor, team leads, workers) with supervisor budget $B = \$50$:

$$ B_{\text{supervisor}} = \$50 \geq \underbrace{b_{\text{lead}_1} + b_{\text{lead}_2}}_{= \$20 + \$20} + b_{\text{reserve}} $$ $$ b_{\text{lead}_1} = \$20 \geq \underbrace{b_{\text{worker}_1} + b_{\text{worker}_2} + b_{\text{worker}_3}}_{= \$6 + \$6 + \$6} + b_{\text{lead reserve}} $$

Each allocation is a hard ceiling. Workers cannot request additional budget from the supervisor — they can only use what their team lead allocated. This prevents any single branch from consuming the entire budget.

Foundations

Budget cascading works like a company's departmental budgets. The CEO allocates to VPs; VPs allocate to directors; directors allocate to teams. Each level can only spend what it was given. No one can reach up to grab more money. If your team runs out, you stop and report back.

Worked Example: Governed Multi-Agent System

Goal

Build a governed multi-agent system for HDB price analysis in Singapore. A supervisor agent delegates to three specialists: DataScientist, ModelSelector, and ReportWriter. Each agent has a defined role, budget, and task envelope. Unauthorised access is blocked. Budget overruns are prevented. Every decision is logged.

Step 1 — Declare the org in YAML and load it.

Modern PACT loads the organisation from a YAML file. The grammar is strict: every D (department) or T (task) is immediately followed by exactly one R (responsible role). Addresses are dash-delimited, e.g. D1-R1-T1-R1. 

# org/mortgage_risk.yaml
org_id: mortgage_risk_assessment
departments:
  - id: D1
    name: mortgage-risk
    head: { id: R1, name: risk-supervisor }
    tasks:
      - id: T1
        name: assessment
        responsible: { id: R1, name: data-scientist }
      - id: T2
        name: selection
        responsible: { id: R1, name: model-selector }
      - id: T3
        name: reporting
        responsible: { id: R1, name: report-writer }
from pact import GovernanceEngine, load_org_yaml

loaded = load_org_yaml("org/mortgage_risk.yaml")

compiled = engine.get_org()
# compiled.nodes keys are the dash-delimited addresses
print(list(compiled.nodes.keys()))
# → ['D1-R1', 'D1-R1-T1-R1', 'D1-R1-T2-R1', 'D1-R1-T3-R1']
Step 2 — Define operating envelopes for each role.

Envelopes are ConstraintEnvelopeConfig instances with the five canonical dimensions (financial, operational, temporal, data access, communication) plus a confidentiality clearance and a delegation-depth cap. Step 3 — Attach envelopes to roles and verify the budget sum.

Budget cascading in the modern API is a property of the envelope tree: the parent holds the total, children hold shares, and validate_tightening catches any child that tries to exceed its parent's cap. You still check the sum manually — the framework won't silently reject over-allocation, it will just refuse to run the offending delegation at verify time. 

# Attach each envelope to its D/T/R address
engine.set_role_envelope(RoleEnvelope(
    id="supervisor_role_env",
    defining_role_address="D1-R1",
    target_role_address="D1-R1",
    envelope=supervisor_env,
))

for task_id, env in [("T1", data_scientist_env),
                     ("T2", model_selector_env),
                     ("T3", report_writer_env)]:
    engine.set_role_envelope(RoleEnvelope(
        id=f"{env.id}_role",
        defining_role_address="D1-R1",
        target_role_address=f"D1-R1-{task_id}-R1",
        envelope=env,
    ))

# Verify the child budgets don't exceed the parent
child_total = sum(env.financial.max_spend_usd
                  for env in [data_scientist_env, model_selector_env, report_writer_env])
assert child_total <= supervisor_env.financial.max_spend_usd, "over-allocation"
print(f"Total allocated to children: ${child_total:.2f}")
print(f"Supervisor reserve: ${supervisor_env.financial.max_spend_usd - child_total:.2f}")
Step 4 — Construct the GovernedSupervisor.

GovernedSupervisor replaces the legacy PactGovernedAgent wrapper. It is not a wrapper — it is a standalone planning supervisor that accepts a model, budget, tool list, and clearance string, and runs a two-layer protocol where your execute_node callback performs the actual LLM work. 

from kaizen_agents import GovernedSupervisor

governed_supervisor = GovernedSupervisor(
    model="gpt-4o-mini",
    budget_usd=10.00,
    tools=["delegate", "allocate_budget", "aggregate_results"],
    data_clearance="confidential",  # canonical: public/restricted/confidential/secret/top_secret
)

# execute_node is YOUR callback — runs the LLM (or an offline stub for tests)
async def executor(spec, inputs):
    """Dispatch to the right specialist given the planned step."""
    if spec.role == "data-scientist":
        return await run_data_scientist(inputs)
    if spec.role == "model-selector":
        return await run_model_selector(inputs)
    return await run_report_writer(inputs)

result = await governed_supervisor.run(
    objective="Assess default risk for HDB flat portfolio",
    execute_node=executor,
)
# result.success         → bool
# result.budget_consumed → float
# result.audit_trail     → list[dict] (every plan, call, and verdict)
Step 5 — Test access controls with verify_action. 
# Verify access: data scientist can explore data
verdict = engine.verify_action(
    role_address="D1-R1-T1-R1",
    action="explore_data",
    context={"cost": 0.10, "data_classification": "restricted"},
)
assert verdict.allowed
print(f"{verdict.level}: {verdict.reason}")
# → "allowed: action 'explore_data' in envelope.operational.allowed_actions"

# Verify denial: data scientist cannot deploy models
verdict = engine.verify_action(
    role_address="D1-R1-T1-R1",
    action="deploy_model",
    context={},
)
assert not verdict.allowed
print(f"{verdict.level}: {verdict.reason}")
# → "blocked: action 'deploy_model' in envelope.operational.denied_actions"

# Fail-closed: unknown role address denies immediately (Step 1 of the 5-step algorithm)
verdict = engine.verify_action(
    role_address="D99-R99-T99-R99",
    action="any_action",
    context={},
)
assert not verdict.allowed
assert verdict.level == "blocked"

trail = governed_supervisor.audit.to_list()

Every governance decision — granted and denied — is logged with a timestamp, the D/T/R address, the resource, the decision, and the reason. This audit trail is what MAS (or any regulator) will review.

Clearance Levels

PACT's confidentiality clearance adds graduated access based on data sensitivity. Rather than binary allow/deny, clearance levels enable progressive trust: a new role starts with a PUBLIC envelope (read-only, low budget) and earns higher clearance through demonstrated reliability. Each envelope carries a confidentiality_clearance field of type ConfidentialityLevel.

LevelNameAccessTypical budget
1PUBLICRead-only data access$1
2RESTRICTEDRead + compute (statistics, predictions)$5
3CONFIDENTIALRead + compute + train models$20
4SECRETFull access including deploy$50

Promotion in the modern API is not a separate object — it is an envelope replacement on the role. The supervisor (defining_role_address) issues a new RoleEnvelope at a higher clearance, and the framework enforces monotonic tightening structurally: a child envelope may never exceed the parent envelope on any dimension. 

from pact import (
    RoleEnvelope, ConstraintEnvelopeConfig, FinancialConstraintConfig,
    OperationalConstraintConfig, TemporalConstraintConfig,
    DataAccessConstraintConfig, CommunicationConstraintConfig,
    ConfidentialityLevel,
)

def build_envelope(level: ConfidentialityLevel, budget_usd: float,
                    actions: list[str]) -> ConstraintEnvelopeConfig:
    return ConstraintEnvelopeConfig(
        id=f"env_{level.name.lower()}",
        description=f"{level.name} clearance",
        confidentiality_clearance=level,
        financial=FinancialConstraintConfig(max_spend_usd=budget_usd),
        operational=OperationalConstraintConfig(allowed_actions=actions),
        temporal=TemporalConstraintConfig(),
        data_access=DataAccessConstraintConfig(),
        communication=CommunicationConstraintConfig(),
        max_delegation_depth=0,
    )

# New role starts at PUBLIC (Level 1)
initial = build_envelope(ConfidentialityLevel.PUBLIC, 1.00, ["read_data"])
engine.set_role_envelope(RoleEnvelope(
    id="new_role_env_v1",
    defining_role_address="D1-R1",
    target_role_address="D1-R1-T1-R1",
    envelope=initial,
))

# After 100 successful operations with no violations, promote to RESTRICTED
promoted = build_envelope(
    ConfidentialityLevel.RESTRICTED, 5.00,
    ["read_data", "compute_statistics", "run_predictions"],
)
engine.set_role_envelope(RoleEnvelope(
    id="new_role_env_v2",
    defining_role_address="D1-R1",
    target_role_address="D1-R1-T1-R1",
    envelope=promoted,
))
# The supervisor is the defining role, so this replacement is authorised.
# If an unauthorised role tried to promote, verify_action would deny.
Advanced

Clearance promotion can be automated with a verification gradient. Define a scoring function that measures agent reliability (successful operations / total operations), governance compliance (violations / total checks), and cost efficiency (actual spend / allocated budget). When the score exceeds a threshold over a defined period, the agent is eligible for promotion. Demotion is immediate on any critical violation — trust is earned slowly and lost quickly.

Governance Testing Patterns

Governance testing is not optional — it is the most important testing you do. Functional tests verify that the system works. Governance tests verify that the system fails safely. Here are the four patterns: 

import pytest
from pact import (
    RoleEnvelope, ConstraintEnvelopeConfig, FinancialConstraintConfig,
    OperationalConstraintConfig, TemporalConstraintConfig,
    DataAccessConstraintConfig, CommunicationConstraintConfig,
    ConfidentialityLevel, MonotonicTighteningError,
)

# Pattern 1: Verify access denial
def test_data_scientist_cannot_deploy():
    verdict = engine.verify_action(
        role_address="D1-R1-T1-R1",
        action="deploy_model",
        context={},
    )
    assert not verdict.allowed
    assert verdict.level == "blocked"

# Pattern 2: Verify fail-closed on unknown roles
def test_unknown_role_is_denied():
    verdict = engine.verify_action(
        role_address="D99-R99",         # does not exist in compiled org
        action="read_data",
        context={},
    )
    assert not verdict.allowed
    assert verdict.level == "blocked"

# Pattern 3: Verify monotonic tightening — child cannot exceed parent
def test_child_envelope_cannot_loosen_parent():
    parent = ConstraintEnvelopeConfig(
        id="parent", description="",
        confidentiality_clearance=ConfidentialityLevel.RESTRICTED,
        financial=FinancialConstraintConfig(max_spend_usd=5.00),
        operational=OperationalConstraintConfig(allowed_actions=["read"]),
        temporal=TemporalConstraintConfig(),
        data_access=DataAccessConstraintConfig(),
        communication=CommunicationConstraintConfig(),
        max_delegation_depth=1,
    )
    child = ConstraintEnvelopeConfig(
        id="child", description="",
        confidentiality_clearance=ConfidentialityLevel.CONFIDENTIAL,  # ← tries to escalate
        financial=FinancialConstraintConfig(max_spend_usd=10.00),     # ← exceeds parent
        operational=OperationalConstraintConfig(allowed_actions=["read"]),
        temporal=TemporalConstraintConfig(),
        data_access=DataAccessConstraintConfig(),
        communication=CommunicationConstraintConfig(),
        max_delegation_depth=1,
    )
    with pytest.raises(MonotonicTighteningError):
        RoleEnvelope.validate_tightening(parent, child)

# Pattern 4: Verify budget exhaustion stops the supervisor
async def test_budget_stops_supervisor():
    from kaizen_agents import GovernedSupervisor
    tiny = GovernedSupervisor(
        model="gpt-4o-mini",
        budget_usd=0.01,                 # tiny budget
        tools=["answer_question"],
        data_clearance="public",
    )
    async def expensive_executor(spec, inputs):
        return {"result": "...", "cost": 0.02,  # exceeds $0.01
                "prompt_tokens": 100, "completion_tokens": 50}
    result = await tiny.run(objective="analyse portfolio",
                             execute_node=expensive_executor)
    assert not result.success
    assert result.budget_consumed <= 0.01 or "budget" in result.failure_reason.lower()

# Pattern 5: Verify tamper-evident audit chain
def test_audit_chain_is_valid(governed_supervisor):
    # Run some operations that populate the trail
    trail = governed_supervisor.audit.to_list()
    assert len(trail) >= 1
    assert governed_supervisor.audit.verify_chain(), "audit chain is broken"
Critical principle

Governance tests are negative tests. They test what does NOT happen. Every assert not verdict.allowed and every pytest.raises(MonotonicTighteningError) is a guardrail that prevents the system from doing something dangerous. If these tests fail, it means your governance has a hole. Treat governance test failures as critical — they are security vulnerabilities.

Common Mistakes

Mistake 1: Fail-open governance

If the governance check fails (network error, config issue, missing rules), deny access. Never default to allow. A system that allows everything when governance breaks is worse than a system with no governance at all, because it creates a false sense of security. PACT's default is enforcement="block" with fail-closed semantics.

Mistake 2: Testing only the happy path

"The analyst can read data" is a happy path test. The governance tests that matter are: "The analyst cannot deploy models", "The analyst cannot exceed budget", "The analyst cannot loosen their envelope". Test what is denied, not just what is granted.

Mistake 3: No audit trail

Compliance requires proving every access decision was checked. Log every can_access() call — both grants and denials. The audit trail is not for debugging; it is for the regulator. MAS, PDPA, and GDPR all require demonstrable access control.

Mistake 4: Governance as philosophy instead of code

"We have an AI ethics framework" is not governance. Governance is engine.can_access(addr, resource) returning False and the tool call being blocked. If it is not in code, it is not enforced. If it is not tested, it does not work. This lesson is engineering, not philosophy.

Reflection

Governance is the hardest lesson in the programme to appreciate — until you deploy without it and face the consequences. An ungoverned agent with access to a production database can delete records, leak personal data, or spend thousands of dollars in minutes. An ungoverned multi-agent system can cascade errors across agents with no way to trace what happened.

What you built today — D/T/R addressing, operating envelopes, budget cascading, audit trails, governance tests — is the difference between a prototype and a production system. Prototypes work when everything goes right. Production systems work when everything goes wrong. PACT ensures that "wrong" is bounded, logged, and recoverable.

In Lesson 6.8, you deploy this governed system via Nexus — three channels, authenticated, rate-limited, monitored. That is the capstone. That is the finish line.
