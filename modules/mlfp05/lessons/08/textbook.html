Lesson 5.8 Reinforcement Learning

All deep learning so far learns from static data — images, text, sequences, graphs. Reinforcement learning (RL) learns from interaction with an environment. The agent takes actions, receives rewards, and learns a policy that maximises cumulative reward. Today you derive the Bellman equations from first principles, implement DQN (Deep Q-Network) with a target network for a discrete-action problem, implement PPO (Proximal Policy Optimization) with a clipped objective for a continuous-action problem, build a custom Gymnasium environment for Singapore supply chain optimisation, and understand the bridge from PPO to RLHF — the technique that aligns LLMs with human preferences in M6.

Why This Matters

Supervised learning needs a dataset: pairs of inputs and correct outputs. But many real-world problems have no "correct output" — only a reward signal that tells you how well you did. A Singapore port authority wants to minimise container ship waiting time — there is no labelled dataset of optimal schedules, only the daily reality of ships, berths, cranes, and delays. A DBS dynamic pricing engine needs to set interest rates that balance profit and customer retention — the "correct" rate depends on market conditions that change hourly.

RL learns through trial and error: try an action, observe the consequence, adjust the policy, repeat. This makes it uniquely suited to sequential decision-making where the optimal action depends on state and future consequences — not just the current input.

More importantly for your M6 journey: RLHF (Reinforcement Learning from Human Feedback) uses PPO to align LLMs with human preferences. Understanding PPO here makes RLHF (and its successor DPO) in Lesson 6.3 straightforward instead of mysterious.

Foundations

Think of RL as learning to ride a bicycle. Nobody gives you a dataset of "correct handlebar angles" — instead, you try steering, fall over (negative reward), adjust, try again, and eventually learn a policy that keeps you upright. The reward signal (staying balanced vs falling) is the only feedback. Every RL problem has this structure: an agent that acts in an environment, receives rewards, and improves through experience.

Value-based vs policy-based methods

Value-based (DQN)

Learn $Q(s, a)$: the value of taking action $a$ in state $s$. Select the action with the highest Q-value. Works for discrete action spaces (choose from a finite set).

Examples: buy/hold/sell a stock, route a package to warehouse A/B/C.

Policy-based (PPO)

Learn $\pi(a|s)$ directly: a probability distribution over actions given state. Sample actions from the distribution. Works for continuous action spaces (any real number).

Examples: set an interest rate (any value in [0%, 15%]), control a robot arm (joint angles).

Math Foundations

The Bellman expectation equation

The value of a state is the expected immediate reward plus the discounted value of the next state. This recursive definition is the foundation of all RL algorithms:

$$ V^\pi(s) = \mathbb{E}_\pi\bigl[R_{t+1} + \gamma\, V^\pi(S_{t+1}) \mid S_t = s\bigr] $$
$V^\pi(s)$ — value of state $s$ under policy $\pi$ $R_{t+1}$ — immediate reward $\gamma$ — discount factor (typically 0.99) $\mathbb{E}_\pi$ — expectation under policy $\pi$

In words: "The value of being in state $s$ and following policy $\pi$ is the reward I expect to get immediately, plus the discounted value of wherever I end up next." This is recursive — to know $V(s)$ you need $V(s')$ — which is why RL algorithms iterate until convergence.

The Bellman optimality equation

The optimal policy $\pi^*$ selects the action that maximises the action-value:

$$ Q^*(s, a) = \mathbb{E}\bigl[R_{t+1} + \gamma\,\max_{a'} Q^*(S_{t+1}, a') \mid S_t = s, A_t = a\bigr] $$
$Q^*(s,a)$ — optimal action-value function $\max_{a'}$ — best action in the next state $R_{t+1} + \gamma\,\max_{a'} Q^*$ — bootstrap target

The key difference from the expectation equation: the max over actions. The optimal Q-function assumes you will make the best possible decision at every future step.

Theory

The Bellman equations are instances of dynamic programming — breaking an optimisation problem into overlapping sub-problems. Each state's value depends on future states' values, creating a system of equations. In tabular settings (small state space), you solve these equations directly. In deep RL (large or continuous state spaces), you approximate the value function with a neural network and solve iteratively via gradient descent.

Connecting to M5.8 → M6.3

In RLHF, the "environment" is a reward model that scores LLM outputs. The "state" is the prompt. The "action" is the generated text. The "reward" is the human preference score. PPO fine-tunes the LLM to maximise this reward while staying close to the original policy. Understanding PPO here makes RLHF mechanical, not magical.

DQN — Deep Q-Network

DQN (Mnih et al., 2015) approximates $Q^*(s, a)$ with a neural network. Two innovations make this stable:

  1. Experience replay buffer: store past transitions $(s, a, r, s')$ and sample random mini-batches for training. This breaks the correlation between consecutive samples that would otherwise destabilise SGD.
  2. Target network: a slow-updating copy of the Q-network used to compute the target $r + \gamma \max_{a'} Q_{\text{target}}(s', a')$. Without it, the target changes with every gradient step, creating a moving-target problem.

DQN loss function

$$ \mathcal{L}_{\text{DQN}} = \mathbb{E}_{(s,a,r,s') \sim \mathcal{D}}\Bigl[\bigl(r + \gamma\,\max_{a'} Q(s', a'; \theta^-) - Q(s, a; \theta)\bigr)^2\Bigr] $$
$\mathcal{D}$ — replay buffer $\theta^-$ — frozen target network params $\theta$ — online Q-network params (updated)

DQN implementation

import gymnasium as gym
import numpy as np
import torch
import torch.nn as nn
from collections import deque
import random

class QNetwork(nn.Module):
    def __init__(self, state_dim, action_dim):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(state_dim, 128), nn.ReLU(),
            nn.Linear(128, 128), nn.ReLU(),
            nn.Linear(128, action_dim)
        )

    def forward(self, x):
        return self.net(x)

class ReplayBuffer:
    def __init__(self, capacity=10000):
        self.buffer = deque(maxlen=capacity)

    def push(self, state, action, reward, next_state, done):
        self.buffer.append((state, action, reward, next_state, done))

    def sample(self, batch_size):
        batch = random.sample(self.buffer, batch_size)
        states, actions, rewards, next_states, dones = zip(*batch)
        return (torch.FloatTensor(np.array(states)),
                torch.LongTensor(actions),
                torch.FloatTensor(rewards),
                torch.FloatTensor(np.array(next_states)),
                torch.FloatTensor(dones))

class DQNAgent:
    def __init__(self, state_dim, action_dim, lr=1e-3, gamma=0.99):
        self.q_net = QNetwork(state_dim, action_dim)
        self.target_net = QNetwork(state_dim, action_dim)
        self.target_net.load_state_dict(self.q_net.state_dict())
        self.optimizer = torch.optim.Adam(self.q_net.parameters(), lr=lr)
        self.buffer = ReplayBuffer()
        self.gamma = gamma
        self.epsilon = 1.0  # Exploration rate

    def select_action(self, state):
        if random.random() < self.epsilon:
            return random.randint(0, 1)  # Explore
        with torch.no_grad():
            q_values = self.q_net(torch.FloatTensor(state))
            return q_values.argmax().item()  # Exploit

    def train_step(self, batch_size=64):
        if len(self.buffer.buffer) < batch_size:
            return
        states, actions, rewards, next_states, dones = \
            self.buffer.sample(batch_size)

        # Current Q-values
        q_values = self.q_net(states).gather(1, actions.unsqueeze(1))

        # Target Q-values (from frozen target network)
        with torch.no_grad():
            next_q = self.target_net(next_states).max(1)[0]
            targets = rewards + self.gamma * next_q * (1 - dones)

        loss = nn.MSELoss()(q_values.squeeze(), targets)
        self.optimizer.zero_grad()
        loss.backward()
        self.optimizer.step()

    def update_target(self):
        self.target_net.load_state_dict(self.q_net.state_dict())

PPO — Proximal Policy Optimization

DQN learns which actions are valuable (value-based). PPO directly learns which actions to take (policy-based). It maintains two networks: an actor (policy $\pi_\theta$) and a critic (value function $V_\phi$).

The clipped objective

PPO's key innovation is the clipped surrogate objective that prevents destructively large policy updates:

$$ L^{\text{CLIP}} = \mathbb{E}_t\Bigl[\min\bigl(r_t(\theta)\,\hat{A}_t,\;\text{clip}\bigl(r_t(\theta),\,1-\epsilon,\,1+\epsilon\bigr)\,\hat{A}_t\bigr)\Bigr] $$
$r_t(\theta) = \frac{\pi_\theta(a_t | s_t)}{\pi_{\theta_{\text{old}}}(a_t | s_t)}$ — probability ratio $\hat{A}_t$ — advantage estimate (how much better than average) $\epsilon \approx 0.2$ — clip range

The probability ratio $r_t$ measures how much the new policy differs from the old policy. If $r_t$ is too far from 1 (the policy changed too much), the clipping prevents the objective from incentivising further change. This keeps the policy update proximal — close to the old policy — which prevents catastrophic forgetting.

Theory

PPO is a simplification of TRPO (Trust Region Policy Optimization). TRPO constrains the KL divergence between old and new policies, which requires computing second-order derivatives (expensive). PPO achieves a similar effect with the simple clipping trick — first-order optimisation only. This makes PPO the de facto standard for policy optimisation, including in RLHF where it is used to align LLMs (M6.3).

The advantage function

The advantage $\hat{A}_t$ measures how much better an action was compared to the average action from that state. PPO uses Generalised Advantage Estimation (GAE):

$$ \hat{A}_t = \sum_{l=0}^{T-t} (\gamma\lambda)^l \delta_{t+l} \qquad\text{where}\qquad \delta_t = r_t + \gamma V(s_{t+1}) - V(s_t) $$
$\delta_t$ — TD error (temporal difference) $\lambda \approx 0.95$ — GAE parameter (bias-variance tradeoff) $V(s)$ — critic's value estimate

Custom Gymnasium Environments

Gymnasium (the successor to OpenAI Gym) provides a standard interface for RL environments. You subclass gymnasium.Env and implement four methods: 

import gymnasium as gym
from gymnasium import spaces
import numpy as np

class SupplyChainEnv(gym.Env):
    """Singapore supply chain optimisation environment.

    State: [inventory_level, demand_forecast, lead_time, days_remaining]
    Action: order_quantity (continuous, 0 to 100 units)
    Reward: -(holding_cost + stockout_penalty + order_cost)
    """

    def __init__(self):
        super().__init__()
        # Continuous action: how much to order
        self.action_space = spaces.Box(
            low=0, high=100, shape=(1,), dtype=np.float32
        )
        # State: 4 continuous values
        self.observation_space = spaces.Box(
            low=0, high=np.inf, shape=(4,), dtype=np.float32
        )
        self.holding_cost_per_unit = 0.5   # SGD per unit per day
        self.stockout_penalty = 10.0        # SGD per unit short
        self.order_cost_per_unit = 2.0      # SGD per unit ordered
        self.reset()

    def reset(self, seed=None, options=None):
        super().reset(seed=seed)
        self.inventory = 50.0
        self.day = 0
        self.max_days = 90  # One quarter
        state = self._get_state()
        return state, {}

    def step(self, action):
        order_qty = float(np.clip(action[0], 0, 100))

        # Simulate demand (stochastic)
        demand = np.random.normal(10, 3)  # Mean 10 units/day
        demand = max(0, demand)

        # Update inventory
        self.inventory += order_qty - demand
        self.day += 1

        # Compute reward (negative cost)
        holding = max(0, self.inventory) * self.holding_cost_per_unit
        stockout = max(0, -self.inventory) * self.stockout_penalty
        ordering = order_qty * self.order_cost_per_unit
        reward = -(holding + stockout + ordering)

        # Clamp inventory (cannot go below -20)
        self.inventory = max(-20, self.inventory)

        done = self.day >= self.max_days
        state = self._get_state()
        return state, reward, done, False, {}

    def _get_state(self):
        demand_forecast = 10.0  # Simple forecast
        lead_time = 2.0         # Days until order arrives
        days_remaining = self.max_days - self.day
        return np.array([
            self.inventory, demand_forecast,
            lead_time, days_remaining
        ], dtype=np.float32)

Worked Example: DQN for Customer Churn Prevention

Goal

Build a DQN agent that decides monthly retention actions for Singapore telco customers (3 discrete actions: no action, discount offer, personal call). The agent observes customer state (tenure, monthly charges, contract type, usage patterns) and receives reward +1 for retained customers and -5 for churned customers.

Step 1. Define the environment. 
class ChurnPreventionEnv(gym.Env):
    """Each step = one month. Agent picks retention action per customer."""

    def __init__(self, customer_data):
        super().__init__()
        self.action_space = spaces.Discrete(3)  # no_action, discount, call
        self.observation_space = spaces.Box(
            low=0, high=1, shape=(8,), dtype=np.float32
        )
        self.customers = customer_data
        self.action_costs = [0, 5, 15]  # SGD per action
        self.reset()

    def step(self, action):
        # Compute churn probability (influenced by action)
        base_churn_prob = self._base_churn_prob()
        action_effect = [0, -0.15, -0.25][action]  # Discount/call reduces churn
        churn_prob = max(0, min(1, base_churn_prob + action_effect))

        churned = np.random.random() < churn_prob
        cost = self.action_costs[action]
        reward = (-5 if churned else 1) - cost / 10  # Normalise cost

        self.month += 1
        done = self.month >= 12 or churned
        return self._get_state(), reward, done, False, {}
Step 2. Train the DQN agent. 
env = ChurnPreventionEnv(customer_data)
agent = DQNAgent(state_dim=8, action_dim=3)

for episode in range(1000):
    state, _ = env.reset()
    total_reward = 0

    while True:
        action = agent.select_action(state)
        next_state, reward, done, _, _ = env.step(action)
        agent.buffer.push(state, action, reward, next_state, float(done))
        agent.train_step()
        total_reward += reward
        state = next_state
        if done:
            break

    # Decay exploration
    agent.epsilon = max(0.01, agent.epsilon * 0.995)

    # Update target network periodically
    if episode % 10 == 0:
        agent.update_target()

    if episode % 100 == 0:
        print(f"Episode {episode}: reward={total_reward:.1f}, "
              f"epsilon={agent.epsilon:.3f}")
Step 3. Evaluate the learned policy. 
# Evaluate: what actions does the trained agent take?
agent.epsilon = 0  # No exploration during evaluation
state, _ = env.reset()
actions_taken = []

while True:
    action = agent.select_action(state)
    actions_taken.append(action)
    state, _, done, _, _ = env.step(action)
    if done:
        break

# Expected: agent uses discount offers for at-risk customers,
# no action for loyal customers, and personal calls for
# high-value customers about to churn.

DPO (Direct Preference Optimization) in M6.3 bypasses the reward model entirely — it derives the optimal policy directly from preference pairs, removing the RL loop. This is why DPO is simpler to implement: no reward model, no PPO, no replay buffer. But understanding PPO makes DPO's derivation transparent.

Common Mistakes

Mistake 1: Reward shaping too complex

Start with simple, sparse rewards: +1 for success, -1 for failure. Complex reward functions with many terms create unexpected optima — the agent finds ways to maximise reward that do not correspond to the desired behaviour (reward hacking). Add shaping terms only if the agent fails to learn with simple rewards.

Mistake 2: No reward normalisation

PPO needs normalised advantages — subtract mean and divide by standard deviation. Without normalisation, advantages with large magnitude cause instability: the probability ratio $r_t$ swings wildly, overwhelming the clipping mechanism.

Mistake 3: Too-small replay buffer

DQN's replay buffer should hold at least 10,000-100,000 transitions. A small buffer causes the agent to train on recent, correlated samples — defeating the purpose of experience replay. The buffer should be large enough that samples from different episodes and strategies are mixed.

Mistake 4: Not enough exploration

Without exploration, DQN converges to the first decent policy it finds and never discovers better ones. Use epsilon-greedy with a slow decay: $\epsilon$ starts at 1.0 (pure random) and decays to 0.01 over thousands of episodes. Too-fast decay causes premature convergence.
