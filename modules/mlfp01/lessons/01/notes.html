Speaker Notes — Lesson 1.1 Your First Data Exploration

Total time: ~45 minutes  ·  16 slides  ·  ~2.5 min per slide average

This is the first lesson of the entire course. Most learners are nervous — some have never written a line of code, others are senior data scientists who signed up for the certification. Both audiences must feel valued in this first session, or you will lose them. Use the three-layer markers (Foundations / Theory / Advanced) explicitly so each learner knows which parts are for them.

Slide 1~2 minTitle slide

Your First Data Exploration

Open with energy. "Welcome to MLFP. Over the next 12 weeks we will turn you into a working ML practitioner. By the end of this lesson — 45 minutes from now — you will have written your first Python program against real Singapore weather data. If you have never coded before, that is exactly the right starting point. If you have been coding for ten years, the next 45 minutes are a warm-up — but pay attention to the Theory and Advanced callouts because the rest of the course assumes you can read Polars idiomatically."

Take 30 seconds to introduce yourself: your name, what you do, what you expect to learn. Make it personal — the rest of the course is the syllabus.

If beginners look terrified: "There is nothing to memorise today. We will write 12 lines of code together and you can copy every one. By the end you will understand each line."
If experts look bored: "I promise the pace picks up. Today's deliverable is the muscle memory we will rely on later — and the Theory callouts in the slides have content you may not have seen, like why describe() uses n - 1 for variance."
→ "Three steps, every dataset, every project."
Slide 2~2 minThree-step diagram

Three steps you will use forever

Walk the diagram left to right. "Load: get the file off disk into memory as a structured object. Inspect: figure out what is in there. Summarise: compute statistics that tell you the shape of the data. These are the same three steps whether the data is 12 rows of weather or 50 million rows of HDB transactions."

Emphasise that this is universal — pandas, R, Excel, SQL all start the same way. The vocabulary differs but the steps are identical.

For beginners: compare to opening a new spreadsheet. "When you open a CSV in Excel, you load it. Then you scroll through to see what is there — that is inspecting. Then you might compute averages — that is summarising. Same three steps, different tools."
→ "But why bother? Three real Singapore questions show why."
Slide 3~2 minReal motivation

Three real Singapore questions

Bring it home with concrete use cases. The retail one is the most intuitive — every learner has bought an umbrella when it rained. The property one connects to HDB which everyone in Singapore cares about. The logistics one mentions the December monsoon which is shared experience.

Tell them: "These are not toy questions. These are exactly the kind of decisions Singapore companies make every week. Today we learn the steps; in Module 3 we will train models to actually answer them."

→ "Step zero — before any DataFrame, we need variables."
Slide 4~3 minVariables diagram

Variables: names bound to values

Walk the diagram explicitly. Point at city — say "this is just a label, a sticky note we made up". Point at "Singapore" — say "this is the actual data sitting in memory". Point at the type tag — say "Python automatically figured out this is a string because it starts with a quote".

Repeat for the float and the bool. Make the metaphor stick: variables are names for values. The same value can have multiple names; the same name can be moved to a different value. The label is just a string of characters Python uses to look things up.

If beginners are confused about types: "You don't need to declare the type yourself — Python figures it out. 28.5 has a decimal point so it's a float. True with a capital T is a boolean. "Singapore" in quotes is a string. Don't memorise the rules; you'll absorb them in the first hour."
For experts (skip if running long): mention that Python bindings live in dict scopes — locals, globals, builtins. Look-up is O(1) hash lookup. Reference semantics for mutable objects matter; immutables (str, int, float, bool) are safe.
→ "Now the four types you'll see every day."
Slide 5~2 minFour primitive types

Four types you will use constantly

Walk the table. For each row, give the example, then tell the use case. "Strings are for text — names of people, places, products. Integers are counts that have no fractional meaning — you cannot have 3.7 people, you have 3 or 4. Floats are measurements — temperature, price, distance. Booleans are flags — was it raining, did the test pass, is the customer Singaporean."

Then look at the code block. Point out that # str is a comment — Python ignores it; it is for humans. Run the four assignments in your head and announce: "After this code runs, Python has four variables in memory each holding a different type."

→ "Once we have variables, we want to print them — and f-strings are how."
Slide 6~3 minf-strings

f-strings: variables inside text

F-strings are the single most useful Python feature for data work. Before f-strings (introduced in Python 3.6) you had to use the awkward .format() method or even older % formatting. F-strings make string interpolation natural.

Read the example aloud: "f-quote — the f means formatted — open paren — sorry, open brace — city — close brace — that gets replaced by the value of city."

Then explain the format spec. :.1f means "as a float, with one digit after the decimal". The colon separates the variable from the format spec. Walk through the table of common specs without belabouring it — they will memorise these by lesson 1.4.

If beginners struggle with the spec syntax: "You don't need to memorise this. I'll show you again in every lesson. The key thing today is to recognise that {variable:format} means 'put the variable here, formatted this way'."
→ "Now we're ready for the main event — the DataFrame."
Slide 7~3 minDataFrame intro

The DataFrame: a typed table

Show the diagram. Walk it horizontally first: "Four columns: month, temperature, rainfall, humidity. Each one has a name — that's the header row in teal. And each one has a single type — string, float, float, int — that's the second row."

Then walk it vertically: "Each row is one observation — one month. Row 0 is January with 26.0 degrees and 194.7 mm of rain. Row 1 is February. And so on, twelve rows for twelve months."

Make the analogy explicit: "If you have used a spreadsheet, you already understand this picture. The difference is the rules."

→ "And here are the three rules every DataFrame obeys."
Slide 8~2 minThe three rules

Three rules every DataFrame obeys

Walk the three rules with examples of violations.

  1. "Every column has one type — so you cannot have a temperature column with values 26.0, 27.5, and "unknown". Polars would refuse to load that column as a float; it would either coerce everything to string or raise an error."
  2. "Every row has the same shape — so you cannot have row 5 with an extra column for typhoon name. Missing values are explicit nulls, not absent columns."
  3. "Rows are ordered — row 0 is the first, row 11 is the last. When you sort, the order changes; when you slice, the order is preserved."

Tell them: "These rules sound restrictive after Excel's freedom. But they are exactly why DataFrame operations are fast and predictable. Lesson 1.4 you will see how powerful those rules become when joining tables."

→ "Enough theory — time to load real data."
Slide 9~3 minLoad — code

Step 1: Load the data

This is their first encounter with imports. "import polars as pl means: load the polars library and let me call it pl for short. We will use pl hundreds of times this course; the as pl shortcut saves typing."

The MLFPDataLoader hides path complications. "On your laptop the file is in data/mlfp01/sg_weather.csv. On Colab it is on Google Drive. On a teammate's machine it might be somewhere else. MLFPDataLoader handles that for you so we can write the same code regardless of where you are running it."

Then run it. (12, 3) appears. "Twelve rows, three columns. That matches what we expected from the schema diagram."

→ "Loading is easy. Now the most important habit: looking at the data."
Slide 10~3 minInspect — the 30-second ritual

Step 2: Inspect — the 30-second ritual

This is the most important habit you will teach in this entire course. Stop and emphasise it. "Before you do anything else with a new dataset — before you build a model, before you write a query, before you make a chart — spend thirty seconds running these four lines. They will catch almost every mistake before it costs you an hour."

Run each line, read the output aloud, interpret it. Make a point of saying things like: "I see the temperature is a float with values around 26 — that matches Singapore. I see no nulls. The first row is January. OK, this looks like what I expected." This narration is what distinguishes a careful analyst from a sloppy one.

If beginners worry they will forget the four methods: "Don't memorise. I will repeat this ritual in every demo for the next eight lessons. By Lesson 1.8 it will be automatic."
→ "Now we summarise."
Slide 11~3 mindescribe()

Step 3: Summarise with describe()

Walk the table row by row. "Count: 12 — confirms we have all twelve months. Mean: 26.68 — Singapore's annual average temperature. Std: 0.43 — that is the standard deviation, a measure of spread, and a value of 0.43 is tiny — every month is within half a degree of average. Min: 26.0 in April. Max: 27.2 in May."

Make the contrast pop: "Now look at rainfall. Mean is 152, std is 56. That is a coefficient of variation of about 37 percent — meaning rainfall varies much more month to month than temperature. Singapore has 'one season' for temperature and 'two seasons' for rain — that single table tells you that, in two seconds."

→ "But what does mean actually compute? Let's go a level deeper."
Slide 12~3 minMean — math

What does mean really compute?

Walk the formula. "Sum of all the values, divided by the count." Plug in the numbers verbally: "26 plus 26.5 plus 27 plus … all twelve values … divided by 12 equals 26.68. describe() just did that for us."

If beginners flinch at the formula: "Read it left to right. The big sigma is just 'add things up'. The little i = 1 means start at the first value, the n on top means stop at the last value. Below the line, n is the count we divide by. That is all there is — pure addition and division."
For experts: point out the optimisation property. "The mean is the unique value c that minimises the sum of squared deviations from c. We will use this in Module 2 to derive ordinary least squares — OLS regression is just 'fit a line that minimises squared deviations', and the intercept of that line is the mean of y when there are no predictors."
→ "And the standard deviation?"
Slide 13~3 minVariance and std

Variance: how spread out is the data?

"Variance measures the typical squared distance from the mean. Take each value, subtract the mean, square the difference, average them up. The squaring is so positive and negative deviations don't cancel. The downside is the units are also squared — squared degrees Celsius is meaningless."

"So we take the square root and call it the standard deviation. Standard deviation is in the original units, which is why it's the number we actually quote: 0.43 degrees, not 0.18 squared degrees."

On the n-1 vs n: do not get into the full derivation. Just signal: "Why n-1 instead of n? It's called Bessel's correction. The intuition is that we have already used one degree of freedom to compute the mean, so only n-1 of the deviations are 'free'. The full proof is in Lesson 2.2."

The dart-board metaphor: "Picture twelve darts on a target. The mean is the centre of mass of the dart holes. Standard deviation is the typical distance from that centre. Tight cluster, low std. Wild scatter, high std. That's it."
→ "Theory done — now back to questions."
Slide 14~3 minFilter — find hottest month

Answering questions: the hottest month

Three lines of code with different jobs. Walk them slowly.

  1. "Line 1 computes the maximum temperature — a single number, 27.2."
  2. "Line 3 says: give me the rows where the temperature column equals that maximum. pl.col(...) builds an expression that says 'the temperature column'. The double-equals turns it into a boolean test. filter keeps the rows where the test is True."
  3. "Line 5 prints the answer using f-string formatting. We index into the filtered DataFrame to pull out the month name."

Run the code. Output: "Hottest: May at 27.2°C". Validate it against the earlier describe() output where max was Sep 27.2 — different tie-breaking; both are correct.

If beginners are puzzled by pl.col(...): "Don't worry about why it has its own syntax. Polars expressions are how we describe what we want to do without doing it yet. We'll see this pattern dozens of times."
→ "Three traps to watch for."
Slide 15~3 minCommon mistakes

Three rookie mistakes to avoid

Walk each one. Mistake 1 (assignment vs equality) is universal — every new programmer makes it. Tell them Python will refuse to run if city = "Singapore", which is the language being kind.

Mistake 2 (Polars immutability) is the trap that catches pandas refugees. "If you have used pandas before, you may have seen df.drop(...) modify df in place. Polars never does that. Every operation returns a new DataFrame. Always assign the result."

Mistake 3 (column name vs column value) is a slip beginners make in their first hour. Show them: "mean_temperature_c" is just a string you typed; it has no special connection to the DataFrame until you write df["mean_temperature_c"].

→ "Reflect, then we move on."
Slide 16~3 minReflection + bridge to 1.2

What you can now do

Run through the bullet list. Pause after each one and let them silently confirm to themselves "yes, I can do that". This builds confidence.

Then tease 1.2: "Today we worked with 12 rows. In Lesson 1.2 we will work with 50,000 rows of HDB resale transactions — every public housing sale in Singapore for the past several years. The same three steps (load → inspect → summarise) but now with a real-world dataset where you'll need to filter to specific towns, specific date ranges, and create new columns like price-per-square-metre."

Take 5 minutes for questions before moving on. The most common questions after Lesson 1.1 are: "Do I need to install anything?" (no — Colab has everything), "What if I get an error?" (debugging is its own skill — we'll cover it in 1.2), "Can I use pandas instead?" (no — Polars is faster and we use it consistently throughout the course).

→ End of Lesson 1.1. Take a 5-minute break.
