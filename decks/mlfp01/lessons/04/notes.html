Speaker Notes — Lesson 1.4 Joins and Multi-Table Data

Total time: ~55 minutes  ·  14 slides  ·  ~4 min average

This is the first time learners combine multiple data sources. The main trap is silent row loss from inner joins — make sure every student sees that trap live. The second trap is unhandled nulls. If they leave this lesson with those two instincts, you have won.

Slide 1~2 minTitle

Joins and Multi-Table Data

Open with: "Every real question is a join question. HDB prices alone tell you a lot, but they tell you far more when you combine them with MRT proximity and school counts. Today we learn how to do that in Polars."

→ "Three tables, one enriched result."
Slide 2~3 minThree-table diagram

Real questions need multiple tables

Walk the three source tables on the left. "HDB transactions — 500,000 rows, one per sale. MRT stations — one row per town, tells you the closest station. Schools — one row per school, we'll aggregate to one per town."

Then the big joined table on the right. "Every HDB row now carries its town's spatial context — nearest MRT, distance, school count. Ready for analysis."

→ "Before we join, two Python tools we need."
Slide 3~3 minImports

Imports: loading libraries

Walk the three forms. import X, import X as Y, from X import name. Tell them the Polars convention is always import polars as pl.

Make the "install once, import anywhere" point. PyPI has hundreds of thousands of packages. Every ML and data tool they will ever use is one pip install away.

For beginners: "Think of import like borrowing a book from the library. The library has millions of books. You borrow the ones you need for today's project."
→ "And if/elif/else."
Slide 4~3 minif/elif/else code

if / elif / else: code-level decisions

Walk the price_range_label example. "Python checks each condition in order. The first one that's True wins. Everything after is skipped."

Contrast with Lesson 1.2's pl.when / then / otherwise: "That one chooses values inside a column — it operates on the whole column at once. if/elif/else chooses what the program does at the code level. Different jobs, same logic shape."

→ "Follow the flow."
Slide 5~2 minFlow diagram

Flow of an if/elif/else

Walk the diagram. Price enters the top. First check fails, arrow down. Second check passes, arrow to result. Other branches are greyed out — never evaluated.

Emphasise: "Once a branch matches, the rest are skipped. This is why the order matters — if you put Luxury first with a catch-all, nothing else would ever run."

→ "Now the main event. Joins."
Slide 6~4 minBasic join code

The basic join

Run it live. Point at each argument: "left DataFrame before .join, right DataFrame as the first positional argument, on names the key, how is the join type."

Print the shape before and after. Make them see that left join preserves row count. "Same number of rows — we added columns without adding or removing rows."

For experts: "Under the hood Polars builds a hash map from the right-side key column, then scans the left side, looking each key up in the hash map. Single-pass, parallel, SIMD-accelerated."
→ "Four join types you need to know."
Slide 7~5 minJoin types venn

Four join types

Walk the four venn diagrams. For each, describe what survives in the output:

  • Inner: only rows where the key appears on both sides. Good for "give me only the matches".
  • Left: all left rows, matches from the right. Good for enrichment — "keep every HDB row and attach MRT where available".
  • Right: mirror image. Rarely used — you can always swap sides and use left.
  • Outer/Full: every row from both sides. Good for reconciliation — "show me everything from both sources, flag what's missing".

Then the rule of thumb: "For enrichment, which is the main job in Module 1, use left join 95% of the time. Inner is a gotcha — you lose rows silently."

→ "Always check keys before joining."
Slide 8~4 minKey overlap check

Check the keys before joining

Run it live. "HDB has 26 towns. MRT has 24 towns. Matched 24. Two HDB towns have no MRT data — those will become nulls under a left join, or disappear under an inner join."

Make this a habit they form today. "Every join, key check first. It takes two lines and catches most bugs."

For beginners: "Sets are just uniqueness containers. set(list) gives you the unique values. & on two sets gives you the intersection. - gives you the difference."
→ "When keys aren't unique on one side, aggregate first."
Slide 9~3 minAggregate before join

Aggregate one side if keys aren't unique

Explain the problem: "Schools has one row per school. If we joined directly, every HDB transaction in Bishan would be duplicated for every school in Bishan — eight copies of every row if Bishan has eight schools. We don't want that."

Show the fix: group the schools table to one row per town first, then join. This is a common pre-join step whenever the 'right side' is granular.

→ "And handling the nulls the join produces."
Slide 10~3 minNull handling

Handling nulls after a left join

Three options: count, fill, drop. Walk each. The fill_null(0) example is the most common — "no school data" means "zero schools", and filling with zero lets arithmetic continue working.

The strict rule: "Never let a post-join null silently propagate. Decide on the first line after the join: fill, drop, or flag. If you skip this decision the nulls will bite you in the next aggregation."

For experts: "Nulls in Polars are a separate state from NaN. null_count(), fill_null(), and is_not_null() are the core three methods — they work on every dtype."
→ "The set theory behind it."
Slide 11~3 minSet operations

Set theory under the hood

Walk the three formulas. Intersection — both sides. Union — either side. Difference — left but not right. Every join type maps to one of these set operations on the keys.

Tell them: "Once you see joins as set operations, the choice becomes mechanical. Inner = intersection. Left = all-of-H, matches-from-M. Outer = union. Difference = the unmatched rows."

→ "Putting it all together."
Slide 12~4 minDesirability classifier

A desirability classifier

Walk the function. It uses if/elif/else on two inputs (distance and schools) to classify towns into tiers. Point out the if distance_km is None guard — without it the comparison would fail for unmatched towns.

Then show how it connects to the group_by: aggregate to town level, apply the function. This is the full pipeline — load, join, aggregate, classify.

→ "Four traps to avoid."
Slide 13~3 minCommon mistakes

Four traps when joining

Walk each one. The inner/left mistake is the one that burns everyone. Show it live: run an inner join, show the row count drop, ask what happened. Then the left-join version preserves the count.

The Cartesian-product mistake is less common but catastrophic when it happens. Show the ugly version: join schools directly, watch the row count explode 5× or 10×.

The case-mismatch trap: "BISHAN" vs "Bishan" is a real Singapore example — different agencies capitalise differently. Teach them to normalise with str.to_uppercase() before joining.

→ "Reflect and bridge."
