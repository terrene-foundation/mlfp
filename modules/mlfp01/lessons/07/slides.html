Running your first profile


from kailash_ml import DataExplorer
from shared import MLFPDataLoader, run_profile

loader = MLFPDataLoader()
df = loader.load("mlfp01", "sg_economic_indicators.csv")

explorer = DataExplorer()
profile = run_profile(explorer, df)

print(f"Alerts fired: {len(profile.alerts)}")
for alert in profile.alerts:
    print(f"  {alert.type}: {alert.column} — {alert.message}")

run_profile() is a sync wrapper around the async explorer.profile(df). Same result, simpler call.

Configuring thresholds


from kailash_ml import AlertConfig

config = AlertConfig(
    null_threshold=0.10,       # fire at 10%, not 5%
    skew_threshold=3.0,        # tolerate more skew
    correlation_threshold=0.90,# stricter correlation
    outlier_iqr_factor=2.5,    # tighter outlier fence
)

profile = run_profile(explorer, df, config=config)
🔵 Theory: There is no universal "correct" threshold. Thresholds encode your domain expectations. Economic data tolerates more skew than medical data.

compare(): before vs after


# Clean the data
df_clean = (
    df.with_columns(pl.col("date").str.to_date("%Y-%m"))
    .drop_nulls(subset=["cpi", "fx_rate"])
    .drop("country")
)

# Compare original vs cleaned
comparison = run_profile(explorer, df, df_compare=df_clean)

for diff in comparison.differences:
    print(f"  {diff.column}: {diff.metric} "
          f"{diff.before:.2f} → {diff.after:.2f}")
🟢 Foundations: compare() answers: "did my cleaning actually help?"

Comparison result

Cleaning reduced alerts from 6 to 1 — the remaining alert (high correlation) is a domain fact, not a data quality issue.

Classes: using, not building


# You USE DataExplorer — you don't build it
explorer = DataExplorer()   # create an instance
profile  = run_profile(explorer, df)   # call a method

# The class hides complexity behind a simple interface
# Same pattern for every Kailash engine:
#   engine = EngineClass()
#   result = engine.do_something(data)

A class bundles data and methods together. You will build your own classes in Module 3. Today: consumer only.

Handling errors: try/except


try:
    df = loader.load("mlfp01", "nonexistent.csv")
    profile = run_profile(explorer, df)
except FileNotFoundError as e:
    print(f"File missing: {e}")
except ValueError as e:
    print(f"Bad data: {e}")
⚠ Warning: Never write except: pass — that silently swallows every error. Always name the exception type and handle it explicitly.

compare() for train vs test


# Split data
n = len(df_clean)
df_train = df_clean.head(int(n * 0.8))
df_test  = df_clean.tail(n - int(n * 0.8))

# Compare distributions
comparison = run_profile(explorer, df_train, df_compare=df_test)

# If distributions differ → data leakage or temporal shift
for diff in comparison.differences:
    if diff.severity == "high":
        print(f"⚠ Distribution shift: {diff.column}")
🟣 Advanced: This is the preview of drift monitoring (Module 3). Same concept: compare two distributions, flag differences.

Three mistakes to avoid

1. Ignoring alerts because there are too many
Retune the thresholds — do not silence the engine.
2. Profiling before fixing obvious issues
Fix mixed types and obviously wrong schemas first. Then profile.
3. Auto-dropping correlated columns
Alerts prompt investigation, not automated removal.
