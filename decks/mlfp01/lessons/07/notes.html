Speaker Notes — Lesson 1.7 Automated Data Profiling

Total time: ~50 minutes  ·  16 slides  ·  ~3 min per slide average

This lesson marks a shift from hand-built exploration to engine-powered automation. Learners who have been writing describe() and null_count() for six lessons should feel the relief of having one engine do it all. Experts should appreciate the configurable thresholds and the compare() feature. Use the messy economic dataset throughout — its three date formats and mixed null representations are common in real Singapore government data releases.

Slide 1~2 minTitle slide

Automated Data Profiling

Open with: "For six lessons you have been the profiler — inspecting shape, columns, nulls, distributions. Today you meet the engine that does all of it in one call. DataExplorer checks every column for eight kinds of data quality problems and gives you a structured report."

Set expectations: "By the end of this lesson you will be able to profile any dataset, tune the alert thresholds, and compare two versions of a dataset — before and after cleaning."

If beginners look overwhelmed by 'engine': "Think of it as a pre-built function that does in one line what took you 20 lines before. You don't need to understand how it works internally — just what it gives you and how to configure it."
→ "Why do we need automation?"
Slide 2~3 minManual vs automated

Manual profiling does not scale

Walk the diagram. Left side: what they have been doing. Right side: what DataExplorer does. Emphasise the gap: "Manual profiling catches the obvious — nulls, shape, basic stats. But it misses the subtle: a date column with three different formats, a column that is 98% zero, two columns that are 99% correlated. Those are the issues that cost you hours downstream when your model behaves strangely."

Real example: "I once spent three hours debugging a regression model before discovering that the date column had been silently parsed as a string in half the rows. DataExplorer catches that in the first second."

→ "Here's how the engine is structured."
Slide 3~3 minArchitecture diagram

DataExplorer in one picture

Walk the flow: "DataFrame in, three things out — alerts, statistics, and optionally a comparison. The alerts are the headline news. The statistics are the detail behind each alert. The comparison is for when you have two versions of the data."

Point out AlertConfig sitting inside the engine: "This is how you tune what counts as an alert. Default thresholds are sensible starting points but every domain has different expectations."

For experts: mention that DataExplorer is async internally (it profiles columns in parallel) but the sync wrapper hides that. The async API is available for advanced users who want to integrate profiling into async pipelines.
→ "What exactly does it check for?"
Slide 4~4 minEight alert types

Eight alert types

Walk the table row by row. For each alert type, give the intuition:

  1. "high_nulls: more than 5% of values missing. Decide whether to impute, drop, or investigate."
  2. "outliers: values beyond 3 IQR fences. Could be errors or genuine extremes."
  3. "duplicates: identical rows. Could be double-counting."
  4. "high_skewness: distribution is lopsided. May need log transform."
  5. "high_correlation: two columns carry almost the same information. Multicollinearity risk."
  6. "low_cardinality: fewer than 3 unique values. Possibly should be treated as categorical."
  7. "constant: zero variance. Useless as a predictor — drop it."
  8. "type_inference: stored as string but looks like a number, or mixed formats."
For beginners: "Don't memorise the thresholds. Memorise the eight kinds of problem. If you can name the problem, you can configure the threshold later."
→ "Let's run it."
Slide 5~3 minFirst profile — code

Running your first profile

Run this live. The dataset is the messy Singapore economic indicators file with three date formats and mixed null representations. Walk each line: "Create the engine, load the data, profile it, print the alerts."

When the alerts print, read them aloud. Compare to what they would have found by hand: "Would you have caught that the date column has three formats? Probably not until something broke."

→ "Let's look at the full profile output."
Slide 6~3 minProfile output diagram

What the profile returns

Walk the two boxes. Left: the alerts list — these are the headlines. Right: the column_stats dictionary — this is the detail. "For every column, you get mean, std, null percentage, skewness, unique count, and more. The alerts are just the statistics that crossed a threshold."

Point out that you can access the raw stats programmatically: profile.column_stats["cpi"].mean. This is useful when you want to build automated cleaning rules based on the profile.

→ "Now let's tune the thresholds."
Slide 7~3 minAlertConfig

Configuring thresholds

"Default thresholds are the engine's opinion. Your domain might disagree. Economic data with 10% nulls might be fine — government agencies revise figures quarterly. Medical trial data with 2% nulls might be a crisis."

Walk the code: "We create an AlertConfig with custom thresholds and pass it to run_profile. Now only alerts that cross our custom thresholds fire."

For experts: "The correlation threshold is the most important to tune. In economic data, CPI and employment rate are genuinely correlated at 0.94 — that is a domain fact, not a data quality issue. Set the threshold to 0.97 if you only want near-duplicates."
→ "Real messy data from Singapore."
Slide 8~3 minMessy data example

The messy economic dataset

Show the diagram and let it sink in. "This is real. Government data releases in Singapore — and everywhere else — come with inconsistencies. Three date formats: ISO (2020-01), US-style (01/2020), and concatenated (202004). Nulls represented as NaN, 'N/A', and an em-dash."

"DataExplorer catches all of these in the type_inference and high_nulls alerts. By hand, you would need to inspect every column individually."

Reassurance: "You are not expected to know how to fix all of these yet. Lesson 1.8 will show you how to build a cleaning pipeline. Today the goal is to find the problems."
→ "Let's read the actual profile output."
Slide 9~3 minProfile walkthrough

Reading the profile

Walk each alert. For each one, ask: "What should we do about this?"

  • high_nulls on CPI: impute with interpolation or forward-fill
  • high_nulls on fx_rate: same strategy
  • type_inference on date: standardise to ISO format
  • constant on country: drop — adds no information
  • high_skewness on CPI: consider log transform
  • high_correlation: keep both but note the collinearity risk

"The profile gives you the diagnosis. You still make the treatment decision. That is the human-in-the-loop part."

→ "Now let's verify the cleaning worked."
Slide 10~3 mincompare()

compare(): before vs after

"After you clean data, how do you know the cleaning helped? Re-profile and compare. DataExplorer's compare() does this in one call."

Walk the code: clean the date column, drop nulls, drop the constant column. Then compare. The comparison shows what changed — alert count dropped from 6 to 1, null percentages went to zero, format issues resolved.

→ "The comparison as a picture."
Slide 11~3 minComparison diagram

Comparison result

Walk the before/after diagram. "Before: 6 alerts, 14% nulls in CPI, 3 date formats, 120 rows. After: 1 alert, 0% nulls, 1 date format, 103 rows. We lost 17 rows to null removal — that is the cost of cleaning."

"The remaining alert — high correlation — is a domain fact, not a data quality issue. CPI and employment genuinely move together. The profile correctly flags it; we correctly decide to keep both columns."

→ "A new mental model: using engines, not building them."
Slide 12~3 minClasses as users

Classes: using, not building

"DataExplorer is a class. You create an instance with (), then call methods on it. This is the same pattern for every Kailash engine: create, configure, call. You do not need to understand the internals — just the interface."

"In Module 3 you will build your own classes. Today we establish the consumer pattern: instantiate, configure, call, interpret."

If beginners ask 'what is a class?': "Think of it as a toolkit in a box. DataExplorer is the box. profile() and compare() are the tools inside. You open the box, pick a tool, and use it."
→ "What happens when things go wrong?"
Slide 13~3 mintry/except

Handling errors: try/except

"Real pipelines encounter errors: missing files, bad schemas, network timeouts. try/except lets you catch the error and handle it instead of crashing."

Walk the code: "The try block is the happy path. The except blocks catch specific error types. Always name the exception — never write bare except:."

If beginners ask 'when do I use this?': "Whenever you load external data — files, APIs, databases. If the data source might not be there, wrap it in try/except."
→ "Beyond cleaning: comparing train vs test."
Slide 14~3 minTrain/test comparison

compare() for train vs test

"Here is a preview of Module 3. When you split data into train and test, the distributions should be similar. If they differ significantly, your model may fail on the test set. compare() catches this."

Walk the code. Point out the severity check: "We only flag high-severity differences. Small statistical fluctuations are normal."

For experts: "This is the same concept as drift monitoring in production. In Module 3 you will use DriftMonitor to run this comparison continuously on deployed models."
→ "Traps to avoid."
Slide 15~2 minCommon mistakes

Three mistakes to avoid

Walk each one. Mistake 1 is the most common: "Beginners see 40 alerts and disable the engine. The correct response is to retune the thresholds so each alert means something."

Mistake 2: "Profile after cleaning the basics, not before. Fix obvious type issues first so the profile shows real problems, not noise."

Mistake 3: "The correlation alert says 'these two columns carry similar information'. It does not say 'drop one'. That decision requires domain knowledge."
