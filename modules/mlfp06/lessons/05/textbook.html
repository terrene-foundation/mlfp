Lesson 6.5 AI Agents — ReAct, Tool Use, and Function Calling

An LLM that only generates text is like a brain without hands. It can reason about problems but cannot solve them — it cannot query a database, call an API, train a model, or check a fact. Agents give the LLM the ability to act: reason about a task, select a tool, execute it, observe the result, and iterate until the task is complete. Today you build a ReAct agent using Kaizen that wraps Kailash's DataExplorer, TrainingPipeline, and ModelVisualizer as tools — an autonomous data scientist that explores a dataset, selects a model, trains it, and reports results. All under a cost budget that prevents runaway LLM spending.

Why This Matters

A Singapore logistics company receives 200 datasets per quarter from port authorities, customs, and carrier APIs. An analyst manually profiling each one takes 30 minutes: load, inspect, check for nulls, visualise distributions, try a baseline model. An agent does it in two minutes, including writing the summary. More importantly, the agent applies the same methodology every time — it never forgets to check for class imbalance or skips the train-test split.

But an agent without guardrails is dangerous. It might call an expensive API in a loop, hallucinate tool names, or present fabricated results as real. Today you learn the three components that make agents useful and safe: structured reasoning (ReAct), structured tool interfaces (function calling), and cost budgets (LLMCostTracker).

Core Concepts

What is an Agent?

An agent is an LLM wrapped in a loop. On each iteration the LLM:

  1. Reasons about the current state of the task — what it knows, what it still needs.
  2. Decides on an action — which tool to call, with what parameters.
  3. Executes the action — the tool runs and returns a result.
  4. Observes the result — incorporates it into its working memory.
  5. Repeats or answers — loops if more work is needed, or returns a final answer.

This is the ReAct pattern (Reasoning + Acting), introduced by Yao et al. (2022). The key insight is that interleaving reasoning with action produces better results than either pure reasoning (chain-of-thought) or pure action (tool calls without explanation). The reasoning step forces the LLM to plan before acting, reducing random tool invocations.

The ReAct Loop

Foundations

Think of a ReAct agent like a junior analyst who talks through their reasoning before each step. "I think I should check if there are null values first" → runs the null check → "OK, 3% nulls in the income column. I should impute those before training" → runs imputation. The explicit reasoning makes the agent's logic transparent and debuggable.

Chain-of-Thought vs ReAct

Chain-of-thought (CoT) from Lesson 6.1 asks the LLM to reason step-by-step but does not let it act. It reasons about a question and returns an answer in a single pass. ReAct extends CoT by letting the LLM verify its reasoning against reality through tool calls. This matters because LLMs hallucinate — a CoT agent might reason correctly about a formula but apply it to hallucinated numbers. A ReAct agent can check the actual data.

Math Foundations

Agent design has less novel mathematics than deep learning, but there are two formal frameworks worth understanding to reason about agent behaviour.

Agents as Markov Decision Processes

An agent's interaction with its tools can be modelled as a simplified MDP (connecting to the RL formalism from Lesson 5.8). At each step:

$$ s_{t+1}, r_{t+1} = \text{env}(s_t, a_t) $$
$s_t$ — agent's state (conversation history + observations) $a_t$ — action (tool call or final answer) $r_{t+1}$ — implicit reward (task progress)

Unlike RL where the policy is learned through training, a ReAct agent's "policy" is the LLM itself — prompted to select the best action given the current state. The LLM's pre-training acts as a massive prior over useful action sequences.

Expected cost of an agent run

Each agent step incurs token costs. If the agent makes $K$ tool calls with an average of $T_{\text{in}}$ input tokens and $T_{\text{out}}$ output tokens per call:

$$ C_{\text{total}} = \sum_{k=1}^{K} \bigl( c_{\text{in}} \cdot T_{\text{in}}^{(k)} + c_{\text{out}} \cdot T_{\text{out}}^{(k)} \bigr) $$
$c_{\text{in}}$ — cost per input token $c_{\text{out}}$ — cost per output token $K$ — number of LLM calls

Context grows with each step because the agent carries the full conversation. Step $k$ has approximately $T_{\text{in}}^{(k)} \approx T_{\text{system}} + k \cdot T_{\text{avg\_turn}}$ input tokens. This means agent cost grows quadratically with the number of steps — a critical insight for cost budgeting.

Theory

The quadratic cost growth explains why agents stuck in loops are expensive. A 10-step agent costs roughly $\frac{10 \cdot 11}{2} = 55$ "step-units" of input tokens. A 20-step agent (just double the steps) costs $\frac{20 \cdot 21}{2} = 210$ step-units — nearly 4x more. This is why Kaizen's LLMCostTracker is not optional. Without a hard budget, a confused agent can spend hundreds of dollars in minutes.

Function Calling Protocol

An agent needs a structured way to invoke tools. The function calling protocol defines tools as JSON schemas that the LLM can reference by name. The LLM does not execute code — it produces a structured JSON object specifying which function to call and with what arguments. The agent framework parses that JSON and executes the actual function.

Tool schema definition

tools = [
    {
        "type": "function",
        "function": {
            "name": "explore_data",
            "description": "Load and profile a dataset. Returns shape, column types, null counts, and basic statistics.",
            "parameters": {
                "type": "object",
                "properties": {
                    "dataset": {
                        "type": "string",
                        "description": "Dataset filename (e.g. 'churn.csv')"
                    },
                    "show_distributions": {
                        "type": "boolean",
                        "description": "Whether to compute distribution stats",
                        "default": False
                    }
                },
                "required": ["dataset"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "train_model",
            "description": "Train a classification model on the loaded dataset.",
            "parameters": {
                "type": "object",
                "properties": {
                    "algorithm": {
                        "type": "string",
                        "enum": ["logistic_regression", "random_forest", "gradient_boosting"],
                        "description": "ML algorithm to use"
                    },
                    "target_column": {
                        "type": "string",
                        "description": "Name of the target variable"
                    }
                },
                "required": ["algorithm", "target_column"]
            }
        }
    }
]
Foundations

Good tool descriptions are the agent equivalent of good documentation. The LLM decides which tool to call based entirely on the description field and the parameter names. A vague description like "does data stuff" leads to incorrect tool selection. A precise description like "Load and profile a dataset. Returns shape, column types, null counts, and basic statistics" tells the LLM exactly when this tool is useful and what to expect back.

The tool_choice parameter

You can control how aggressively the LLM uses tools:

tool_choiceBehaviourWhen to use
"auto"LLM decides whether to call a toolDefault — let the agent reason
"required"LLM must call at least one toolFirst step, when you know action is needed
{"type": "function", "function": {"name": "..."}}Force a specific toolDebugging, testing a particular path
"none"LLM cannot call toolsFinal summarisation step

Parallel function calling

Some LLM providers support parallel function calling — the LLM can request multiple tool calls in a single response. This is useful when the agent needs independent pieces of information simultaneously (e.g., profile two datasets at once). Kaizen handles parallel calls automatically when the model supports them.

The Kailash Engine: Kaizen Agents

Kailash Engine

Kaizen provides three agent types: ReActAgent (thought-action-observation loop), ChainOfThoughtAgent (reason then answer, no tools), and BaseAgent (build your own loop). All agents use Signature for typed input/output, Delegate for LLM communication, and LLMCostTracker for budget enforcement. The agent framework handles prompt formatting, tool dispatch, error recovery, and streaming.

Building a ReAct agent with Kaizen

from kaizen import ReActAgent, Signature, InputField, OutputField
from kaizen.tools import tool
from kailash_ml import DataExplorer, TrainingPipeline, ModelVisualizer
import polars as pl

# Define typed signature
class DataAnalysis(Signature):
    """Analyse a dataset and recommend the best model."""
    dataset_name: str = InputField(desc="Name of the CSV file to analyse")
    analysis_report: str = OutputField(desc="Markdown report with findings")
    recommended_model: str = OutputField(desc="Best model and why")

# Define tools that wrap Kailash engines
@tool
def explore_data(dataset: str) -> dict:
    """Load a dataset and return profiling statistics."""
    from shared import MLFPDataLoader
    loader = MLFPDataLoader()
    df = loader.load("mlfp06", dataset)
    explorer = DataExplorer()
    profile = explorer.profile(df)
    return {
        "shape": str(df.shape),
        "columns": df.columns,
        "null_pct": profile.null_percentages,
        "stats": profile.summary_stats
    }

@tool
def train_model(dataset: str, target: str, algorithm: str) -> dict:
    """Train a model and return evaluation metrics."""
    from shared import MLFPDataLoader
    loader = MLFPDataLoader()
    df = loader.load("mlfp06", dataset)
    pipeline = TrainingPipeline(algorithm=algorithm)
    result = pipeline.fit(df, target=target)
    return {
        "accuracy": result.metrics["accuracy"],
        "f1": result.metrics["f1_score"],
        "training_time_s": result.training_time
    }

@tool
def visualise_results(dataset: str, target: str) -> str:
    """Generate a feature importance chart and return the file path."""
    viz = ModelVisualizer()
    path = viz.feature_importance(dataset, target, save_to="results/")
    return f"Chart saved to {path}"

# Create and run the agent
agent = ReActAgent(
    signature=DataAnalysis,
    tools=[explore_data, train_model, visualise_results],
    max_iterations=10,
    model="claude-sonnet-4-20250514"
)

Cost budget enforcement

from kaizen import LLMCostTracker

# Set a hard budget — agent stops when exhausted
tracker = LLMCostTracker(budget_usd=2.00)

result = agent.run(
    dataset_name="sg_churn.csv",
    cost_tracker=tracker
)

print(f"Total cost: ${tracker.total_cost:.4f}")
print(f"LLM calls: {tracker.call_count}")
print(f"Tokens used: {tracker.total_tokens:,}")
print(f"Budget remaining: ${tracker.remaining:.4f}")
Theory

The cost tracker operates at the Delegate level — every LLM call passes through it. When the remaining budget drops below the estimated cost of one more call, the agent receives a signal to produce its final answer with whatever information it has gathered so far. This is a graceful degradation — the agent does not crash, it summarises partial results and explains what it could not complete.

Agent Design Framework

Before writing code, answer four questions. This mental framework (from the MLFP agent design methodology) prevents the most common failure: an agent that calls tools randomly because its purpose was never clearly defined.

Advanced

The specialist description becomes the agent's system prompt. A vague specialist ("you are a helpful assistant") produces unfocused behaviour. A sharp specialist ("you are a senior data scientist at a Singapore logistics firm, specialising in churn prediction using gradient boosting on tabular data") produces targeted, domain-appropriate reasoning. The more specific the specialist, the fewer wasted tool calls.

Worked Example: Autonomous Data Analyst

Goal

Build an agent that autonomously analyses Singapore telco customer churn data: profiles the dataset, identifies class imbalance, trains three candidate models, and recommends the best one — all within a $1.00 budget.

Step 1. Define the signature and tools.

We already defined these above. The key design decisions:

  • 3 tools (explore, train, visualise) — enough to complete the task, few enough for reliable selection.
  • Typed outputs — analysis_report (Markdown) and recommended_model (string) force structured results.
  • Tool descriptions are precise — each says what it returns, not just what it does.
Step 2. Run the agent with cost tracking. 
from kaizen import LLMCostTracker

tracker = LLMCostTracker(budget_usd=1.00)

result = agent.run(
    dataset_name="sg_telco_churn.csv",
    cost_tracker=tracker
)

print(result.analysis_report)
print(f"\nRecommended: {result.recommended_model}")
print(f"Cost: ${tracker.total_cost:.4f} ({tracker.call_count} LLM calls)")
Step 3. Observe the agent's trace.

A typical ReAct trace for this task looks like:

[Action]  explore_data(dataset="sg_telco_churn.csv")
[Observe] shape=(5000, 12), target "churned": 73% No / 27% Yes, 2% nulls in "tenure"

[Thought] Class imbalance (73/27). I should use F1 as the metric, not accuracy.
          Let me try three models: logistic regression, random forest, gradient boosting.
[Action]  train_model(dataset="sg_telco_churn.csv", target="churned", algorithm="logistic_regression")
[Observe] accuracy=0.78, f1=0.61, training_time=1.2s

[Action]  train_model(dataset="sg_telco_churn.csv", target="churned", algorithm="random_forest")
[Observe] accuracy=0.83, f1=0.72, training_time=3.4s

[Action]  train_model(dataset="sg_telco_churn.csv", target="churned", algorithm="gradient_boosting")
[Observe] accuracy=0.85, f1=0.76, training_time=5.1s

[Thought] Gradient boosting wins on both accuracy (0.85) and F1 (0.76).
          The F1 gap is significant: 0.76 vs 0.72 for random forest.

Step 4. Evaluate the agent's behaviour.

The agent made sensible decisions: it profiled first (not training blind), recognised class imbalance, chose F1 over accuracy, compared three models, and recommended the winner. Six LLM calls, $0.34, about 90 seconds end-to-end. A human analyst would reach the same conclusion in 20-30 minutes.

Agent Design Considerations

Iterative refinement

A critic agent can review another agent's output and recommend improvements. The data analyst agent produces a report; a reviewer agent checks for missing analyses, statistical errors, or unsupported conclusions. This is more reliable than asking the same agent to self-critique.

Human-in-the-loop

For high-stakes decisions, the agent can pause and request human approval before proceeding. Kaizen supports this with checkpoint callbacks — the agent emits a checkpoint event, the framework surfaces it to the user, and the agent resumes only after approval. This is critical for financial applications where MAS requires human oversight of AI-driven decisions.

Monitoring and logging

Every agent step — thought, action, observation — should be logged for debugging and audit. Kaizen's Delegate emits streaming events that you can capture in real time. This connects directly to the governance audit trails in Lesson 6.7.

Common Mistakes

Mistake 1: Too many tools

Agents perform better with 3-5 focused tools than 20 generic ones. With 20 tools, the LLM spends most of its reasoning capacity on tool selection rather than problem-solving. If you need more than 7 tools, consider splitting into multiple specialised agents (Lesson 6.6).

Mistake 2: No cost budget

An agent stuck in a reasoning loop can spend hundreds of dollars in minutes. Always set a hard budget with LLMCostTracker. A good heuristic: estimate the expected number of steps, multiply by 3x for safety, and compute the token cost. For most data analysis tasks, $1-5 is sufficient.

Mistake 3: Treating agent output as authoritative

Agents hallucinate. The tool results are real (they come from actual code execution), but the agent's interpretation of those results can be wrong. "The model achieved 95% accuracy" might be correct, but "this means the model is ready for production" is the agent's opinion. Always verify critical conclusions against the raw tool outputs.

Mistake 4: Vague tool descriptions

The LLM selects tools based on their description field. "Processes data" is useless — the LLM cannot distinguish it from other data tools. "Load a CSV file and return shape, column types, null percentages, and descriptive statistics as a dictionary" tells the LLM exactly when this tool is appropriate and what to expect in the response.

Reflection

You have built your first autonomous AI agent — a system that reasons about a task, selects and calls tools, observes results, and iterates until it has an answer. The three critical components are the ReAct loop (structured reasoning), function calling (structured tool interfaces), and cost budgets (structured safety).

The agent design framework — goal, process, specialist, tools — is more important than the code. A well-designed agent with 3 tools outperforms a poorly-designed agent with 20. The specialist description is the single highest-leverage prompt engineering decision you make.

In Lesson 6.6, we move from single agents to multi-agent systems: teams of specialists coordinated by a supervisor, communicating through structured protocols, with shared and private memory. The data analyst agent you built today becomes one worker in a larger pipeline.
