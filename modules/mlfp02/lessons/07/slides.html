The problem: noisy experiments

Your A/B test shows a +3% lift in spending, but $p = 0.12$. Not significant.

Options:

  • Run longer (expensive, delays launch)
  • Get more users (not always possible)
  • Reduce the variance (free, immediate)
🟢 Foundations: CUPED uses data you already have (pre-experiment behaviour) to shrink the noise. Same data, more power.

CUPED: the key formula

\[ Y_{\text{adj}} = Y - \theta \cdot (X - \mathbb{E}[X]) \]

Where $\theta^* = \text{Cov}(Y, X) / \text{Var}(X)$ minimises variance.

\[ \text{Var}(Y_{\text{adj}}) = \text{Var}(Y)(1 - \rho^2) \]

CUPED in code


import numpy as np

X_pre = df["spend_pre"].to_numpy()
Y_post = df["spend_post"].to_numpy()

theta = np.cov(Y_post, X_pre)[0, 1] / np.var(X_pre)
Y_adj = Y_post - theta * (X_pre - X_pre.mean())

print(f"Var(Y):     {np.var(Y_post):.2f}")
print(f"Var(Y_adj): {np.var(Y_adj):.2f}")
print(f"Reduction:  {1 - np.var(Y_adj)/np.var(Y_post):.1%}")
🔵 Theory: CUPED is the residual from regressing $Y$ on $X$. The optimal $\theta$ is the regression coefficient.

When does CUPED work?

  • High $\rho$: the pre-metric must correlate with the post-metric. E-commerce spending: typically 0.6-0.8.
  • Pre-experiment only: the covariate must be measured before randomisation.
  • Same users: you need pre and post measurements for the same individuals.
⚠ Warning: Never use a post-experiment covariate. It may be affected by the treatment — adjusting for it removes the effect you are measuring.

SRM: is your experiment broken?


from scipy import stats as scipy_stats

n_control, n_treatment = 4800, 5200
n_total = n_control + n_treatment
chi2 = sum((obs - n_total/2)**2 / (n_total/2)
           for obs in [n_control, n_treatment])
p_srm = 1 - scipy_stats.chi2.cdf(chi2, df=1)

if p_srm < 0.01:
    print("SRM detected — do not trust results")

Common SRM causes

  • Bot filtering: bots disproportionately hit one variant
  • Redirect bugs: one variant's page fails to load for some users
  • Population filters: filters applied after randomisation exclude users unevenly
  • Bucketing errors: hash function not uniformly distributed
Rule: Always check SRM before computing treatment effects.

When you cannot randomise

Singapore's cooling measures apply to all districts. You cannot randomly assign the policy. But you can compare districts more affected vs less affected, before and after.

\[ \text{ATT} = (\bar{Y}_{T,\text{post}} - \bar{Y}_{T,\text{pre}}) - (\bar{Y}_{C,\text{post}} - \bar{Y}_{C,\text{pre}}) \]

The first difference removes the group level effect. The second removes the time trend.

DiD: the picture

The parallel trends assumption

DiD assumes that, without treatment, both groups would have followed the same trend.

  • Check: plot pre-period trends for both groups — should be parallel
  • Placebo test: run DiD on a fake treatment date in the pre-period — should find no effect
  • If violated: DiD is biased — consider synthetic control methods instead
🔵 Theory: Parallel trends is untestable (it is about the counterfactual). Pre-period parallelism is a necessary but not sufficient condition.

DiD in code


means = (
    df.group_by(["group", "period"])
    .agg(pl.col("price").mean().alias("mean_price"))
)
# ATT = (treat_post - treat_pre) - (ctrl_post - ctrl_pre)
treat_pre  = means.filter(...).item()
treat_post = means.filter(...).item()
ctrl_pre   = means.filter(...).item()
ctrl_post  = means.filter(...).item()

att = (treat_post - treat_pre) - (ctrl_post - ctrl_pre)
print(f"ATT: {att:,.0f}")

Three mistakes to avoid

1. Post-experiment covariate in CUPED
Removes part of the treatment effect. Always use pre-experiment data.
2. Skipping SRM check
A broken experiment gives meaningless p-values. Check split first.
3. DiD without parallel trends check
Divergent pre-trends bias the estimate. Always plot and test.
