Speaker Notes — Lesson 1.5 Window Functions and Trends

Total time: ~55 minutes · 13 slides · ~4 min average

This lesson introduces window functions — the second-most-important Polars pattern after group_by. The core idea (enrich without collapsing) often clicks slowly. Use the before/after diagram liberally.

Slide 1~2 minTitle

Window Functions and Trends

Open with: "Yesterday you joined space — HDB plus MRT plus schools. Today we join time. You'll compute rolling means, year-over-year changes, and rank districts by growth. And you'll meet lazy frames, which is how you make long pipelines fast."

→ "Why we need windows."
Slide 2~3 minRaw vs smoothed chart

Raw data is noisy — trends live in windows

Walk the three lines. Red dashed = raw monthly prices, bouncing around. Blue solid = 3-month rolling mean, still wobbly but clearer. Teal thick = 12-month rolling mean, smooth enough to show the real trend.

Tell them: "You cannot see whether Bishan is appreciating by looking at the red points alone. You need to smooth. Window functions are how you smooth."

→ "How windows differ from group_by."
Slide 3~4 mingroup_by vs window diagram

Windows don't collapse — they enrich

The conceptual core. Walk the left side: group_by collapses Bishan's three rows into one row with the mean. You lost the month detail.

Right side: the window computes the same mean but broadcasts it back to every row. Four input rows, four output rows, with a new column. Every row now carries its group's mean alongside the original value.

Tell them: "If you need to compare each row to its group's average — 'how does this transaction compare to the town's typical price?' — you need a window, not group_by."

For beginners: "group_by makes a summary table. Window adds a summary column. Different tools for different questions."
→ "The Polars syntax."
Slide 4~4 min.over syntax

The .over modifier

Walk the code. Read the chain aloud: "pl.col picks the column. rolling_mean applies a rolling 12-row window. over partitions by town so windows don't cross town boundaries. alias names the result."

Run it live. Print Bishan's last 15 months. Show the null early rows (insufficient history) and the real values that follow. Make sure they see the null — tell them it's expected.

→ "The math behind rolling_mean."
Slide 5~3 minRolling mean formula

Rolling mean formula

Walk the formula. Unweighted mean of the last w observations. The sum from $j=i-w+1$ to $i$ is just "the window of size w ending at row i".

Mention the variants briefly: rolling_sum, rolling_min, rolling_max, rolling_std. Same shape, different aggregation.

→ "Choosing the window size."
Slide 6~3 minWindow trade-off table

Window size trade-off

Walk the table. Short windows are reactive but noisy. Long windows are smooth but lag. There's no right answer — it depends on what you're trying to detect.

Give the technical analysis analogy: "In finance, traders plot a 50-day and a 200-day moving average together. When the short one crosses above the long one — the 'golden cross' — they call it a buy signal. Same idea applies to HDB prices."

For experts: "The variance of the smoothed curve is $\sigma^2/w$. That's why doubling the window halves the wobble. The cost is a lag of $\sim w/2$ — the rolling mean always trails real trend changes."
→ "Another window primitive — shift."
Slide 7~3 minshift

shift — lag features

Run it live. Show that shift(12) pushes the value from 12 rows earlier into the current row. First 12 rows of every town become null.

Explain why we use over("town"): without it, shift would pull the last 12 rows of Bishan into Jurong's first 12 rows. Always partition when you have groups.

→ "Combining them — YoY."
Slide 8~4 minYoY formula

Year-over-year change

Walk the formula. Current minus 12 months ago, divided by 12 months ago, times 100. Standard percent change, applied to the shifted column.

The seasonality argument is the key teaching moment. "Month over month is dominated by calendar effects — Chinese New Year suppresses transactions, year-end pushes them up. Year over year compares March to March, so those effects cancel."

For beginners: "If you want to know whether your business is really growing, compare this month to the same month last year, not to last month. Same logic here."
→ "Putting it all together — ranking."
Slide 9~3 minRanking code

Ranking the hottest districts

Walk the code. Filter to the latest date. Drop rows with null YoY (no history). Sort by YoY descending. Print top 5, bottom 5.

Run it live. Show the results. Ask which towns they expected to see at the top — usually the central/mature estates. Sometimes there are surprises.

→ "The production-speed version — lazy frames."
Slide 10~4 minLazy frames diagram

🟣 Lazy frames — global optimisation

Walk the before/after. Left side (eager): each step materialises a DataFrame, even if the next step will immediately shrink it. Right side (lazy): each step adds a plan node. collect() executes the whole plan in one optimised pass.

The killer optimisation: filter pushdown. "If your chain filters to year ≥ 2020, lazy can push that filter into the parquet scan — it never even loads the rows before 2020. Eager would load all 500,000 rows, then throw 320,000 away."

For experts: "Polars uses a rule-based optimiser similar to Spark's Catalyst. Projection pushdown, predicate pushdown, operator fusion, parallel scan. The plan is visible via .explain() if you're curious."
→ "The code."
Slide 11~3 minLazy code

Lazy pipeline in code

Walk the code. Point out the two key differences from eager: start with scan_parquet instead of read_parquet, end with collect(). Everything in between is identical.

Tell them: "scan reads lazy. read reads eager. collect executes. If you learn those three keywords you can convert any pipeline to lazy mode by changing two lines."

Give the rule of thumb: "Lazy for production and big data. Eager for interactive exploration on small data. Interactive feedback is nicer; production throughput is faster."

→ "Common mistakes."
Slide 12~3 minTraps

Four traps

Walk each. The sort trap is the most common — show it live by computing a rolling mean on unsorted data and pointing at the mess.

The over trap is easy to miss if they don't have Polars habit yet. Show it: rolling mean without over gives nonsense values where one town's window bleeds into another's data.

The early-null trap is philosophical but important. "Those nulls mean 'insufficient history'. Filling with zero is injecting fake averages into your model. Drop them instead."
