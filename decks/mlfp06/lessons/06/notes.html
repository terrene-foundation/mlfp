Speaker Notes — Lesson 6.6Multi-Agent Orchestration and MCP

Total time: ~90 minutes  ·  9 slides  ·  ~10 min per slide

This lesson scales agents from individuals to teams. The four patterns (Slide 2) are the conceptual anchor. The MCP server (Slide 5) is the most practically useful — it exposes tools that any agent can use. Live demo the multi-agent ML pipeline if possible. The security discussion (Slide 7) is critical for enterprise contexts.

Slides 1–2~10 minTitle + patterns

Four patterns

Walk the pattern table. "Supervisor-worker: a manager delegates sub-tasks. Sequential: assembly line — each agent's output feeds the next. Parallel: independent tasks run simultaneously and results are merged. Handoff: one agent transfers to another when the topic changes — like being transferred to a different department." Singapore context: "A GovTech chatbot uses handoff — if a user asks about CPF, it hands off to the CPF specialist agent."

→ "Let's build a multi-agent pipeline."
Slides 3–5~25 minPipeline, memory, MCP

Pipeline code, memory, and MCP server

Walk the supervisor code. Then the memory table: "Short-term is the current conversation. Long-term persists across sessions — the agent remembers your preferences. Entity memory structures knowledge about specific things — people, companies, regulations." Then the MCP server: "MCP is a protocol for exposing tools. Build an MCP server once, and any MCP client — Claude, ChatGPT, your own agents — can use those tools."

→ "Security."
Slides 6–9~15 minA2A, security, mistakes, recap

A2A, security, and recap

Walk A2A briefly. Security in depth: "In a financial system, the customer-facing agent must not have access to internal compliance data. The compliance agent must not see customer PII. Agent boundaries enforce this — like departments in an organisation." Preview 6.7: "Next we engineer governance — PACT D/T/R addressing, operating envelopes, budget cascading. The code IS the governance."

→ End of lesson.
