Lesson 1.7 Automated Data Profiling

For six lessons you have profiled data by hand: describe, head, null counts, correlations, histograms. Today you meet the engine that does all of it in one call and adds eight kinds of quality alerts you might not have thought to check. DataExplorer is the first Kailash engine you use as a consumer rather than a builder — the same shift you will make again and again as the course progresses toward higher-level tools.

Why This Matters

Real datasets are messy. Sensors drop readings. Systems change schemas mid-stream. Manual entry produces "N/A", "n/a", "unknown", and blank cells in the same column. Dates arrive as three different string formats. Economic indicators spike during crises and break your assumptions about normality. Every one of these issues costs hours to diagnose if you find it late; it costs seconds to catch if you find it early.

The ritual you built in Lesson 1.1 — shape, columns, head, describe — catches the obvious problems. But it misses the subtler ones: a column with 12% nulls (bad but not empty), a date column with three formats hidden inside, a feature that is 98% zero, two columns that are 99% correlated and add nothing independent. DataExplorer catches all of those automatically.

Core Concepts

Engines as services

Up to now you have used Polars primitives and written your own logic. DataExplorer is the first time you consume a service — a pre-built engine whose job is to look at any DataFrame and tell you what is wrong with it. You do not need to know how the internals work; you only need to know the shape of the input, the shape of the output, and the configuration knobs you can turn.

🟢 Foundations

The shift from "write it yourself" to "configure a service" is the single biggest productivity jump in data engineering. Services handle edge cases you would not think of, scale to data sizes you cannot profile manually, and come with test coverage you cannot match in a notebook. The trade-off is that you cede control to the service's defaults. That is why you need to understand what it is doing — so you can tune it when the defaults are wrong.

The eight alert types

DataExplorer runs each of its eight quality checks on every column and returns an alert if any threshold is exceeded. Each alert has a type, a column, a value, and a severity.

AlertMeaningFix
high_nullsColumn has too many missing valuesImpute, drop, or flag
constantColumn has ≤ 1 unique value (no information)Drop the column
high_cardinalityNear-unique categorical columnBin, hash, or treat as ID
high_skewnessExtreme right or left tailLog-transform or winsorise
high_zerosColumn has many exact zerosCheck if zero is real or missing-coded
high_correlationPair of columns is near-collinearDrop one, or flag for VIF check
duplicatesDataset has duplicated rowsDeduplicate or investigate the pipeline
imbalancedCategorical column dominated by one valueRebalance, SMOTE, or stratify

AlertConfig: tuning the thresholds

The default thresholds are set for typical tabular ML data. Your data is probably not typical. Economic time series are structurally correlated (high_correlation fires constantly with defaults). Property data has long right tails (high_skewness fires constantly). Every dataset has its own shape, and every shape has its own sensible thresholds. 

from kailash_ml.engines.data_explorer import AlertConfig

alert_config = AlertConfig(
    high_correlation_threshold=0.95,   # raise from 0.80 default
    high_null_pct_threshold=0.10,      # tighten from 0.15 default
    skewness_threshold=3.0,            # raise for crisis-era data
    zero_pct_threshold=0.30,           # relax for sparse trade data
)

explorer = DataExplorer(alert_config=alert_config)

Every threshold should be a deliberate choice, not a default. Document the rationale in a comment so a reviewer understands why you picked those numbers.

Async profiling

DataExplorer.profile() is an async function. Async is a Python feature for running operations concurrently — while one column is being profiled, Polars can start on the next. For wide DataFrames this is much faster than profiling columns sequentially. 

import asyncio

async def profile_dataset():
    explorer = DataExplorer(alert_config=alert_config)
    profile = await explorer.profile(df)
    return profile

profile = asyncio.run(profile_dataset())

Two pieces of vocabulary: the async def marks the function as asynchronous; the await keyword waits for the result of another async operation. asyncio.run(...) creates an event loop, runs the function, and returns the result. You do not need to understand the internals — treat the pattern as a template.

🔵 Theory: sync vs async

Synchronous code runs top to bottom, blocking on every I/O or CPU operation. Asynchronous code can suspend at an await point and let other work run in the meantime. For single-CPU work async is rarely faster, but for profiling many columns or files simultaneously it can cut total time by the number of parallel operations. Full async is a chapter of its own — for this lesson, the template above is all you need.

Error handling with try / except

Real profiling pipelines fail. The file might not exist. The schema might have changed. A column might have values describe() cannot handle. Python's try / except lets you catch errors gracefully and continue — or re-raise with a clearer message. 

try:
    profile = await explorer.profile(df)
except FileNotFoundError as e:
    print(f"Dataset not available: {e}")
    profile = None
except ValueError as e:
    print(f"Bad schema: {e}")
    profile = None

The Kailash Engine: DataExplorer in detail

⚙ Kailash Engine

DataExplorer profiles any Polars DataFrame and returns a DataProfile object with structured metadata: per-column statistics (type, mean, std, nulls, unique count, skewness), pairwise correlations (Pearson and Spearman), duplicate counts, and a list of alerts. It also supports compare(df_a, df_b) for distribution drift between two snapshots. Module 4's DriftMonitor uses the same comparison primitive to catch production data drift in deployed models.

Worked Example: Profiling Messy Singapore Economic Data

📝 Goal

Load three messy Singapore economic datasets (monthly CPI, quarterly employment, daily FX rates), merge them onto a monthly spine, and profile the result with DataExplorer. Interpret every alert and decide what to do about it.

Step 1.

Load the three datasets and inspect them individually. Notice the deliberately different reporting frequencies and date formats.

import asyncio
import polars as pl
from kailash_ml import DataExplorer
from kailash_ml.engines.data_explorer import AlertConfig
from shared import MLFPDataLoader

loader = MLFPDataLoader()
cpi = loader.load("mlfp01", "sg_cpi.csv")
employment = loader.load("mlfp01", "sg_employment.csv")
fx_rates = loader.load("mlfp01", "sg_fx_rates.csv")

print(f"CPI:         {cpi.height:,} rows (monthly)")
print(f"Employment:  {employment.height:,} rows (quarterly)")
print(f"FX:          {fx_rates.height:,} rows (daily)")
Step 2.

Normalise the messy CPI date column. The source has three formats in one column: 01/2000, 2000-02, and 200011.

cpi = cpi.with_columns(
    pl.col("date")
    .str.replace(r"^(\d{2})/(\d{4})$", "$2-$1-01")
    .str.replace(r"^(\d{4})(\d{2})$", "$1-$2-01")
    .str.replace(r"^(\d{4})-(\d{2})$", "$1-$2-01")
    .str.to_date("%Y-%m-%d")
    .alias("date")
)

Chained str.replace calls walk through the three possible formats, converting each to YYYY-MM-01. Then str.to_date parses the clean form. This is the pattern for any "three formats in one column" mess.

Step 3.

Merge to monthly using left joins and forward-fill for the quarterly employment series.

date_range = pl.date_range(
    cpi["date"].min(), cpi["date"].max(), interval="1mo", eager=True
)
monthly_spine = pl.DataFrame({"date": date_range})

# Forward-fill quarterly employment onto monthly spine
employment_monthly = (
    monthly_spine.join(employment, on="date", how="left")
    .sort("date")
    .with_columns([
        pl.col(c).forward_fill() for c in employment.columns if c != "date"
    ])
)

# Aggregate daily FX to monthly mean
fx_monthly = (
    fx_rates.group_by(pl.col("date").dt.truncate("1mo").alias("date"))
    .agg([pl.col(c).mean() for c in fx_rates.columns if c != "date"])
)

economic = (
    cpi.join(employment_monthly, on="date", how="left")
    .join(fx_monthly, on="date", how="left")
    .sort("date")
)
Step 4.

Configure alerts for the specifics of economic data. Structural correlation is expected; crisis-era skew is expected; hard thresholds would fire too many false alarms.

alert_config = AlertConfig(
    # Macro indicators are structurally correlated — only flag near-collinearity
    high_correlation_threshold=0.95,
    # Quarterly data forward-filled will have ~0% nulls mid-series, some at edges
    high_null_pct_threshold=0.10,
    # Constant = pipeline failure
    constant_threshold=1,
    # Crises create extreme skew; only flag severe
    skewness_threshold=3.0,
    # Some trade-flow columns legitimately zero
    zero_pct_threshold=0.30,
)
Step 5.

Run the profiler asynchronously and interpret each alert.

async def profile_economic():
    explorer = DataExplorer(alert_config=alert_config)
    profile = await explorer.profile(economic)

    print(f"Rows: {profile.n_rows}  Columns: {profile.n_columns}")
    print(f"Duplicates: {profile.duplicate_count} "
          f"({profile.duplicate_pct:.1%})")

    print(f"\n--- {len(profile.alerts)} Alerts ---")
    for alert in profile.alerts:
        print(f"[{alert['severity']}] {alert['type']}: "
              f"{alert.get('column', 'N/A')} = {alert.get('value')}")

    return profile

profile = asyncio.run(profile_economic())
Step 6.

Translate each alert into an action. Walk the output and decide.

# Example alert interpretation loop
for alert in profile.alerts:
    alert_type = alert["type"]
    col = alert.get("column", "N/A")
    value = alert.get("value")

    if alert_type == "high_nulls":
        print(f"  {col} has {value:.1%} nulls — "
              f"forward-fill or drop rows")
    elif alert_type == "constant":
        print(f"  {col} is constant — DROP, carries no info")
    elif alert_type == "high_skewness":
        print(f"  {col} skew={value:.2f} — log-transform or winsorise")
    elif alert_type == "high_correlation":
        print(f"  {col}: r={value:.2f} — drop one or run VIF")
    else:
        print(f"  {alert_type} on {col}: {value}")
Step 7.

Generate an HTML report with the built-in visualiser.

async def generate_report():
    explorer = DataExplorer(alert_config=alert_config)
    report = await explorer.visualize(economic)
    # The report is a Plotly figure or dict of figures
    # Write each to an HTML file
    return report

asyncio.run(generate_report())
print("✓ Profile report generated — check ex7_profile_*.html")

Mathematical Foundations: Skewness

Two of the alerts above — high_skewness and high_correlation — deserve a short mathematical tour.

Skewness

Skewness measures the asymmetry of a distribution. A symmetric distribution has zero skewness; a right tail has positive skewness; a left tail has negative skewness.

$$g_1 = \frac{1}{n} \sum_{i=1}^n \left(\frac{x_i - \bar{x}}{s}\right)^3$$
$\bar{x}$ — mean $s$ — standard deviation cubed so that sign is preserved

Why cube the standardised deviations? Cubing preserves sign (unlike squaring) so positive deviations (right tail) and negative deviations (left tail) do not cancel. The third power also exaggerates distant observations, which is exactly what we want — outliers should pull the metric more than average observations.

Rules of thumb: $|g_1| < 0.5$ is approximately symmetric; $0.5 \le |g_1| < 1$ is moderately skewed; $|g_1| \ge 1$ is highly skewed. A skewness above 3 is the threshold we set above — only severe crises produce tails that extreme.

Pearson correlation

The correlation coefficient $r$ measures the strength of a linear relationship between two variables:

$$r = \frac{\sum (x_i - \bar{x})(y_i - \bar{y})}{\sqrt{\sum (x_i - \bar{x})^2 \sum (y_i - \bar{y})^2}}$$
$r \in [-1, 1]$ $r = 1$ perfect positive $r = -1$ perfect negative

$|r| \ge 0.95$ is near-collinearity — the two variables carry almost identical information. Including both in a regression model inflates the coefficient standard errors (the variance inflation factor, VIF, you will meet in Module 2) and makes interpretation unreliable.

Common Mistakes

⚠ Mistake 1: Ignoring alerts because defaults fire too often

Default thresholds are a starting point, not scripture. If the defaults produce 40 false alarms, tune them. Do not silence them; retune them so each alert still means something.

⚠ Mistake 2: Profiling before cleaning basic issues

Mixed date formats, unparseable numbers, and obviously wrong types should be cleaned before profiling. Otherwise the profile shows garbage-in statistics and the real issues hide in the noise.

⚠ Mistake 3: Fixing correlation alerts by dropping columns automatically

"CPI is 99% correlated with employment so I'll drop employment" — but which one is your causal driver? Alerts prompt investigation, not automated removal. Understand the relationship first.

⚠ Mistake 4: Using sync profile() in an async context

profile() is async. You cannot call it directly from synchronous code — wrap it in asyncio.run(...) or the helper shared.run_profile(). Calling an async function without await produces a coroutine object, not a profile, and nothing useful happens.

Reflection

Automation is how you scale good habits. The 30-second ritual from Lesson 1.1 works for one dataset; it does not work for the hundreds of datasets a real data professional touches in a year. DataExplorer scales the ritual: one call, every column, eight alert types. You still have to interpret each alert and decide what to do — the engine does not make domain judgements for you — but it surfaces the decisions automatically instead of leaving them for you to stumble over later.

The mental shift today is toward consuming services instead of building them. Every higher-level Kailash engine you meet from here on (PreprocessingPipeline, TrainingPipeline, InferenceServer, DriftMonitor) follows the same pattern: a small API, a focused job, tunable configuration, and opinionated defaults. Your job is to know what the engine does, what it does not do, and how to configure it for your problem.

You will combine everything from this module — filter, aggregate, join, window, visualise, profile — into a single end-to-end pipeline that turns messy Singapore taxi trip data into a clean, model-ready dataset with an HTML report at the end. It is your first taste of production-shaped work.
