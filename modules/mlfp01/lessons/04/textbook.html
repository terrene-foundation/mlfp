Lesson 1.4 Joins and Multi-Table Data

Real questions almost never fit in a single table. "Is Bishan more expensive because it is closer to the MRT?" requires an HDB table and an MRT table, joined on a shared key. Today you learn how to combine datasets with join, when to use left versus inner versus outer joins, and how to handle the nulls that appear when one side has no match. You also meet Python's if/elif/else statement and the import keyword — the final pieces needed to write multi-file programs.

Why This Matters

Every real analytics question is a join question waiting to happen. The HDB price you computed in Lesson 1.3 is a single number; the context that makes it meaningful lives in other tables: MRT proximity, school density, nearby amenities, historical cooling measures, household income for the postcode. Each of those data sources arrives as its own CSV or parquet file. Your job is to line them up by a shared key — usually a town name, a postal code, a date — and pull the relevant columns across.

Core Concepts

Imports and packages

Before we can join anything, a word about the tools. The import keyword tells Python to load a package and make its contents available. You have already used import polars as pl and from shared import MLFPDataLoader. Two forms, same idea: 

import polars as pl              # load whole package, call it pl
from shared import MLFPDataLoader  # load one name from a package

The as clause is an alias — a short nickname. The Polars convention is pl. When you type pl.col(...) you are saying "the col function inside the Polars package".

🟢 Foundations

A package is just a folder of Python files that someone has published so you can reuse their code. PyPI (the Python Package Index) hosts hundreds of thousands. You install packages with pip install <name>; you use them with import <name>. That two-step — install once, import whenever — is how every non-trivial Python program is built.

if / elif / else: code-level decisions

Lesson 1.2 introduced pl.when / then / otherwise for choosing values inside a column. Python also has an if statement for choosing what to do at the code level. The two are not interchangeable: pl.when operates on whole columns; if operates on whole program paths. 

def price_range_label(price: float) -> str:
    """Classify a resale price into a human-readable tier."""
    if price < 350_000:
        return "Budget"
    elif price < 500_000:
        return "Mid-range"
    elif price < 700_000:
        return "Premium"
    else:
        return "Luxury"

Read it: if the price is below 350k return "Budget". Otherwise, if it is below 500k return "Mid-range". Otherwise, if below 700k return "Premium". Otherwise return "Luxury". Python checks each condition in order and stops at the first one that is True.

Joins: the core vocabulary

A join combines two tables by matching rows on a shared key. Suppose the HDB table has a column town and the MRT table also has a column town. A join on that key lines up rows where the two town values are equal and produces a new table with columns from both sides. 

hdb_enriched = hdb.join(
    mrt_stations.select("town", "nearest_mrt", "distance_to_mrt_km"),
    on="town",
    how="left",
)

Four things to notice: the left-hand DataFrame (hdb) calls .join; the right-hand DataFrame is the first argument; the on parameter names the shared key; the how parameter chooses which kind of join.

Handling nulls after a join

A left join keeps every row from the left table. For rows where the right table has no matching key, the new columns are filled with null. You almost always need to decide what to do with those nulls. 

# Count unmatched rows
n_null = hdb_enriched["distance_to_mrt_km"].null_count()
print(f"HDB rows with no MRT data: {n_null:,}")

# Fill with a sentinel (0, -1, "Unknown")
hdb_enriched = hdb_enriched.with_columns(
    pl.col("school_count").fill_null(0),
)

# Or drop the unmatched rows entirely
hdb_matched = hdb_enriched.filter(pl.col("distance_to_mrt_km").is_not_null())

Which choice is correct depends on the question. If you are counting schools and "no match" really does mean "zero schools", fill with 0. If "no match" means "data missing" and you cannot model missingness, drop. Never silently let nulls propagate into calculations without deciding.

🔵 Theory: Cardinality of joins

A join is one-to-one, one-to-many, or many-to-many depending on how unique the key is in each table. MRT stations has one row per town — the key is unique on the right side. HDB has many rows per town. The join is therefore many-to-one, and the output has the same row count as HDB. If both sides had duplicate keys the output would be a Cartesian product of the matches, which can blow up memory in a hurry. Always check the key uniqueness on both sides before joining.

Worked Example: Enriching HDB with Spatial Features

📝 Goal

Combine the HDB resale data with MRT station proximity and school counts, then produce a district summary that ranks towns by a composite "desirability" score computed with an if/elif/else function.

Step 1.

Load all three datasets.

import polars as pl
from shared import MLFPDataLoader

loader = MLFPDataLoader()
hdb = loader.load("mlfp01", "hdb_resale.parquet")
mrt_stations = loader.load("mlfp_assessment", "mrt_stations.parquet")
schools = loader.load("mlfp_assessment", "schools.parquet")

print(f"HDB:      {hdb.height:,} rows")
print(f"MRT:      {mrt_stations.height} rows")
print(f"Schools:  {schools.height} rows")
Step 2.

Check the overlap of join keys before joining. This is the step that catches 90% of join errors.

hdb_towns = set(hdb["town"].unique().to_list())
mrt_towns = set(mrt_stations["town"].unique().to_list())

matched = hdb_towns & mrt_towns       # intersection
unmatched = hdb_towns - mrt_towns     # in HDB but not in MRT

print(f"HDB towns:       {len(hdb_towns)}")
print(f"MRT towns:       {len(mrt_towns)}")
print(f"Matched:         {len(matched)}")
print(f"Unmatched HDB:   {len(unmatched)}")
if unmatched:
    print(f"Missing from MRT: {sorted(unmatched)}")

Matched towns will gain MRT columns; unmatched towns will get nulls under a left join (or disappear under an inner join). This check tells you what to expect before you run the join.

Step 3.

Left-join MRT stations onto HDB. We use .select(...) on the right side to limit which columns come across — otherwise every column of the MRT table would be attached.

hdb_enriched = hdb.join(
    mrt_stations.select("town", "nearest_mrt", "distance_to_mrt_km"),
    on="town",
    how="left",
)

print(f"Rows after MRT join: {hdb_enriched.height:,}")
# Should equal hdb.height — left join preserves all left rows
Step 4.

Schools is one row per school, so we need to aggregate to one-row-per-town before joining.

school_counts = (
    schools
    .group_by("town")
    .agg(pl.col("school_name").count().alias("school_count"))
)

hdb_enriched = hdb_enriched.join(
    school_counts,
    on="town",
    how="left",
).with_columns(
    pl.col("school_count").fill_null(0),  # no match → zero schools
)

The .fill_null(0) step matters. Without it, towns with zero schools in the source data would show up as null, which is different from zero — any arithmetic with nulls produces more nulls, so you would silently lose those towns in downstream calculations.

Step 5.

Write a helper function that classifies desirability using if / elif / else.

def desirability_tier(distance_km: float, schools: int) -> str:
    """Classify a town into four desirability tiers."""
    if distance_km is None:
        return "Unknown"
    if distance_km < 1.0 and schools >= 5:
        return "Prime (close MRT, many schools)"
    elif distance_km < 1.0:
        return "Good MRT access"
    elif schools >= 5:
        return "School-rich, farther MRT"
    else:
        return "Further out"
Step 6.

Aggregate to town level and apply the function using map_elements.

district_summary = (
    hdb_enriched.group_by("town")
    .agg(
        pl.col("resale_price").median().alias("median_price"),
        pl.col("distance_to_mrt_km").first().alias("distance_to_mrt_km"),
        pl.col("school_count").first().alias("school_count"),
        pl.len().alias("transactions"),
    )
    .with_columns(
        pl.struct(["distance_to_mrt_km", "school_count"])
        .map_elements(
            lambda x: desirability_tier(x["distance_to_mrt_km"], x["school_count"]),
            return_dtype=pl.String,
        )
        .alias("tier"),
    )
    .sort("median_price", descending=True)
)

print(district_summary.head(10))

pl.struct([...]) bundles two columns into a single argument for map_elements, which calls your Python function once per row. This is the bridge between vectorised Polars expressions and plain Python logic — use it sparingly, because it's slower than a pure Polars expression.

Mathematical Foundations: Set Operations

Joins are a direct application of set theory. Treat each table's key column as a set. Let $H$ be the set of HDB towns and $M$ be the set of MRT towns.

$$H \cap M = \{t : t \in H \text{ and } t \in M\}$$
$\cap$ — intersection the towns present in both tables

An inner join keeps exactly the rows whose keys lie in $H \cap M$. A left join keeps all rows from $H$, attaching right-side columns when the key lies in $H \cap M$ and nulls otherwise. A full outer join keeps all rows from the union $H \cup M$.

The set difference $H \setminus M = \{t : t \in H, t \notin M\}$ tells you which HDB towns have no matching MRT row — exactly the nulls you saw in Step 2 of the worked example.

Common Mistakes

⚠ Mistake 1: Inner join when left was meant

You start with 500,000 HDB rows. You inner-join with MRT. You end with 480,000 rows. Twenty thousand transactions silently disappeared — the ones whose towns had no MRT match. Left join preserves them; inner drops them. Always know which you wanted.

⚠ Mistake 2: Duplicate keys on both sides

If both sides have many rows per key, the join produces a Cartesian product of matches. 5 HDB rows × 5 MRT rows for the same town → 25 output rows instead of 5. Aggregate one side to one-row-per-key before joining (as we did with school_counts).

⚠ Mistake 3: Ignoring nulls after a join

Left joins produce nulls for unmatched rows. Arithmetic with nulls propagates nulls (null + 5 = null). If you do not fill or drop those nulls, downstream aggregations silently become nulls and you lose data without noticing.

⚠ Mistake 4: Mismatched key types or cases

Joining "BISHAN" with "Bishan" produces zero matches — the strings are not equal. Before joining, normalise both sides with str.to_uppercase() or str.strip_chars().

Reflection

Today you learned the vocabulary of multi-table work: import to load libraries, if/elif/else for code-level branching, and join to combine DataFrames by key. You built a concrete enrichment pipeline: start with HDB transactions, attach MRT proximity, attach school counts, produce a ranked district summary with a Python-function classifier.

The most important habit is the key check before every join — count the unique keys on each side, compute the intersection, decide what to do with the difference. A join without that check is how analysts silently lose twenty thousand rows and ship a broken report.
