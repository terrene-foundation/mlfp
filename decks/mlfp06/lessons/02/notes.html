Speaker Notes — Lesson 6.2 LoRA, Adapters, and the Fine-Tuning Landscape

Total time: ~75 minutes · 19 slides · ~4 min per slide average

This lesson is the mathematical spine of Module 6. By the end of it students should be able to draw LoRA on a whiteboard from memory and name the three most common reasons it fails. The central move is to connect Lesson 4.3 (SVD and low rank) to LoRA — students will feel they already understand the core idea, which lowers anxiety about fine-tuning. Keep the rank-8 arithmetic visible; the 250× reduction number is what makes this lesson memorable.

Slide 1~2 minTitle

LoRA, Adapters, and the Fine-Tuning Landscape

Open with a reframe. "Last lesson you squeezed every drop of behaviour out of a frozen model with prompting. Today we crack it open and change the weights. But we do it surgically — LoRA updates less than 1% of the parameters, and I will show you exactly why and how much less."

Name the two deliverables: (1) a LoRALayer written from scratch, (2) a completed AlignmentPipeline run on sg_domain_qa.parquet with a versioned adapter in the registry.

If beginners are anxious about "fine-tuning": reassure them. "We are not going to train a 7B model from scratch. We are going to update a handful of small matrices on a 1.1B base — the same API used in production, just a smaller base for teaching clarity."
→ "Why bother at all?"
Slide 2~3 minMotivation

When prompting is not enough

Walk the four bullets. Each one is a real scenario you have almost certainly seen. The regulatory tone one is particularly concrete in Singapore — "try asking gpt-4o to write in MAS guideline style and see how close it gets".

Then set up the constraint. "Full fine-tuning updates all 7 billion weights. On a 7B model in fp16 that is 14 GB just for the weights, plus 56 GB for Adam optimiser state. Nobody runs that on a laptop. So we need a compromise."

→ "The key observation is that the compromise is not just engineering — it is mathematical."
Slide 3~4 minLoRA equation

The LoRA insight

Read the equation aloud. "W equals W-zero plus delta W, and delta W is factored as B times A where rank r is much smaller than d or k."

Then connect it to Lesson 4.3 (SVD): "This is exactly low-rank matrix approximation. SVD says any matrix of rank r can be written as the product of a d-by-r and an r-by-k. LoRA asks the opposite question — what if we assume the update is low rank and train it in that parametrisation? The paper's empirical discovery is that fine-tuning updates really do have low intrinsic rank, so this assumption costs you almost nothing."

Work through the numbers on the board: d = 4096, k = 4096, r = 8, so full is 16.8M params, LoRA is 65k params, a 250× reduction. Pause and let them write it down.

For experts: mention that LoRA updates the "intrinsic dimension" of fine-tuning (Aghajanyan 2020) — the number of directions where loss actually decreases. Empirically ~1% of the parameter budget is all you need.
→ "Picture this explicitly."
Slide 4~3 minLoRA diagram

Picture of LoRA

Walk the diagram. Point at W_0 (frozen, dashed border, grey — "we never touch this"). Point at B (tall skinny matrix, d rows by r columns). Point at A (short wide matrix, r rows by k columns). Show that their product BA has the same shape as W_0 and can be added to it.

Emphasise that after training you can multiply B and A and add the result to W_0 once, producing a single fused matrix with no inference overhead. That "no inference overhead" property is why LoRA beats adapters for most production use cases.

→ "The forward pass and the init trick."
Slide 5~3 minForward pass + init

Forward pass + init

Read the formula. Explain the alpha-over-r scaling: "alpha decouples how much effect LoRA has from how much capacity LoRA has. You can bump the rank from 8 to 16 without having to retune the learning rate, because the scaling absorbs the change."

Then the init trick. "B is initialised to zero, A to Kaiming uniform. That means at step 0, BA equals zero, and the LoRA branch contributes nothing. The forward pass is numerically identical to the frozen base. That is why we don't need to worry that the very first training step might make the model worse — it literally cannot, because the perturbation is zero."

For beginners: "Think of it as bolting on an extra little module that starts out silent. Training is how we let that module learn to contribute something useful, one small step at a time."
→ "Show them the code."
Slide 6~4 minLoRALayer code

LoRA from scratch

Walk the class line by line. Freeze the base layer's parameters (the for p in ... loop). Create lora_A and lora_B. Initialise A with Kaiming, B with zeros. Forward pass computes base + scaled delta.

Point out the subtle matrix multiply order in the forward pass: (x @ lora_A.T) @ lora_B.T. This is because in PyTorch, linear layers store weights as (out, in), so the forward is x @ W.T. The LoRA branch follows the same convention.

For experts: in real libraries, the LoRA forward is actually x @ lora_A.T @ lora_B.T via two linear layers without bias, which PyTorch fuses efficiently. Our implementation is pedagogical.
→ "There's another PEFT — adapters."
Slide 7~3 minAdapter diagram

The other PEFT — adapter bottlenecks

Walk the diagram. "Activations come in. Project down to a small bottleneck dimension m. Non-linearity. Project back up. Add a residual connection. That's an adapter."

Highlight the "near-identity init" trick — zero the up-projection weights. Same idea as LoRA's zero-init on B: the module starts as a no-op and gradually learns to contribute.

→ "Which should you use?"
Slide 8~3 minLoRA vs adapters

LoRA vs adapters

Walk the table row by row. The decisive row is "inference cost" — LoRA is zero because it merges, adapters are +5–10% because they stay as extra modules. For most production use cases, merged LoRA wins.

Then the exception: "Adapters are better when you want multiple adapters active at once in the same forward pass. For example, one adapter for Singapore domain and another for safety, both on at once. LoRA forces you to merge — you cannot have two merged LoRAs stacked without mathematical interference."

→ "The full landscape."
Slide 9~4 min10-technique landscape

The 10-technique landscape

Walk the spectrum left to right. Point at each node as you name it. The colour coding matters: green = soft prompts, teal = PEFT, blue = training strategies on top of SFT, purple = heavy guns.

Deliver the main message: "LoRA is the default. Prefix/prompt tuning is for extreme parameter budgets. Distillation and full SFT are for when you have serious compute. DPSGD is for when your data is sensitive. Know that each one exists so you can reach for the right tool — do not memorise the details."

→ "Quick reference table."
Slide 10~2 minTechnique table

When to reach for which

Do not read this table aloud — it is a reference. Instead, ask them: "Put up a hand if you think your team's next LLM project will need privacy-preserving fine-tuning." (Usually zero hands.) "Put up a hand if you think it will need domain-specific knowledge absorption." (Usually most hands.) "Right — that's LoRA. The rest of the table is there for the 5% of cases that are not LoRA."

→ "But sometimes you want to combine fine-tunes."
Slide 11~3 minModel merging

Model merging

Name the motivating scenario: "You trained one LoRA for legal Q&A and another for retail support. Both work. But now you want one model that does both. Retraining from scratch is wasteful. Merging lets you linearly combine them."

Walk TIES conceptually: trim, elect sign, merge. Walk DARE: drop most updates, rescale. Walk SLERP: spherical interpolation between two checkpoints. Task arithmetic: add or subtract delta vectors. Keep it qualitative.

→ "SLERP is simple enough to show the formula."
Slide 12~3 minSLERP

SLERP in one formula

Walk the formula. The key intuition is that parameter vectors live on a high-dimensional sphere (their norms are roughly constant), and linear interpolation in Euclidean space takes you through a region where the norm drops. SLERP follows the great circle on the sphere, keeping the norm roughly constant through the interpolation.

For beginners: "Imagine two points on the Earth's surface. Linear interpolation would tunnel through the crust. SLERP walks along the surface. The surface is the right metaphor because neural network weight magnitudes are roughly constrained to live in a narrow band."
→ "Quantisation — how you actually fit this on your hardware."
Slide 13~3 minQuantisation / QLoRA

Quantisation & QLoRA

Walk the four formats briefly. GPTQ and AWQ are the production inference formats. GGUF is for llama.cpp. bitsandbytes is what you use during training.

Then the QLoRA story: "4-bit base + fp16 LoRA = fine-tune a 65B model on a single 48 GB GPU. This is the recipe that made large-model customisation accessible outside of big tech."

→ "Let's actually run something."
Slide 14~3 minAlignmentConfig

Configure AlignmentPipeline

Walk the config. Each line is a decision:

  • method="sft" — supervised fine-tuning
  • base_model — the starting point; use a small model for teaching
  • train_data / eval_data — preferably already split
  • lora_r=8, lora_alpha=16 — the canonical starting point
  • target_modules=["q_proj", "v_proj"] — the safe default
  • Training hyperparameters: 3 epochs, batch size 4, LR 2e-4
→ "Train and register."
Slide 15~3 minTrain + register

Train and register the adapter

Walk the code. pipeline.train() is async because it streams progress events. result.adapter_path is the directory containing A and B. registry.register(...) writes a manifest entry so future lessons can load the adapter by name and version.

Stress the registry pattern: "In Lesson 6.8 we will load sg_domain_sft:v1 from exactly this registry, wrap it in PACT governance, and deploy via Nexus. The fact that we save it cleanly today is what makes that possible."

→ "How do you know it's working?"
Slide 16~3 minReading loss curves

Reading the loss curves

Walk the four patterns. Healthy, overfitting, underfitting, leakage. For each, name the symptom and the fix.

Tell them the order of operations when diagnosing: (1) check the split first — leakage is embarrassing. (2) If underfitting, raise the rank before the LR. (3) If overfitting, add dropout before shortening epochs. (4) If nothing works, the base model may be wrong for the task.

→ "Parameter reduction arithmetic."
Slide 17~3 minReduction formula

Parameter reduction

Walk the formula. Plug in r=4, 8, 16, 32 and read the reduction percentages. "Notice that doubling the rank doubles the parameters. There is no free lunch — higher rank means more capacity and more compute."

Give them the mental model: "Rank is like a dial. Start at 8. If the model is underfitting, turn it up to 16 or 32. If it's already fitting fine, leave it alone — more rank wastes parameters."

→ "Four traps."
Slide 18~3 minCommon mistakes

Four common mistakes

Walk each one. Mistake 1 (target modules) is the one experts get wrong. Mistake 2 (rank too small) is the symptom of a plateau. Mistake 3 (forgetting to load) is the silent failure that makes you think LoRA didn't work when actually you were benchmarking the base model. Mistake 4 (catastrophic forgetting) is the reason dropout and progressive freezing exist.

→ "Reflect and move on."
Slide 19~3 minReflection / bridge

What you can now do

Read the bullets slowly. Stress the from scratch angle — "you just wrote LoRA yourself. Every library you use for the rest of your career is just a fancier version of what you already understand."

Tease Lesson 6.3: "We have made the model fit our data. But fitting data is not the same as being good. A model trained on random internet text will happily help someone do harmful things. Lesson 6.3 introduces preference alignment — we will derive DPO from the RLHF objective, see how to encode human preferences as training signal, and watch a base model become a safer one. The pipeline is the same; the loss function changes."

Take 5 minutes for questions. The most common after 6.2 are: "How do I know what rank to start with?" (8), "Can I train LoRA on CPU?" (technically yes, practically no), "Do I need an H100?" (no — the worked example runs on a T4), and "What's the difference between LoRA and adapter in practice?" (for a single task, almost none — pick LoRA for the zero-inference-cost win).

→ End of Lesson 6.2. Take a 10-minute break.
