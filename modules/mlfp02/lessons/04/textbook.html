Lesson 2.4 A/B Testing and Experiment Design

An A/B test is a hypothesis test with a budget, a timeline, a randomisation scheme, and a stakeholder waiting for an answer. This lesson assembles everything from 2.3 into an end-to-end experimental workflow: pre-registering hypotheses, computing sample sizes before data collection, running sanity checks (SRM detection), applying the right test, and delivering a business recommendation.

Why experiments beat observation

The fundamental problem of causal inference is that we can never see the counterfactual — we see what happened in the treatment group, and we see what happened in a different group, but we never see what would have happened to the treatment group without treatment. Randomisation is the one reliable way to ensure that the treatment and control groups are, on average, identical in every way except the treatment.

Pre-registering the hypothesis

Before you touch the data you must commit to:

  1. The unit of randomisation (user, session, device).
  2. The metric you are testing — one primary metric, up to three secondary.
  3. The direction of the alternative ("treatment improves conversion").
  4. The decision rule — $\alpha$, power target, and minimum detectable effect.
  5. The stopping criterion — when to stop collecting data.

Pre-registration is not bureaucracy; it is the discipline that keeps you from p-hacking your way to a false positive. The most common failure mode of industrial A/B tests is running until the metric is significant and then stopping — a procedure that guarantees a false positive rate far higher than $\alpha$.

Power analysis: sizing the experiment

In Lesson 2.3 you saw the power formula. Rearranged for sample size per arm:

$$n \;=\; \left(\frac{z_{1-\alpha/2} + z_{1-\beta}}{\delta/\sigma}\right)^2 \cdot 2$$
$\delta$ — minimum effect you care about $\sigma$ — per-user std of the metric $z_{1-\alpha/2}, z_{1-\beta}$ — Normal quantiles

For a proportion test with a baseline conversion rate $p_0$ and an absolute lift $\Delta$, the per-arm sample size is:

$$n \;=\; \frac{2\,(z_{1-\alpha/2} + z_{1-\beta})^2 \cdot p_0(1 - p_0)}{\Delta^2}$$

Sanity checks: Sample Ratio Mismatch

Before you look at the outcome metric, look at the allocation. If you randomised 50/50 and the logs show 52/48, something is wrong with the pipeline — the randomiser may be broken, bot filtering may be unequal, or a redirect bug may be dropping some users. This is called a Sample Ratio Mismatch (SRM), and its presence invalidates every subsequent analysis.

Test for SRM with a one-df chi-squared test:

$$\chi^2 \;=\; \sum_{k} \frac{(O_k - E_k)^2}{E_k}$$
$O_k$ — observed count in arm $k$ $E_k$ — expected under the randomisation ratio
⚠ Strict SRM threshold

Use $p < 0.0001$ — not $0.05$ — as your SRM alarm. SRM is a data-quality red flag; the only reason to run the test is to catch catastrophic pipeline errors. If you use a loose threshold, you will false-alarm on many healthy experiments and waste engineering time debugging nothing.

The Data Collection Framework

Before you instrument an experiment you need to answer five questions. This is the Why/What/Where/How/Frequency framework from the original MLFP curriculum; it scales from one-off A/B tests to enterprise data pipelines.

Why

  • What hypothesis is being tested?
  • What is the value/impact of answering it?
  • What decision will the result drive?

What

  • What is the ideal information?
  • What do you have budget/time for?
  • Primary + secondary + guardrail metrics

Where

  • Internal sources: CRM, POS, app events, finance
  • External sources: Kaggle, data.gov.sg, vendor APIs
  • Existing vs to be instrumented

How

  • At start: review, discover, validate existing data
  • Continuous: automated (real-time / batch)
  • Human augmented: manual annotation, QA

The fifth question — frequency — is about cadence. Per-transaction vs batch; collection frequency vs analysis frequency; and the minimum duration required for statistical power. A seven-day experiment cannot see weekly cycles. An experiment that runs for fewer users than your MDE requires is a wasted experiment.

Worked Example: BOGO A/B Experiment

📝 The setup

A Singapore e-commerce platform wants to know whether a Buy-One-Get-One (BOGO) promotion increases conversion rate. Baseline conversion is $p_0 = 0.08$. The product manager cares about detecting a $\Delta = 0.01$ (one percentage point) lift with 80% power at $\alpha = 0.05$.

Step 1 — Compute required sample size. 
import numpy as np
from scipy import stats

p0, delta = 0.08, 0.01
alpha, power_target = 0.05, 0.80

z_alpha = stats.norm.ppf(1 - alpha / 2)
z_beta  = stats.norm.ppf(power_target)
n_per_arm = int(np.ceil(
    2 * (z_alpha + z_beta)**2 * p0 * (1 - p0) / delta**2
))
print(f"n per arm = {n_per_arm:,}")

Expect around $n \approx 5{,}800$ per arm. Run for at least that many per group before peeking.

Step 2 — Load the data and check allocation (SRM). 
import polars as pl
from shared import MLFPDataLoader

df = MLFPDataLoader().load("mlfp02", "ab_test.parquet")

counts = df.group_by("arm").agg(pl.len().alias("n"))
print(counts)

n_c = counts.filter(pl.col("arm") == "control")["n"][0]
n_t = counts.filter(pl.col("arm") == "treatment")["n"][0]
expected = [(n_c + n_t) / 2] * 2
chi2, p_srm = stats.chisquare([n_c, n_t], f_exp=expected)
print(f"SRM chi² = {chi2:.3f}, p = {p_srm:.4f}")
assert p_srm > 1e-4, "SRM detected! Investigate pipeline."
Step 3 — Run the primary test. 
from statsmodels.stats.proportion import proportions_ztest, confint_proportions_2indep

k_c = df.filter(pl.col("arm") == "control")["converted"].sum()
k_t = df.filter(pl.col("arm") == "treatment")["converted"].sum()

z, p = proportions_ztest([k_t, k_c], [n_t, n_c])
lift = k_t / n_t - k_c / n_c
lo, hi = confint_proportions_2indep(k_t, n_t, k_c, n_c, method="wald")

print(f"Lift     = {lift:.4f}")
print(f"95% CI   = [{lo:.4f}, {hi:.4f}]")
print(f"p-value  = {p:.4f}")
Step 4 — Report.

A business-ready summary uses plain language, reports uncertainty, and avoids "we prove the treatment works":

Recommendation: Ship BOGO. The treatment increased conversion rate by 1.3 percentage points (95% CI: 0.6 to 2.0, $p = 0.003$), exceeding the pre-registered MDE of 1pp. At current traffic, this corresponds to approximately $\$240{,}000$ additional revenue per quarter. No SRM detected; no guardrail metrics degraded.

Common mistakes

⚠ Mistake 1: Peeking

Checking the p-value every hour and stopping as soon as it crosses $0.05$. This inflates the false-positive rate far above $\alpha$. If you must monitor, use a sequential test (Lesson 2.7).

⚠ Mistake 2: Running without power analysis

An underpowered experiment produces two bad outcomes: (1) no significance, so you dismiss a real effect; (2) significance by luck, so you ship something with a much smaller true effect than the observed one. Pre-register the sample size.

⚠ Mistake 3: Ignoring SRM

If allocation is off, the outcome analysis is meaningless. SRM comes first. Skip this at your peril.
