Lesson 1.3 Functions and Aggregation

Yesterday you filtered Bishan, then Toa Payoh, then Queenstown — one filter at a time. Today you learn two ideas that make that approach obsolete. Functions let you package reusable logic once and call it as many times as you like. Group-by aggregation lets Polars compute a summary for every town, every flat type, and every year in a single call. Together they turn what would have been an afternoon of cut-and-paste into six lines of code.

Why This Matters

Singapore has 26 HDB towns. Yesterday's pattern — filter, then summarise, then write down the numbers — would mean 26 filters, 26 summaries, and 26 rows of notes. If you then want the same summary by flat type (seven categories) you multiply: 182 filters. By year (twelve years)? 2,184 filters. No analyst does that by hand. They use group_by.

The mental shift is from one-at-a-time to all-at-once. Instead of asking "what is the median price in Bishan?", you ask "what is the median price in every town?" — and get a 26-row table back. Instead of asking "how many transactions happened in 2023?", you ask "how many transactions happened in every year?". The single-row answers you used yesterday become rows in a bigger answer.

Core Concepts

Functions: reusable logic

A function is a block of code you name so you can call it again. The syntax has four ingredients: the keyword def, the name, the parameter list in parentheses, and the indented body. A return statement sends a value back to the caller. 

def format_sgd(amount: float) -> str:
    """Format a number as Singapore dollars with thousands separator."""
    return f"S${amount:,.0f}"

print(format_sgd(485_000))   # → S$485,000
print(format_sgd(1_234_567)) # → S$1,234,567

The colon after the parameter name is a type hint — it documents what type you expect. The arrow -> after the parameter list is the return type. Python does not enforce these hints at runtime; they are for humans and static analysers. Use them anyway — they double as documentation.

🟢 Foundations

A function you write is a noun. You can pass it around, call it, even store it in a variable. Think of it as a tiny reusable machine: feed in the arguments, get back a result. The machine does not care who calls it, as long as the arguments match what it expects.

Why functions matter

Three reasons: reuse, readability, and testability. If you find yourself copying the same four lines of code, wrap them in a function. Now instead of four lines repeated five times (20 lines, four places to fix a bug), you have a five-line function plus five one-line calls (10 lines, one place to fix). Halved work, halved risk.

Lists and dictionaries

Two collections you will see constantly in Python. A list is an ordered sequence; a dictionary is a mapping from keys to values. 

central_towns = ["BISHAN", "TOA PAYOH", "QUEENSTOWN", "BUKIT MERAH"]
print(central_towns[0])    # → BISHAN
print(len(central_towns))  # → 4

town_medians = {"BISHAN": 715_000, "TOA PAYOH": 680_000, "JURONG WEST": 455_000}
print(town_medians["BISHAN"])  # → 715_000

for loops

A for loop walks through a sequence one item at a time. Combined with functions, it lets you apply the same operation to every element of a list. 

for town in central_towns:
    subset = hdb.filter(pl.col("town") == town)
    print(f"{town}: {subset.height} transactions")

You could do all four towns in a group_by instead — and for large data you should. But the for loop is still useful when you want to print each result, or save a file per town, or do anything Polars cannot vectorise.

group_by and agg

The star of the lesson. group_by splits the DataFrame into groups by one or more columns; agg computes one or more statistics for each group. The result has one row per group and one column per aggregation. 

district_stats = (
    hdb.group_by("town")
    .agg(
        pl.len().alias("transaction_count"),
        pl.col("resale_price").mean().alias("mean_price"),
        pl.col("resale_price").median().alias("median_price"),
        pl.col("resale_price").std().alias("std_price"),
    )
    .sort("median_price", descending=True)
)

Read it: "group the HDB data by town; for each group compute four aggregates; sort the result by median price descending." The output is a 26-row DataFrame — one row per town — with four numeric columns. You just replaced 26 manual filters.

Multi-key grouping

Pass multiple columns to group_by and Polars groups by every unique combination. "Every (town, flat_type) pair" produces ~180 groups — one for each combination that actually appears in the data. 

town_flat_stats = (
    hdb.group_by("town", "flat_type")
    .agg(
        pl.len().alias("count"),
        pl.col("resale_price").median().alias("median_price"),
    )
    .sort("town", "flat_type")
)
🔵 Theory

Internally, Polars computes the hash of the group key for each row and accumulates values in a hash table keyed on that hash. For a single integer key the cost is essentially $O(n)$ — one pass through the data, one hash per row, constant-time insertion. For $k$ keys the same cost applies with a slightly larger hash. This is the same algorithm as SELECT ... GROUP BY ... in SQL; Polars just runs it locally, in parallel, over a columnar format.

Worked Example: A District Price Report

📝 Goal

Produce a ranked report of every HDB town, with the transaction count, median resale price, price spread (IQR), and median price per square metre. Print the top 10 most expensive towns.

Step 1.

Load the data and add the price-per-sqm column.

import polars as pl
from shared import MLFPDataLoader

loader = MLFPDataLoader()
hdb = loader.load("mlfp01", "hdb_resale.parquet").with_columns(
    (pl.col("resale_price") / pl.col("floor_area_sqm")).alias("price_per_sqm"),
)
Step 2.

Write a helper function that formats a number as Singapore dollars — we will call it dozens of times across the report.

def format_sgd(amount: float) -> str:
    """Format a number as Singapore dollars with thousands separator."""
    return f"S${amount:,.0f}"
Step 3.

The group-by aggregation. One call gives us six statistics per town.

district_stats = (
    hdb.group_by("town")
    .agg(
        pl.len().alias("transaction_count"),
        pl.col("resale_price").median().alias("median_price"),
        pl.col("resale_price").quantile(0.25).alias("q25"),
        pl.col("resale_price").quantile(0.75).alias("q75"),
        pl.col("price_per_sqm").median().alias("median_price_sqm"),
    )
    .with_columns(
        (pl.col("q75") - pl.col("q25")).alias("iqr_price"),
    )
    .sort("median_price", descending=True)
)

Notice we chained with_columns after the agg to derive the IQR. Once the aggregation is done, the result is just another DataFrame — all the Lesson 1.2 tools still work.

Step 4.

Iterate over the top 10 rows with a for loop and print a formatted report line using our helper function.

def district_line(row: dict) -> str:
    return (
        f"  {row['town']:<20} "
        f"{format_sgd(row['median_price']):>12}  "
        f"{row['transaction_count']:>8,}  "
        f"IQR={format_sgd(row['iqr_price']):>10}  "
        f"{format_sgd(row['median_price_sqm']):>10}/sqm"
    )

print("=" * 70)
print("  SINGAPORE HDB — TOP 10 MOST EXPENSIVE TOWNS")
print("=" * 70)
for row in district_stats.head(10).iter_rows(named=True):
    print(district_line(row))
print("=" * 70)

.iter_rows(named=True) yields each row as a dictionary so you can access fields by name (row["town"]) rather than by index. The format specifiers inside the f-string align the columns into neat output.

Step 5.

A cross-district summary on top of the aggregated table.

medians = district_stats["median_price"]
print(f"Most expensive district:   {format_sgd(medians.max())}")
print(f"Least expensive district:  {format_sgd(medians.min())}")
print(f"Average district median:   {format_sgd(medians.mean())}")
print(f"Price spread:              {format_sgd(medians.max() - medians.min())}")

Mathematical Foundations: Measures of Centre and Spread

agg gives you access to a dozen summary statistics. Four of them carry most of the conceptual weight.

Mean and median

The mean is the arithmetic average (Lesson 1.1 already introduced it):

$$ \bar{x} = \frac{1}{n} \sum_{i=1}^{n} x_i $$

The median is the middle value when the data is sorted. For an odd $n$ it is the single middle element; for an even $n$ it is the average of the two middle elements.

🟢 Mean vs median

Mean is pulled by outliers; median is not. One billion-dollar flat in a district of 300 flats will shift the mean by several thousand dollars but will not move the median at all. For skewed distributions like property prices, always report median for "typical" and mean for "total market value".

Interquartile range (IQR)

Quartiles split sorted data into four equal parts. $Q_1$ (the 25th percentile) is the value below which 25% of the data lies; $Q_3$ (the 75th percentile) is the value below which 75% lies. The interquartile range is

$$\text{IQR} = Q_3 - Q_1$$
$Q_3$ — 75th percentile $Q_1$ — 25th percentile spread of the middle 50%

IQR is the standard measure of spread for skewed data, for the same reason median is the standard measure of centre: it ignores outliers. A district with a tight middle 50% has a small IQR even if one or two flats were absurdly expensive.

Coefficient of variation

Sometimes you want to compare spread across districts. But standard deviation alone is misleading — a 50k std on a 200k mean is huge, while the same 50k std on a 1M mean is tiny. The coefficient of variation normalises:

$$ \text{CV} = \frac{s}{\bar{x}} \times 100\% $$

CV is unitless and directly comparable across groups. A CV of 25% means the standard deviation is a quarter the size of the mean — the data is moderately spread. A CV of 45% means the data is widely spread relative to its centre.

Common Mistakes

⚠ Mistake 1: Hardcoding values inside a function

A function that only works on one dataset is not a function — it is a script in disguise. def get_bishan_median(): return hdb.filter(...)["resale_price"].median() takes no parameters, so you cannot reuse it for Toa Payoh. Parameterise: def get_median(df, town): ....

⚠ Mistake 2: Forgetting .alias(...)

Inside agg, Polars will auto-name columns if you do not alias them, and the names are often unhelpful ("resale_price", "resale_price_2", …). Always give every aggregation an explicit .alias(...).

⚠ Mistake 3: Using a Python loop instead of group_by

A Python for loop that filters one town at a time and appends to a list is ten to a hundred times slower than the equivalent group_by. If you find yourself writing for town in towns: subset = df.filter(...), stop and rewrite as group-by.

Reflection

Two big ideas today: functions package logic for reuse, and group_by/agg computes statistics for every group in one call. The combination turns scripts into libraries. You wrote a format_sgd helper and used it in a district report, and you produced a 26-row summary table with a single Polars chain — no loops, no hand-copied filters.

The statistical vocabulary — mean, median, IQR, CV — is the one you will use for the next ten modules. Mean and median go together; IQR is the spread that ignores outliers; CV normalises spread across groups. When you see a table with a large IQR and a moderate std, you know instantly that the middle of the data is spread out but the tails are not.

In Lesson 1.4 you meet join — the operation that combines two DataFrames on a shared key. HDB prices on their own tell you nothing about MRT access; joined with an MRT stations table, they do.
