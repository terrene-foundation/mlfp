Speaker Notes — Lesson 3.7 Workflow Orchestration, Model Registry, and Hyperparameter Search

Total time: ~55 minutes  ·  13 slides  ·  ~4 min per slide average

This is the ML engineering lesson. The shift from "data scientist who writes scripts" to "ML engineer who builds systems" happens here. WorkflowBuilder may feel like over-engineering to learners who have been writing scripts for 20 lessons — show them the error handling and conditional branching to demonstrate the value.

Slides 1-2~4 minMotivation

From scripts to workflows

"Your script from Lesson 1.8 works. But what happens when train() fails? evaluate() never runs. You get a traceback and nothing else. WorkflowBuilder handles errors, retries, logs, and conditional branching."

→ "The node graph."
Slides 3-4~8 minWorkflowBuilder

WorkflowBuilder

Slide 3: Walk the diagram. "Five nodes, three types: PythonCodeNode for computation, ConditionalNode for branching, and the connections between them. The runtime executes in dependency order."

Slide 4: Walk the code. "Create the builder, create nodes, connect them, execute. The conditional gate checks if AUC exceeds 0.85 before promoting."

For beginners: "Think of it as Lego blocks. Each block does one thing. You snap them together in the order you want. If one block fails, the system tells you which one and why."
→ "How to find the best hyperparameters."
Slides 5-8~12 minHyperparameterSearch

Bayesian hyperparameter search

Slide 5: Walk the Bayesian optimisation diagram. "The surrogate model (green line) approximates the true objective (dashed). The uncertainty band shows where the model is least confident. The next point is chosen where expected improvement is highest."

Slide 6: Walk the code. Point out the log scale for learning_rate: "Learning rate varies over orders of magnitude, so we search in log space."

Slide 7: Walk the comparison table. "Grid search: exponential in the number of hyperparameters. Random: uninformed but covers the space. Bayesian: informed, uses 50 evaluations instead of 3000."

For experts: "The acquisition function (expected improvement, UCB, or probability of improvement) controls the explore-exploit tradeoff. Expected improvement is the default and works well in practice."
→ "What to do with the best model."
Slides 9-10~8 minModelRegistry

ModelRegistry and lifecycle

Slide 9: Walk the code. "Register with metrics, signature, and tags. The signature validates inputs in production — a model without a signature is a model that will break silently."

Slide 10: Walk the lifecycle diagram. "Experiment → staging → production → retired. Each transition is gated. Never auto-promote."

→ "Mistakes."
Slides 11-12~5 minMistakes + reflection

Wrap-up

Walk the three mistakes. Bridge to 3.8: "You can build the pipeline, tune the model, and register it. Tomorrow we deploy it: DataFlow for persistence, DriftMonitor for ongoing surveillance, Model Cards for documentation, and conformal prediction for uncertainty."

→ End. Break.
