Lesson 6.6 Multi-Agent Orchestration and MCP

A single agent is a specialist. A team of agents is an organisation. In Lesson 6.5 you built a ReAct agent that autonomously analyses data. Today you compose multiple agents into coordinated systems using four orchestration patterns — supervisor-worker, sequential, parallel, and handoff. You configure agent memory (short-term, long-term, entity) so agents retain context across turns and sessions. And you build an MCP server that exposes Kailash ML tools to any MCP-compatible client — turning your ML pipeline into a service that any agent, anywhere, can consume.

Why This Matters

A Singapore bank's credit risk team does not consist of one generalist who does everything. It has a data engineer who prepares features, a modeller who trains and validates, a compliance officer who checks regulatory requirements, and a report writer who produces the monthly risk summary. Each specialist has different skills, different tools, and different access levels.

Multi-agent systems mirror this organisational structure. Instead of one agent with 20 tools (which, as we learned in 6.5, performs poorly), you build specialised agents with 3-5 tools each, coordinated by a supervisor that delegates and aggregates. The result is more reliable, more auditable, and more maintainable than a monolithic agent.

MCP (Model Context Protocol) takes this further. Instead of hardcoding tools into each agent, you expose tools as an MCP server — a standardised API that any MCP-compatible client can discover and call. This decouples tool implementation from agent implementation, the same way REST APIs decoupled frontend from backend.

Core Concepts

Four Multi-Agent Patterns

Foundations

Think of multi-agent patterns like management structures. Supervisor-worker is a manager with direct reports. Sequential is an assembly line. Parallel is a team working on independent parts of a project. Handoff is a call centre that transfers you to the right department. Each structure is optimal for a different kind of work.

Math Foundations

Speedup from parallelism

If a task has $n$ independent subtasks, each taking time $t$, the sequential time is $T_{\text{seq}} = n \cdot t$. With $p$ parallel agents:

$$ T_{\text{par}} = \frac{n}{p} \cdot t + t_{\text{overhead}} $$
$n$ — number of subtasks $p$ — number of parallel agents $t_{\text{overhead}}$ — aggregation cost

The speedup $S = T_{\text{seq}} / T_{\text{par}}$ approaches $p$ for large $n$, but is bounded by Amdahl's law: if a fraction $f$ of the task must be sequential (e.g., the supervisor's delegation step), the maximum speedup is $1 / (f + (1-f)/p)$. In practice, the supervisor's planning and aggregation steps are the sequential bottleneck.

Cost model for multi-agent systems

Each agent has its own context window and cost trajectory. For a supervisor with $W$ workers:

$$ C_{\text{total}} = C_{\text{supervisor}} + \sum_{w=1}^{W} C_{\text{worker}_w} $$
$C_{\text{supervisor}}$ — cost of planning + aggregation $C_{\text{worker}_w}$ — cost of worker $w$'s full ReAct loop

The supervisor's cost is typically low (2-3 LLM calls for delegation and aggregation). Workers bear the bulk of the cost. Budget cascading (from Lesson 6.7) allocates a fraction of the total budget to each worker.

Theory

Multi-agent cost is not simply $W$ times single-agent cost. Each worker operates in a smaller context (only its subtask, not the full problem), so individual worker costs are lower. A 4-worker system typically costs 1.5-2x a single agent doing the full task, not 4x. The cost savings come from shorter context windows and fewer reasoning steps per agent.

Implementing Multi-Agent Patterns

Supervisor-Worker

from kaizen import BaseAgent, ReActAgent, Signature, InputField, OutputField
from kaizen.tools import tool

# Worker 1: Data profiler
class ProfileTask(Signature):
    """Profile a dataset and report quality issues."""
    dataset: str = InputField(desc="Dataset filename")
    profile_report: str = OutputField(desc="Data quality report")

profiler_agent = ReActAgent(
    signature=ProfileTask,
    tools=[explore_data, check_nulls, check_distributions],
    model="claude-sonnet-4-20250514"
)

# Worker 2: Model trainer
class TrainTask(Signature):
    """Train candidate models and return comparison."""
    dataset: str = InputField(desc="Dataset filename")
    target: str = InputField(desc="Target column name")
    comparison: str = OutputField(desc="Model comparison table")

trainer_agent = ReActAgent(
    signature=TrainTask,
    tools=[train_model, evaluate_model, cross_validate],
    model="claude-sonnet-4-20250514"
)

# Worker 3: Report writer
class ReportTask(Signature):
    """Write a summary report from profiling and training results."""
    profile_report: str = InputField(desc="From profiler agent")
    model_comparison: str = InputField(desc="From trainer agent")
    final_report: str = OutputField(desc="Executive summary")

report_agent = ReActAgent(
    signature=ReportTask,
    tools=[format_table, generate_chart],
    model="claude-sonnet-4-20250514"
)

# Supervisor orchestrates
class SupervisorTask(Signature):
    """Coordinate data analysis pipeline."""
    dataset: str = InputField(desc="Dataset to analyse")
    executive_report: str = OutputField(desc="Final analysis")

class MLSupervisor(BaseAgent):
    def run(self, dataset: str) -> str:
        # Step 1: Profile
        profile = profiler_agent.run(dataset=dataset)

        # Step 2: Train (uses profiling insights)
        training = trainer_agent.run(
            dataset=dataset,
            target=profile.identified_target
        )

        # Step 3: Report (combines both)
        report = report_agent.run(
            profile_report=profile.profile_report,
            model_comparison=training.comparison
        )
        return report.final_report

Sequential pattern

# Sequential: output of each agent feeds into the next
def sequential_pipeline(dataset: str) -> str:
    """Chain agents: profile → train → report."""
    step1 = profiler_agent.run(dataset=dataset)
    step2 = trainer_agent.run(
        dataset=dataset,
        target=step1.identified_target
    )
    step3 = report_agent.run(
        profile_report=step1.profile_report,
        model_comparison=step2.comparison
    )
    return step3.final_report

Parallel pattern

import asyncio
from kaizen import LLMCostTracker

async def parallel_analysis(datasets: list[str]) -> list[str]:
    """Analyse multiple datasets simultaneously."""
    tracker = LLMCostTracker(budget_usd=5.00)

    tasks = [
        profiler_agent.arun(dataset=ds, cost_tracker=tracker)
        for ds in datasets
    ]
    results = await asyncio.gather(*tasks)

    # Aggregate results
    combined = "\n\n".join(
        f"## {ds}\n{r.profile_report}"
        for ds, r in zip(datasets, results)
    )
    return combined

# Analyse 3 Singapore datasets in parallel
datasets = ["hdb_resale.csv", "mrt_ridership.csv", "telco_churn.csv"]
report = asyncio.run(parallel_analysis(datasets))

A2A — Agent-to-Agent Protocol

When agents communicate, they need a structured protocol — not free-form text that the receiving agent must parse. The Agent-to-Agent (A2A) protocol defines a standard message format: 

from kaizen.protocol import A2AMessage

# Profiler sends structured output to trainer
message = A2AMessage(
    sender="profiler",
    receiver="trainer",
    task_id="analysis-001",
    payload={
        "dataset": "sg_telco_churn.csv",
        "identified_target": "churned",
        "quality_issues": ["2% nulls in tenure", "class imbalance 73/27"],
        "recommended_preprocessing": ["impute_nulls", "oversample_minority"]
    },
    metadata={"cost_so_far": 0.12, "step": 1}
)
Advanced

A2A messages carry metadata (cost, step number, timestamp) alongside the payload. This enables the supervisor to track progress without parsing agent output text. The protocol also supports priority levels and error states — if a worker fails, it sends an error message that the supervisor can handle (retry, skip, or abort).

Agent Memory

Without memory, every agent invocation starts from scratch. The agent does not remember what it profiled five minutes ago, what the user prefers, or what entities it has encountered. Three types of memory address this:

Memory typeScopePersistenceUse case
Short-termCurrent conversationSession onlyTrack the current task's state — what tools have been called, what results observed
Long-termAcross sessionsDatabase-backedRemember user preferences ("always use F1 for imbalanced data"), past analyses, project context
EntityStructured knowledgeDatabase-backedKnowledge about specific entities — "HDB resale prices: median $550K, mean $620K, last updated Q3 2025"
 
from kaizen.memory import ShortTermMemory, LongTermMemory, EntityMemory

# Short-term: conversation buffer
short_mem = ShortTermMemory(max_turns=20)

# Long-term: persistent across sessions
long_mem = LongTermMemory(
    backend="sqlite",
    db_path="agent_memory.db"
)

# Entity: structured knowledge store
entity_mem = EntityMemory(
    backend="sqlite",
    db_path="entity_memory.db"
)

# Attach to agent
agent = ReActAgent(
    signature=DataAnalysis,
    tools=[explore_data, train_model],
    memory=[short_mem, long_mem, entity_mem]
)

# After running, entity memory stores structured facts
entity_mem.get("hdb_resale_dataset")
# → {"rows": 50000, "columns": 11, "last_profiled": "2025-11-15",
#    "target": "resale_price", "best_model": "gradient_boosting"}
Theory

Short-term memory is the agent's "working memory" — analogous to a human holding information in mind while solving a problem. Long-term memory is "experience" — lessons learned that persist across sessions. Entity memory is "knowledge" — structured facts about the world. Together, they give the agent a form of continuity that pure ReAct (which forgets everything after each run) lacks.

The Kailash Engine: MCP Server

Kailash Engine

MCP (Model Context Protocol) is an open standard for exposing tools to AI agents. kailash-mcp provides MCPServer — you decorate functions with @server.tool() and they become discoverable by any MCP client. Transport options: stdio (local, for CLI agents) and HTTP/SSE (remote, for web agents). This turns your ML pipeline into a service.

Building an MCP server

from kailash_mcp import MCPServer, tool

server = MCPServer("ml-tools", version="1.0.0")

@server.tool("explore")
def explore_data(dataset: str) -> dict:
    """Load and profile a dataset. Returns shape, columns, null counts."""
    from shared import MLFPDataLoader
    from kailash_ml import DataExplorer

    loader = MLFPDataLoader()
    df = loader.load("mlfp06", dataset)
    explorer = DataExplorer()
    profile = explorer.profile(df)
    return {
        "shape": list(df.shape),
        "columns": df.columns,
        "null_pct": profile.null_percentages,
        "dtypes": {c: str(t) for c, t in df.schema.items()}
    }

@server.tool("train")
def train_model(dataset: str, target: str, algorithm: str = "gradient_boosting") -> dict:
    """Train a classification model. Returns accuracy, F1, and training time."""
    from shared import MLFPDataLoader
    from kailash_ml import TrainingPipeline

    loader = MLFPDataLoader()
    df = loader.load("mlfp06", dataset)
    pipeline = TrainingPipeline(algorithm=algorithm)
    result = pipeline.fit(df, target=target)
    return {
        "accuracy": round(result.metrics["accuracy"], 4),
        "f1_score": round(result.metrics["f1_score"], 4),
        "training_time_s": round(result.training_time, 2)
    }

@server.tool("feature_importance")
def get_feature_importance(dataset: str, target: str, top_n: int = 10) -> list:
    """Return top-N most important features for the target variable."""
    from kailash_ml import ModelVisualizer
    viz = ModelVisualizer()
    importance = viz.compute_importance(dataset, target)
    return importance[:top_n]

# Run the server
if __name__ == "__main__":
    server.run(transport="stdio")  # or transport="sse" for HTTP

MCP transport options

TransportProtocolUse case
stdioStandard input/outputLocal CLI agents, development, testing
sseHTTP + Server-Sent EventsRemote agents, web applications, production

The MCP server exposes a /tools endpoint that returns the schema of all registered tools. An MCP client (any agent framework that speaks MCP) calls this endpoint to discover available tools, then calls individual tools by name with JSON parameters. The agent never needs to know how the tools are implemented — it only sees the schema.

Worked Example: Multi-Agent ML Pipeline

Goal

Build a supervisor-worker pipeline for Singapore HDB resale price analysis. The DataScientist supervisor delegates to three workers: FeatureEngineer (profiles data, engineers features), ModelSelector (trains and compares models), and ReportWriter (produces the executive summary). All agents communicate via A2A messages and share an entity memory store.

Step 1. Define the three worker agents.

Each worker has a typed signature, 3-4 tools, and access to shared entity memory for cross-agent knowledge:

# Worker 3: Report Writer

    """Synthesise findings into executive report."""
    feature_report: str = InputField()

Step 2. Build the supervisor. 
from kaizen import BaseAgent, LLMCostTracker

class HDBAnalysisSupervisor(BaseAgent):
    def run(self, dataset: str) -> str:
        tracker = LLMCostTracker(budget_usd=3.00)

        # Phase 1: Feature engineering
        features = feature_agent.run(
            dataset=dataset, cost_tracker=tracker
        )

        # Phase 2: Model selection (uses engineered features)
        models = model_agent.run(
            dataset=dataset,
            features=features.engineered_columns,
            cost_tracker=tracker
        )

        # Phase 3: Report (combines both results)
        report = report_writer.run(
            feature_report=features.feature_report,
            model_comparison=models.model_comparison,
            cost_tracker=tracker
        )

        print(f"Total cost: ${tracker.total_cost:.4f}")
        return report.executive_summary

supervisor = HDBAnalysisSupervisor()
result = supervisor.run(dataset="hdb_resale.csv")

Security Considerations

Advanced

Multi-agent systems introduce security concerns that single agents do not have. A worker agent with access to customer PII should not leak that data to a report-writing agent that produces external-facing documents. Design principles: (1) each worker accesses only the data it needs, (2) A2A messages carry sanitised summaries, not raw data, (3) the supervisor enforces access boundaries. Lesson 6.7 formalises this with PACT operating envelopes.

Common Mistakes

Mistake 1: Too many agents

Start with 2-3 agents, add only when you have a clear reason. Each agent adds communication overhead (A2A messages, context management, budget splitting). A 2-agent system is easier to debug than a 6-agent system. The test: if you cannot explain what each agent uniquely contributes, merge it with another.

Mistake 2: Free-form text between agents

Agents communicating through unstructured text create a game of telephone — each agent paraphrases the previous one's output, losing precision. Use typed A2A messages with structured payloads. The receiving agent gets {"accuracy": 0.93}, not "the accuracy was about 93 percent".

Mistake 3: No shared memory

Without shared entity memory, workers repeat work. The profiler discovers the target column is "churned"; the trainer asks the LLM to figure out the target column again. Entity memory stores facts once and makes them available to all agents, eliminating redundant tool calls.

Mistake 4: Ignoring inter-agent security

In production, agents may have different access levels. A data-access agent can see customer PII; a reporting agent should see only aggregated statistics. Without access controls, the reporting agent might inadvertently include individual customer records in a public report.

Reflection

You have built multi-agent systems using all four orchestration patterns, configured agent memory for continuity across turns and sessions, and created an MCP server that exposes ML tools as a discoverable service. The key insight: multi-agent systems are organisational design, not just code. The same principles that make human teams effective — clear roles, structured communication, shared knowledge, appropriate access controls — make agent teams effective.

MCP is particularly significant. By exposing your tools as an MCP server, you decouple tool implementation from agent implementation. Any MCP client — Kaizen, Claude Code, or a third-party framework — can discover and use your ML tools without knowing how they are built. This is the service-oriented architecture pattern applied to AI agents.

In Lesson 6.7, we add the missing piece: governance. The multi-agent pipeline you built today has no access controls, no operating envelopes, and no audit trail. PACT adds all three — making the pipeline not just functional but compliant and auditable.
