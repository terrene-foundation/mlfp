This is the culmination of the six-module programme. You deploy a complete, governed AI system via Nexus — a platform accessible through API, CLI, and MCP simultaneously. The system integrates every major Kailash package: Core SDK for workflow orchestration, ML for model training, Kaizen for agents, PACT for governance, Nexus for deployment, and DriftMonitor for production monitoring. Why This Matters

Consider a Singapore bank building an internal AI system to predict HDB resale prices for mortgage risk assessment. The model must serve predictions to three audiences: relationship managers via a web dashboard (REST API), the risk automation team via a CLI batch pipeline, and an AI assistant via MCP tool calls. Each audience needs authentication. The system must log every prediction for regulatory audit. Model drift must be detected before stale predictions cause bad lending decisions.

Building this from scratch means writing an API server, a CLI parser, an MCP server, an auth layer, a rate limiter, a logging pipeline, and a drift detector — all as separate codebases that need to stay in sync. Or you can use Nexus, which gives you all three channels from a single codebase, with PACT governance and DriftMonitor bolted on. That is what you build today.

Core Concepts

Nexus: One Codebase, Three Interfaces

Nexus is Kailash's multi-channel deployment platform. You register a Python function (or a Kaizen agent) as a service, and Nexus automatically exposes it as a REST endpoint, a CLI command, and an MCP tool. You write the logic once; Nexus handles serialisation, transport, error formatting, and documentation generation for all three channels.

Foundations

Think of Nexus as a universal adapter. Your code speaks one language. Nexus translates it for three different audiences: web clients (API), terminal operators (CLI), and AI agents (MCP). Without Nexus, you would maintain three separate interface layers that inevitably drift apart.

Authentication and Authorisation (RBAC)

Authentication answers "who are you?" — typically via a JWT token. Authorisation answers "what are you allowed to do?" — via role-based access control (RBAC). Nexus supports both out of the box.

Roles cascade from PACT's D/T/R structure (Lesson 6.7). An admin can deploy models and change configurations. An analyst can run predictions but not modify the model. A viewer can see results but not trigger new predictions. Each API call, CLI invocation, and MCP tool call checks the caller's role before executing.

Middleware: Rate Limiting, Logging, CORS

Nexus middleware wraps every request with cross-cutting concerns:

  • Rate limiting prevents abuse. A sensible default for internal tools is 60 requests per minute per user. Public-facing APIs may need stricter limits.
  • Logging captures every request and response for audit. Combined with PACT governance logs, this creates a complete audit trail.
  • CORS (Cross-Origin Resource Sharing) controls which web domains can call the API. For internal dashboards, whitelist your domain; never use * in production.

Production Monitoring with DriftMonitor

A model trained on January data may produce poor predictions by July if the underlying distribution shifts. DriftMonitor (from Lesson 3.8) watches for three kinds of drift:

Drift typeWhat shiftsDetection methodExample
Feature driftInput distributionsKL divergence, PSIMedian flat size increases from 90 to 110 sqm
Prediction driftOutput distributionKS test on predictionsMean predicted price drifts upward 15%
Data qualityMissing/invalid valuesSchema validationNew API version returns null for a required field
Theory

Population Stability Index (PSI) quantifies how much a distribution has shifted. PSI below 0.1 indicates no meaningful change. PSI between 0.1 and 0.25 suggests moderate drift worth investigating. PSI above 0.25 signals significant drift — retrain the model. The formula compares the expected proportion $e_i$ (training data) against the actual proportion $a_i$ (production data) across $k$ bins:

$$ \text{PSI} = \sum_{i=1}^{k} (a_i - e_i) \cdot \ln\!\left(\frac{a_i}{e_i}\right) $$
$a_i$ — actual proportion in bin $i$ $e_i$ — expected proportion in bin $i$ $k$ — number of bins (10-20)

Mathematical Foundations: System Reliability

A production system composed of $n$ sequential components, each with independent availability $p_i$, has overall availability:

$$ A_{\text{system}} = \prod_{i=1}^{n} p_i $$
$p_i$ — availability of component $i$ $n$ — number of components in series

If each of 6 components (training pipeline, data store, agent, governance, Nexus, drift monitor) has 99.5% availability, the system availability is $0.995^6 \approx 0.970$ — only 97%. This is why production systems need health checks, retries, and graceful degradation. Nexus's middleware provides the first layer of defence; DriftMonitor provides the continuous validation.

Advanced

For higher availability, components can be deployed with redundancy. If a component has two independent replicas, each with availability $p$, the redundant availability is $1 - (1 - p)^2$. Two replicas at $p = 0.995$ yield $1 - 0.005^2 = 0.999975$ per component. Kubernetes or cloud auto-scaling implements this transparently for Nexus services.

The Kailash Engine: Nexus

Kailash Engine

Nexus is the deployment layer of the Kailash platform. It takes any Python callable or Kaizen agent and exposes it via REST API, CLI, and MCP simultaneously. Key methods: Nexus(name) to create a platform, add_auth(provider, roles) to configure RBAC, add_middleware(type, **config) for cross-cutting concerns, register_service(name, handler) to add endpoints, and serve(port) to launch all three channels.

Worked Example: Deploying a Governed HDB Price Prediction Platform

Goal

Deploy the governed multi-agent system from Lessons 6.5-6.7 via Nexus. The system predicts HDB resale prices. Three channels (API, CLI, MCP), RBAC authentication, governance checks, and drift monitoring. Singapore- specific: the model was trained on HDB transaction data and serves predictions for flat types across estates like Tampines, Jurong East, and Ang Mo Kio.

Step 1 — Build the platform.

Create a Nexus application, configure auth and middleware.

from nexus import Nexus
from pact import GovernanceEngine, load_org_yaml
from kaizen_agents import GovernedSupervisor
from kailash_ml import TrainingPipeline, DriftMonitor, ModelRegistry

# Create platform
app = Nexus("hdb-price-platform")

# Configure RBAC: three roles
app.add_auth(provider="jwt", roles=["admin", "analyst", "viewer"])

# Add middleware
app.add_middleware("rate_limit", requests_per_minute=60)
app.add_middleware("logging", level="INFO", format="json")
app.add_middleware("cors", origins=["https://dashboard.internal.sg"])
Step 2 — Load the governed supervisor.

Retrieve the governed supervisor from Lesson 6.7. The GovernedSupervisor drives a Kaizen multi-agent pipeline (DataScientist, FeatureEngineer, ModelSelector, ReportWriter) with PACT envelopes attached to every role.

Step 3 — Register services and deploy.

Register the governed agent as a Nexus service. Add a health check and a drift monitoring endpoint. 

# Service wrapper binds the executor to the supervisor's run loop
async def predict_service(payload: dict) -> dict:
    result = await governed_supervisor.run(
        objective=f"Predict price for {payload['flat_type']} in {payload['town']}",
        execute_node=executor,
        context=payload,
    )
    return {"price": result.output["price"],
            "budget_consumed": result.budget_consumed,
            "audit_chain_valid": governed_supervisor.audit.verify_chain()}

# Prediction service — the core endpoint
app.register_service("predict", predict_service,
    roles=["admin", "analyst"])

# Results view — read-only

    roles=["admin", "analyst", "viewer"])

# Deploy endpoint — admin only

    roles=["admin"])

# Health check — no auth required
app.register_service("health", lambda: {"status": "ok"},
    public=True)

# Launch: API on :8000, CLI available, MCP on stdio
app.serve(port=8000)

Attach DriftMonitor to the deployed model. It compares incoming feature distributions against the training data baseline.

Debugging Agent Reasoning Chains

When a governed agent returns an unexpected result, you need to trace the reasoning chain. Kaizen agents emit structured trace events that show every thought, action, observation, and governance check: 

# GovernedSupervisor captures every step in its audit trail
result = await governed_supervisor.run(
    objective="Predict HDB price for Ang Mo Kio 3-room",
    execute_node=executor,
    context={"town": "Ang Mo Kio", "flat_type": "3-room"},
)

# Inspect the reasoning chain via the audit trail
for entry in governed_supervisor.audit.to_list():
    print(f"[{entry['phase']}] {entry['action']}")
    print(f"  Verdict: {entry['verdict']} — {entry['reason']}")
    if entry.get("tool_name"):
        print(f"  Tool: {entry['tool_name']}")

# Verify the chain has not been tampered with
assert governed_supervisor.audit.verify_chain()
Common pitfall

Trace output can be verbose. Filter by step type (thought, action, observation, governance) to focus on the relevant part of the chain. In production, traces should be stored in a structured log (not printed to stdout) for later analysis.

Testing Agents: Automated Validation

Agent testing is fundamentally different from function testing. A function returns the same output for the same input. An agent may take different reasoning paths each time. Testing agents requires behavioural assertions — checking that the agent's actions and final answer are reasonable, not that it followed a specific sequence. 

def test_agent_uses_correct_tools():
    """Verify agent calls DataExplorer before TrainingPipeline."""
    result = agent.run("Analyse and predict HDB prices", trace=True)
    tool_names = [s.tool_name for s in result.trace if s.type == "action"]
    assert "explore_data" in tool_names, "Agent must explore data first"
    assert tool_names.index("explore_data") < tool_names.index("train_model")

async def test_agent_respects_governance():
    """Verify the supervisor stops when its budget is exhausted."""
    from kaizen_agents import GovernedSupervisor
    tiny = GovernedSupervisor(
        model="gpt-4o-mini",
        budget_usd=0.01,                 # tiny budget
        tools=["predict_price"],
        data_clearance="public",
    )
    async def expensive_executor(spec, inputs):
        return {"result": "...", "cost": 0.02,
                "prompt_tokens": 100, "completion_tokens": 50}
    result = await tiny.run(
        objective="Complex multi-step analysis",
        execute_node=expensive_executor,
    )
    assert not result.success
    assert result.budget_consumed <= 0.01 or "budget" in result.failure_reason.lower()

def test_prediction_within_range():
    """Verify prediction is within plausible range for Singapore HDB."""
    result = agent.run("Predict 4-room Tampines price")
    price = result.output["price"]
    assert 200_000 <= price <= 1_200_000, f"Price {price} outside HDB range"

Common Mistakes

Mistake 1: Skipping auth in development

"I'll add auth later" is how production systems ship without auth. Configure RBAC from day one, even in development. Use test tokens with different roles. The three-line Nexus auth configuration has zero excuse for deferring.

Mistake 2: No drift monitoring

A model deployed without monitoring is a time bomb. Singapore's HDB market shifts with government cooling measures, interest rate changes, and demographic trends. A model trained on 2023 data will degrade through 2024 unless monitored. DriftMonitor is not optional — it is the feedback loop that keeps the model honest.

Mistake 3: Testing only the happy path

Your API returns 200 for valid requests — great. But does it return 401 for missing auth? 403 for wrong role? 429 for rate limit breaches? 422 for malformed input? Test every error path. The error paths are what keep a production system safe.

Mistake 4: Forgetting the CLI and MCP channels

Nexus gives you three channels from one codebase. Test all three. The CLI is how batch jobs will consume the service. MCP is how AI agents will consume it. If you only test the REST API, you are testing one third of your deployment surface.
