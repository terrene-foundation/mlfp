Lesson 1.5 Window Functions and Trends

Group-by collapses many rows into one. A window function computes a value for each row using a sliding window of nearby rows — no collapse. Today you learn rolling averages, year-over-year changes, and per-group windows with over(). You will also meet lazy frames, Polars' performance-optimisation mode, and see the single pattern that powers every time-series analysis in Module 2.

Why This Matters

HDB prices bounce around month to month. One Bishan transaction at 720k followed by another at 690k does not mean the market dropped — it means the second flat happened to be smaller, or lower-floor, or had less remaining lease. To see the trend, you smooth the noise with a rolling average. To see the annual rhythm, you compare each month to the same month a year earlier. To see which districts are rising fastest, you rank them by their recent growth rate.

Every one of those operations needs a window — a set of nearby rows to compute against. Group-by is not enough because group-by collapses: you lose the per-month detail. Windows are the answer: keep the rows but enrich each one with context from its neighbours.

Core Concepts

Windows without collapsing

The key difference between group_by / agg and a window function is the output shape. Group-by produces one row per group; a window produces one row per input row, enriched with the aggregation.

The .over(...) modifier

Polars expresses a window with .over("partition_column"). The expression runs independently inside each partition, and the result is broadcast back to every row in that partition. 

monthly_prices = monthly_prices.with_columns(
    pl.col("median_price_sqm")
    .rolling_mean(window_size=12)
    .over("town")
    .alias("rolling_12m_price_sqm"),
)

Read it: "Take the median_price_sqm column. Compute a 12-row rolling mean. Do it independently for each town (so the window never crosses town boundaries). Name the result rolling_12m_price_sqm." The output has the same row count as the input; every row now carries its town's 12-month trend at that point in time.

Rolling aggregations

rolling_mean is the workhorse. Given a window of size $w$ and a row at position $i$, it computes the mean of rows $i-w+1, \ldots, i$ — the last $w$ rows inclusive.

$$\text{rolling\_mean}_i = \frac{1}{w} \sum_{j=i-w+1}^{i} x_j$$
$w$ — window size $x_j$ — value at row $j$ last $w$ rows, inclusive

The first $w - 1$ rows do not have enough history, so Polars fills them with null. That is correct behaviour — there was no prior data to average. Do not try to fix it with zeros; the nulls are telling you "insufficient history here".

Polars has several rolling variants — rolling_sum, rolling_min, rolling_max, rolling_std — all of which follow the same shape.

shift and year-over-year

shift(n) moves every value $n$ positions forward, filling the first $n$ rows with null. Combined with over("town"), each town shifts independently. 

monthly_prices = monthly_prices.with_columns(
    pl.col("median_price_sqm").shift(12).over("town").alias("price_12m_ago"),
)

monthly_prices = monthly_prices.with_columns(
    (
        (pl.col("median_price_sqm") - pl.col("price_12m_ago"))
        / pl.col("price_12m_ago") * 100
    ).alias("yoy_change_pct"),
)

The YoY formula is the standard percent change:

$$\text{YoY}\% = \frac{x_t - x_{t-12}}{x_{t-12}} \times 100\%$$
$x_t$ — price now $x_{t-12}$ — price 12 months ago positive = appreciation
🔵 Theory: Why YoY and not MoM?

Month-over-month change is dominated by seasonality. Singapore HDB transactions peak in certain months (post-Chinese New Year, pre-year-end). A MoM change mixes genuine appreciation with calendar effects. A year-over-year change compares the same calendar month, so seasonal effects cancel and you see the underlying market movement.

Lazy frames (performance preview)

Every chain you have written so far runs eagerly: each filter, with_columns, or group_by executes immediately and hands the result to the next step. For large data, this can waste work — Polars can often skip or reorder steps if it is allowed to plan the whole chain before running anything. A lazy frame gives Polars that permission. 

result = (
    pl.scan_parquet("hdb_resale.parquet")        # lazy: no data loaded yet
    .filter(pl.col("year") >= 2020)
    .group_by("town")
    .agg(pl.col("resale_price").median().alias("median"))
    .sort("median", descending=True)
    .collect()                                    # NOW execute the whole plan
)

scan_parquet returns a lazy frame — a plan, not data. The chain records your intentions without doing any work. When you call collect(), Polars looks at the whole plan, optimises it (pushes the filter into the scan, fuses operations, parallelises where possible), and executes it in one shot. For big data, the speedup is dramatic — often 5× or more.

🟢 When to use lazy frames

Use lazy frames when (a) the data is too big to fit in memory, (b) the pipeline is long enough that optimisation matters, or (c) you are producing the same pipeline in production and want maximum throughput. For interactive exploration on small data, eager frames are fine — you do not need the optimisation and the shorter feedback loop is nicer.

Worked Example: Ranking Singapore's Hottest Districts

📝 Goal

Compute monthly median price per square metre for every HDB town, add a 12-month rolling mean and year-over-year change, and rank the top 5 fastest-growing districts over the last 12 months.

Step 1.

Load, parse dates, and build the monthly series.

import polars as pl
from shared import MLFPDataLoader

loader = MLFPDataLoader()
hdb = loader.load("mlfp01", "hdb_resale.parquet").with_columns(
    pl.col("month").str.to_date("%Y-%m").alias("transaction_date"),
    (pl.col("resale_price") / pl.col("floor_area_sqm")).alias("price_per_sqm"),
)

monthly_prices = (
    hdb.group_by("town", "transaction_date")
    .agg(
        pl.col("price_per_sqm").median().alias("median_price_sqm"),
        pl.len().alias("transaction_count"),
    )
    .sort("town", "transaction_date")
)

Note the sort — rolling windows depend on row order. A window on unsorted data is nonsense.

Step 2.

Add a 12-month rolling mean using over("town"). Each town gets its own independent window.

monthly_prices = monthly_prices.with_columns(
    pl.col("median_price_sqm")
    .rolling_mean(window_size=12)
    .over("town")
    .alias("rolling_12m"),
)

print(monthly_prices.filter(pl.col("town") == "BISHAN").tail(15))
Step 3.

Compute year-over-year change with shift(12).over("town").

monthly_prices = monthly_prices.with_columns(
    pl.col("median_price_sqm").shift(12).over("town").alias("price_12m_ago"),
).with_columns(
    (
        (pl.col("median_price_sqm") - pl.col("price_12m_ago"))
        / pl.col("price_12m_ago") * 100
    ).alias("yoy_change_pct"),
)

The first 12 rows of every town will have null YoY — no 12-month history. That is expected.

Step 4.

Rank towns by their most recent YoY change.

latest_date = monthly_prices["transaction_date"].max()

latest_rankings = (
    monthly_prices
    .filter(pl.col("transaction_date") == latest_date)
    .filter(pl.col("yoy_change_pct").is_not_null())
    .select("town", "median_price_sqm", "rolling_12m", "yoy_change_pct")
    .sort("yoy_change_pct", descending=True)
)

print("=== TOP 5 FASTEST GROWING ===")
print(latest_rankings.head(5))
print("\n=== BOTTOM 5 ===")
print(latest_rankings.tail(5))
Step 5.

Rewrite as a lazy pipeline for production speed.

rankings = (
    pl.scan_parquet(loader.path("mlfp01", "hdb_resale.parquet"))
    .with_columns(
        pl.col("month").str.to_date("%Y-%m").alias("date"),
        (pl.col("resale_price") / pl.col("floor_area_sqm")).alias("psm"),
    )
    .group_by("town", "date")
    .agg(pl.col("psm").median().alias("median_psm"))
    .sort("town", "date")
    .with_columns(
        pl.col("median_psm").shift(12).over("town").alias("psm_12m_ago"),
    )
    .with_columns(
        ((pl.col("median_psm") - pl.col("psm_12m_ago"))
         / pl.col("psm_12m_ago") * 100).alias("yoy_pct"),
    )
    .collect()
)

The whole chain runs in one pass once collect() is called. Polars optimises the plan before executing — filter pushdown, predicate fusion, and parallel scan all happen behind the scenes.

Mathematical Foundations: Moving Averages

A simple moving average (SMA) of window size $w$ is the unweighted mean of the last $w$ observations. Each observation inside the window contributes $\frac{1}{w}$ to the average; observations outside contribute 0. SMA is a low-pass filter — it suppresses high-frequency noise while preserving low-frequency trend.

Larger windows, smoother curves

If the noise is independent across months with variance $\sigma^2$, the variance of the rolling-mean estimator is $\sigma^2 / w$. Doubling the window halves the variance of the smoothed curve, at the cost of lag: the rolling mean always trails a real trend by roughly $w/2$ months.

The responsiveness–smoothness trade-off

WindowNoise reductionLagUse case
3 monthsLow~1.5 moDetect turning points quickly
6 monthsModerate~3 moBalance of speed and smoothness
12 monthsStrong~6 moReveal underlying annual trend
24 monthsVery strong~12 moLong-term structural comparison

A classic pattern in technical analysis: plot both the short-window and long-window rolling means on the same chart. When the short one crosses above the long one ("golden cross"), it signals accelerating growth; when it crosses below ("death cross"), deceleration. The same idea applies to HDB prices, GDP, unemployment — any time-series trend.

Common Mistakes

⚠ Mistake 1: Forgetting to sort before rolling

A rolling window operates on rows in their current order. If the data is not sorted by date (and by partition if you are using over), the "rolling mean" becomes a random scramble of unrelated values. Always sort(partition, date) first.

⚠ Mistake 2: Rolling across groups without over

pl.col("price").rolling_mean(12) without .over("town") computes a window that spans every row — including rows from other towns. The first row of Jurong would be averaged with the last eleven rows of Bishan. Nonsense. Always use .over when you have partitions.

⚠ Mistake 3: Treating early nulls as errors

The first $w-1$ rows of a rolling mean are null because there is not enough history. That is correct, not broken. Do not fill with zeros — you would be injecting fake averages into the model. Filter them out before modelling if they bother you.

⚠ Mistake 4: Calling collect() too early

A lazy pipeline's advantage comes from planning the whole chain before executing. If you call collect() in the middle and then keep building, you have cut the plan in half — the optimiser cannot see the full shape and misses optimisations. Defer collect() to the end.

Reflection

Windows are the bridge from "collapse to one number per group" (group-by) to "enrich each row with context from its neighbours" (rolling means, lag features, YoY changes). Every time-series analysis in the rest of this course builds on the over() + rolling_* + shift() primitives you saw today.

The responsiveness–smoothness trade-off is not just about charts. Every feature you engineer for a forecasting model makes the same trade. Short windows catch turning points but add noise; long windows smooth noise but lag the real trend. The right answer depends on what you are forecasting — and that is a Module 3 question.
