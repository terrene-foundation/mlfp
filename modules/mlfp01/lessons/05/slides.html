Raw data is noisy — trends live in windows

Windows don't collapse — they enrich

The .over modifier


monthly_prices = monthly_prices.with_columns(
    pl.col("median_price_sqm")
      .rolling_mean(window_size=12)
      .over("town")
      .alias("rolling_12m"),
)

Read: "12-row rolling mean of price, independently per town, aliased as rolling_12m."

.over("town") partitions the window so it never crosses town boundaries.

How a rolling window slides

The bracket slides one row at a time. First window_size - 1 positions return null.

Rolling mean formula

\[ \text{rolling\_mean}_i = \frac{1}{w} \sum_{j=i-w+1}^{i} x_j \]

Unweighted mean of the last $w$ observations inclusive.

  • First $w-1$ rows → null (not enough history). That's correct.
  • Polars has rolling_sum, rolling_min, rolling_max, rolling_std — same shape.

Window size trade-off

WindowNoiseLagUse case
3 monthsLow reduction~1.5 moTurning points
6 monthsModerate~3 moBalanced
12 monthsStrong~6 moUnderlying trend
24 monthsVery strong~12 moStructural comparison
🔵 Theory: with independent noise of variance $\sigma^2$, rolling mean has variance $\sigma^2/w$. Doubling $w$ halves variance but adds lag of $\sim w/2$.

shift — lag features


monthly_prices = monthly_prices.with_columns(
    pl.col("median_price_sqm")
      .shift(12)
      .over("town")
      .alias("price_12m_ago"),
)

shift(12) moves every value 12 rows forward, filling the first 12 rows with null. Combined with over("town"), each town shifts independently.

Year-over-year change

\[ \text{YoY}\% = \frac{x_t - x_{t-12}}{x_{t-12}} \times 100\% \] 

monthly_prices = monthly_prices.with_columns(
    (
        (pl.col("median_price_sqm") - pl.col("price_12m_ago"))
        / pl.col("price_12m_ago") * 100
    ).alias("yoy_change_pct"),
)

Why YoY not MoM? Cancels out seasonality — compares the same calendar month to itself a year earlier.

Ranking the hottest districts


latest = monthly_prices["transaction_date"].max()

rankings = (
    monthly_prices
    .filter(pl.col("transaction_date") == latest)
    .filter(pl.col("yoy_change_pct").is_not_null())
    .select("town", "median_price_sqm", "yoy_change_pct")
    .sort("yoy_change_pct", descending=True)
)
print(rankings.head(5))  # top 5 growers
print(rankings.tail(5))  # bottom 5

🟣 Lazy frames — global optimisation

Lazy pipeline in code


result = (
    pl.scan_parquet("hdb_resale.parquet")  # lazy: no data loaded
    .filter(pl.col("year") >= 2020)
    .group_by("town")
    .agg(pl.col("resale_price").median().alias("median"))
    .sort("median", descending=True)
    .collect()                              # execute the whole plan
)

scan_* = lazy. read_* = eager. collect() executes. Polars optimises the whole chain before running anything.

Four traps

1. Not sorting before rolling. Windows depend on row order.
2. Rolling across groups without over. Windows will span unrelated partitions.
3. Filling early nulls with zero. Nulls mean "insufficient history" — that's correct.
4. Calling collect() too early. Breaks the optimiser's view.

What you can now do

  • ✓ Compute rolling means with rolling_mean(window_size=w)

  • ✓ Build lag features with shift(n)

  • ✓ Choose window size for the responsiveness/smoothness trade-off
  • ✓ Convert a pipeline to lazy with scan_* + collect
