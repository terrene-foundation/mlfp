This is the capstone of Module 1. Why This Matters

In a real data role, nobody hands you a clean CSV. Data arrives from APIs, databases, partner feeds, and spreadsheets — each with its own schema, its own null convention, its own date format. Your job is to build a pipeline that takes any version of the raw data and produces the same clean output every time. That pipeline is not a one-off script — it is a repeatable process you can run on tomorrow's data with zero changes.

Null Handling

Nulls are the most common data quality issue. Polars uses null (not NaN) to represent missing values. The three operations you need:

MethodWhat it doesWhen to use
is_null()Boolean mask: True where nullInspecting which rows are missing
fill_null(value)Replace nulls with a valueImputation: median, mean, forward-fill
drop_nulls()Remove rows with any nullWhen missing rows are few and uninformative
🟢 Foundations

Never drop nulls without checking how many there are. If 30% of rows have nulls, dropping them loses a third of your data. Impute first; drop only as a last resort.

Imputation strategies

import polars as pl

# Forward-fill: carry the last known value forward (time-series)
df = df.with_columns(pl.col("fare").forward_fill())

# Median imputation: robust to outliers
median_fare = df["fare"].median()
df = df.with_columns(pl.col("fare").fill_null(median_fare))

# Group-aware imputation: fill with group median
df = df.with_columns(
    pl.col("fare").fill_null(pl.col("fare").median().over("zone"))
)
🔵 Theory

Imputation introduces bias. Forward-fill assumes temporal continuity. Median imputation compresses the distribution toward the centre. Group-aware imputation is less biased but requires a meaningful grouping variable. In Module 2 you will learn when each strategy is statistically defensible.

ETL: Extract, Transform, Load

ETL is the universal pattern for data pipelines:

  • Extract: get the raw data from its source — files, APIs, databases.
  • Transform: clean, encode, scale, engineer features.
  • Load: write the result to a destination — parquet file, database, dashboard.

REST APIs: Extracting Data from the Web

Many datasets are not files — they are live services you query over HTTP. Singapore's OneMap API returns geolocation data for any address. The pattern is always the same: 

import httpx

# GET request with query parameters
response = httpx.get(
    "https://www.onemap.gov.sg/api/common/elastic/search",
    params={"searchVal": "Tampines", "returnGeom": "Y", "getAddrDetails": "Y"}
)

data = response.json()
print(f"Found {data['found']} results")
print(f"First result: {data['results'][0]['SEARCHVAL']}")
🟢 Foundations

A REST API is a URL you send a request to and get structured data back (usually JSON). GET means "give me data". POST means "here is data, do something with it". For this course, you will mostly use GET.

PreprocessingPipeline

⚙ Kailash Engine

PreprocessingPipeline automates the transform step: it detects column types, encodes categoricals, scales numerics, and imputes missing values. Like DataExplorer, you use it as a consumer — create, configure, run. 

from kailash_ml import PreprocessingPipeline

pipeline = PreprocessingPipeline(
    numeric_strategy="median",       # impute with median
    categorical_strategy="onehot",   # one-hot encode
    scale=True,                      # standard scaling
)

df_clean = pipeline.fit_transform(df_raw)
print(f"Before: {df_raw.shape} → After: {df_clean.shape}")
🔵 Theory

fit_transform() fits the pipeline on the data (learns medians, category sets, scaling parameters) and transforms it. When you later process test data, use transform() only — the pipeline applies the same parameters it learned from training data. This prevents data leakage: the test set never influences the preprocessing decisions.

Worked Example: Taxi Trip Pipeline

📝 Goal

Build a complete pipeline for Singapore taxi trip data: extract from file, profile, clean, visualise peak demand patterns, and generate a summary report.

Step 1 — Extract. 
from shared import MLFPDataLoader
from kailash_ml import DataExplorer, PreprocessingPipeline
from shared import run_profile
import polars as pl

loader = MLFPDataLoader()
df_raw = loader.load("mlfp01", "sg_taxi_trips.csv")
print(f"Raw: {df_raw.shape}")
Step 2 — Profile. 
explorer = DataExplorer()
profile = run_profile(explorer, df_raw)
print(f"Alerts: {len(profile.alerts)}")
for a in profile.alerts:
    print(f"  {a.type}: {a.column} — {a.message}")
Step 3 — Transform: nulls and types. 
df = (
    df_raw
    .with_columns(
        pl.col("pickup_datetime").str.to_datetime("%Y-%m-%d %H:%M:%S"),
        pl.col("fare_amount").cast(pl.Float64),
    )
    .filter(pl.col("fare_amount") > 0)
    .filter(pl.col("trip_distance_km") > 0)
    .with_columns(
        pl.col("fare_amount").fill_null(pl.col("fare_amount").median()),
        hour=pl.col("pickup_datetime").dt.hour(),
        day_of_week=pl.col("pickup_datetime").dt.weekday(),
    )
    .drop_nulls(subset=["pickup_zone", "dropoff_zone"])
)
print(f"After cleaning: {df.shape}")
Step 4 — Transform: PreprocessingPipeline. 
pipeline = PreprocessingPipeline(
    numeric_strategy="median",
    categorical_strategy="onehot",
    scale=True,
)
df_model_ready = pipeline.fit_transform(
    df.select(["fare_amount", "trip_distance_km", "hour",
               "day_of_week", "pickup_zone"])
)
print(f"Model-ready: {df_model_ready.shape}")
Step 5 — Visualise and report. 
hourly = (
    df.group_by("hour").agg(pl.count().alias("trips"))
    .sort("hour")
)
# Peak demand: morning rush and evening rush
print(hourly)

# Re-profile the cleaned data
profile_clean = run_profile(explorer, df)
comparison = run_profile(explorer, df_raw, df_compare=df)
print(f"Alerts before: {len(profile.alerts)}")
print(f"Alerts after:  {len(profile_clean.alerts)}")

Project Structure

As pipelines grow, putting everything in one file becomes unmanageable. Structure your project into modules: 

taxi_pipeline/
├── extract.py          # Data loading functions
├── transform.py        # Cleaning and feature engineering
├── visualise.py        # Chart generation
├── report.py           # HTML report generation
└── main.py             # Orchestrates the pipeline
# main.py — the orchestrator
from extract import load_taxi_data
from transform import clean_taxi_data, preprocess
from visualise import create_demand_charts
from report import generate_html_report

df_raw = load_taxi_data()
df_clean = clean_taxi_data(df_raw)
df_model = preprocess(df_clean)
charts = create_demand_charts(df_clean)
generate_html_report(df_model, charts, output="report.html")
🟣 Advanced

This is the simplest form of workflow orchestration — calling functions in sequence. In Module 3 you will use WorkflowBuilder to build the same pipeline as a node graph with automatic error handling, retry logic, and parallel execution.

Common Mistakes

⚠ Mistake 1: Dropping nulls without measuring the impact

Always check df.null_count() before dropping. If 25% of rows have nulls, you are throwing away a quarter of your data. Impute first.

⚠ Mistake 2: Fitting the pipeline on test data

fit_transform() on training data, transform() on test data. Never fit on both — the test set must remain unseen.

⚠ Mistake 3: Hardcoding file paths

Use MLFPDataLoader or environment variables. A pipeline that only works on your laptop is not a pipeline.

⚠ Mistake 4: No re-profiling after cleaning

Always re-profile. The cleaning step might introduce new issues (e.g., imputation changing the distribution shape). Compare before and after with DataExplorer.compare().
