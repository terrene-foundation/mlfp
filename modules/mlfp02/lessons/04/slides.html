A/B Testing and Experiment Design

Randomisation · Power analysis · SRM · Data collection framework

⏱ 45 minutes · Prereq: 2.3 (hypothesis testing)

Why randomisation works

Random assignment distributes all confounders -- measured and unmeasured -- roughly equally across groups.

Pre-register before you touch the data

  1. Unit of randomisation (user, session, device)
  2. Primary metric (one) + secondary (up to 3) + guardrails
  3. Direction of alternative (increase / decrease / two-sided)
  4. Decision rule: $\alpha$, power target, MDE
  5. Stopping criterion
⚠ Peeking inflates the false-positive rate. Decide everything upfront; commit in writing (or in ExperimentTracker).

Power analysis — sizing

Rearrange the power formula for $n$ per arm:

\[ n = \left(\frac{z_{1-\alpha/2} + z_{1-\beta}}{\delta/\sigma}\right)^2 \cdot 2 \]

For a proportion test with baseline $p_0$ and lift $\Delta$:

\[ n = \frac{2(z_{1-\alpha/2} + z_{1-\beta})^2 \cdot p_0(1 - p_0)}{\Delta^2} \]

Detect 1pp lift from 8% baseline, 80% power, $\alpha = 0.05$ → $n \approx 5{,}800$ per arm.

Power curves

Power scales with $\delta^2$: halve the effect → quadruple the sample size.

SRM — the first check you run

Before looking at outcomes, check the allocation.

\[ \chi^2 = \sum_k \frac{(O_k - E_k)^2}{E_k} \] 

n_c, n_t = 5_212, 4_788   # 50/50 intended
expected = [(n_c + n_t)/2]*2
chi2, p = stats.chisquare([n_c, n_t], f_exp=expected)
Use $p < 0.0001$ — not $0.05$. SRM is a data-quality alarm, not a hypothesis.

Common data pitfalls

  • Silos: metrics spread across systems, never joined
  • Red tape: three sign-offs before you can touch data
  • Aggregated only: daily totals, no user-level
  • Selection bias: only completers logged
  • Temporal leakage: experiment window overlaps marketing push

Worked: BOGO sample size


import numpy as np
from scipy import stats

p0, delta = 0.08, 0.01
alpha, power = 0.05, 0.80

z_a = stats.norm.ppf(1 - alpha/2)
z_b = stats.norm.ppf(power)
n = int(np.ceil(2 * (z_a + z_b)**2 * p0*(1-p0) / delta**2))
# n ≈ 5,800

Worked: load and SRM check


df = MLFPDataLoader().load("mlfp02", "ab_test.parquet")
counts = df.group_by("arm").agg(pl.len().alias("n"))
n_c = counts.filter(pl.col("arm")=="control")["n"][0]
n_t = counts.filter(pl.col("arm")=="treatment")["n"][0]
expected = [(n_c + n_t)/2]*2
chi2, p_srm = stats.chisquare([n_c, n_t], f_exp=expected)
assert p_srm > 1e-4, "SRM detected!"

from statsmodels.stats.proportion import (

k_c = df.filter(pl.col("arm")=="control")["converted"].sum()
k_t = df.filter(pl.col("arm")=="treatment")["converted"].sum()

z, p = proportions_ztest([k_t, k_c], [n_t, n_c])

Reporting to the business

Recommendation: Ship BOGO. Conversion rate increased by 1.3 percentage points (95% CI: 0.6 to 2.0, p = 0.003), exceeding the pre-registered MDE of 1pp. Approximately $240K additional revenue/quarter. No SRM. No guardrail metrics degraded.

Report in plain language. Quote uncertainty, not just point estimates. Connect to business value. Never say "we prove".

Three rookie mistakes

1. Peeking. Monitoring p-values and stopping when significant. Inflates false-positive rate massively.
2. No power analysis. Underpowered experiments waste time or worse, ship wrong.
3. Skipping SRM. If allocation is broken, nothing else matters.
