The End-to-End Statistical Pipeline

FeatureEngineer

⚙ Kailash Engine

FeatureEngineer automates feature creation: temporal features (lag, rolling, day-of-week), interaction terms, polynomial expansions, and domain-specific transformations. It does not replace domain knowledge — it amplifies it. 

from kailash_ml import FeatureEngineer

engineer = FeatureEngineer()

# Temporal features
df_features = engineer.create_temporal_features(
    df, date_col="date",
    features=["lag_1", "lag_3", "rolling_mean_6", "month", "quarter"]
)

# Interaction terms
df_features = engineer.create_interactions(
    df_features,
    pairs=[("floor_area", "age"), ("town_premium", "floor_level")]
)

print(f"Original: {df.shape[1]} cols → Engineered: {df_features.shape[1]} cols")

FeatureStore

⚙ Kailash Engine

FeatureStore manages features with point-in-time correctness: when you ask for features as of January 2024, you get exactly what was available in January 2024 — not features computed from future data. This prevents data leakage in time-series problems. 

from kailash_ml import FeatureStore

store = FeatureStore()

# Register features with timestamps
store.register(
    name="hdb_features",
    df=df_features,
    timestamp_col="date",
    entity_col="town",
)

# Retrieve point-in-time correct features
features_jan = store.get_features(
    name="hdb_features",
    as_of="2024-01-01",
    entities=["TAMPINES", "BISHAN"]
)
🔵 Theory

Point-in-time correctness is critical for any time-series model. If your training data for January includes features computed from February data, your model learns from the future — and performs worse in production where the future is unknown. FeatureStore enforces this automatically.

Common Mistakes

⚠ Mistake 1: Feature leakage from the future

If you compute a 6-month rolling mean and include future months in the window, your model sees the future. Use FeatureStore's point-in-time retrieval to prevent this.

⚠ Mistake 2: Engineering features without domain rationale

Creating 100 random interactions and selecting the ones that correlate with the target is p-hacking with extra steps. Every feature should have a domain reason.

⚠ Mistake 3: Reporting p-values without effect sizes

"Alcohol is significant ($p < 0.001$)" tells the reader nothing useful. "Each additional percentage point of alcohol increases quality by 0.3 points ($p < 0.001$)" tells them what to do.
