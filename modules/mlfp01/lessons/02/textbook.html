Lesson 1.2 Filtering and Transforming Data

Twelve rows of weather is the toy problem. Today we jump to 500,000 rows of real Singapore HDB resale transactions and learn the three operations that take a dataset from "everything" to "exactly the rows and columns I need to answer my question": filter, select, and with_columns. By the end you will chain them together into a single readable pipeline — the Polars idiom that the rest of the course assumes.

Why This Matters

Every analysis you will ever do starts with too much data. Singapore's Housing & Development Board publishes every public-housing resale transaction since 1990 — about 900,000 rows spanning 26 towns, seven flat types, four decades. A question like "what is the median price of a 4-room flat in Bishan last year?" touches fewer than 300 of those rows. Your first job is to throw away the 899,700 you do not need.

Beginners throw data away badly. They copy the file into Excel, drag-filter columns, paste values elsewhere, lose track of what they did, and end up with spreadsheets nobody can reproduce. Polars makes the same work declarative: you write an expression describing which rows you want, and the engine executes it. The expression is code — it can be read, reviewed, versioned, and re-run.

These three operations — filter, select, with_columns — are the vocabulary of the entire Polars library. Every later lesson in this module (joins, windows, visualisation, profiling, pipelines) builds on them. Once you can chain them confidently, you can express ninety percent of the data work real professionals do.

Core Concepts

Boolean expressions

A boolean is a value that is either True or False. Python has six comparison operators that produce booleans:

OperatorMeaningExampleResult
==equal to"BISHAN" == "BISHAN"True
!=not equal to"BISHAN" != "JURONG"True
>greater than500_000 > 350_000True
<less than100 < 50False
>=greater or equal500_000 >= 500_000True
<=less or equal26.5 <= 28.0True

Underscores inside numbers (500_000) are a Python syntax convenience: they are ignored by the interpreter but let you separate thousands for readability. 500_000 and 500000 are the same number.

🟢 Foundations

Note that = (one equals sign) is assignment — "set this variable". == (two equals signs) is comparison — "are these two equal?". This is the single most common error new Python programmers make. Python will refuse to run if price = 500_000 which is the language protecting you.

Polars column expressions

In Polars, a column expression is a recipe that describes what to do with a column — without actually doing it yet. The building block is pl.col("column_name"), which means "the column with this name". You can combine expressions with the same operators you use on regular Python values: 

pl.col("resale_price") > 500_000
# → an expression: "the resale_price column, compared to 500,000"

pl.col("town") == "BISHAN"
# → an expression: "the town column, compared to the string BISHAN"

pl.col("floor_area_sqm") * pl.col("price_per_sqm")
# → an expression: "floor area times price per sqm"

On its own, an expression does nothing. It becomes useful when you hand it to a method like filter, with_columns, or select, which tells Polars "apply this expression across the whole DataFrame".

🔵 Theory

Separating the description of work from its execution is what lets Polars optimise your pipeline. When you write df.filter(A).filter(B).select(C), Polars can reorder the filters, push them down below select, or fuse them into a single scan — none of which would be safe if filters mutated the DataFrame in place. You will see this again in Lesson 1.5 when we introduce lazy frames, where the separation becomes explicit.

Combining conditions with & and |

Real filters usually have more than one condition. Polars uses the bitwise operators & (and) and | (or) to combine expressions, and you must wrap each condition in parentheses: 

(pl.col("town") == "BISHAN") & (pl.col("flat_type") == "4 ROOM")
# → True only if BOTH conditions hold

(pl.col("town") == "BISHAN") | (pl.col("town") == "TOA PAYOH")
# → True if EITHER condition holds
⚠ The parentheses are not optional

Python's & has higher precedence than ==, so pl.col("town") == "BISHAN" & pl.col("flat_type") == "4 ROOM" is parsed as pl.col("town") == ("BISHAN" & pl.col("flat_type")) == "4 ROOM" which is gibberish. Parenthesise every condition and the problem disappears.

For sets of acceptable values, .is_in([...]) is cleaner than chaining a dozen OR conditions: 

central = ["BISHAN", "TOA PAYOH", "QUEENSTOWN", "BUKIT MERAH"]
df.filter(pl.col("town").is_in(central))

Method chaining

Polars methods almost always return a new DataFrame, which lets you chain them like a conversation: take this, then filter it, then select these columns, then sort it. The fluent style reads top to bottom like a recipe: 

result = (
    hdb
    .filter(pl.col("year") >= 2020)
    .filter(pl.col("price_tier").is_in(["premium", "luxury"]))
    .select("transaction_date", "town", "flat_type", "resale_price")
    .sort("resale_price", descending=True)
)

The outer parentheses let you put each step on its own line. Every line is a separate method call on the DataFrame produced by the previous line. When you read a chain later, you can delete any step and still get a valid pipeline — which makes debugging a matter of bisection.

with_columns: deriving new features

with_columns adds new columns (or replaces existing ones) without removing any others. It is how you engineer features — new fields computed from the fields you already have. 

hdb = hdb.with_columns(
    (pl.col("resale_price") / pl.col("floor_area_sqm")).alias("price_per_sqm"),
)

.alias("name") tells Polars what to call the new column. You can add several columns in one call — more efficient than chaining multiple with_columns because Polars can plan them together: 

hdb = hdb.with_columns(
    pl.col("month").str.to_date("%Y-%m").alias("transaction_date"),
    pl.col("month").str.slice(0, 4).cast(pl.Int32).alias("year"),
    (pl.col("resale_price") / pl.col("floor_area_sqm")).alias("price_per_sqm"),
)

when / then / otherwise: conditional columns

Sometimes the new column depends on a condition — "if price is under 350k call it budget, else if under 500k call it mid-range, else luxury". Polars expresses this with pl.when(...).then(...).otherwise(...), which chains like an if/elif/else: 

hdb = hdb.with_columns(
    pl.when(pl.col("resale_price") < 350_000).then(pl.lit("budget"))
    .when(pl.col("resale_price") < 500_000).then(pl.lit("mid_range"))
    .when(pl.col("resale_price") < 700_000).then(pl.lit("premium"))
    .otherwise(pl.lit("luxury"))
    .alias("price_tier")
)

pl.lit(value) wraps a constant into an expression — without it, Polars would interpret the string "budget" as a column name and raise an error. Any time you mean "the literal value X", use pl.lit(X).

Worked Example: Finding Bishan's Million-Dollar Flats

📝 Goal

Load the HDB resale dataset (~500,000 rows) and answer: how many 4-room flats in Bishan sold for more than S$800,000 in 2023, and what was their median price per square metre?

Step 1.

Load the data and take the 30-second look.

import polars as pl
from shared import MLFPDataLoader

loader = MLFPDataLoader()
hdb = loader.load("mlfp01", "hdb_resale.parquet")

print(f"Shape: {hdb.shape}")
print(f"Columns: {hdb.columns}")
print(hdb.head(3))

Typical output: Shape: (500123, 11) — roughly half a million transactions. The columns include month, town, flat_type, floor_area_sqm, resale_price, and several others we will ignore today.

Step 2.

Add the two derived columns we need: a proper year integer (to filter by 2023) and price per square metre (to answer the final question).

hdb = hdb.with_columns(
    pl.col("month").str.slice(0, 4).cast(pl.Int32).alias("year"),
    (pl.col("resale_price") / pl.col("floor_area_sqm")).alias("price_per_sqm"),
)

str.slice(0, 4) takes the first four characters of the month string (e.g. "2023-07" → "2023"), and cast(pl.Int32) converts that string to an integer so we can compare it with == or >=.

Step 3.

Build the filter chain. We want Bishan, 4-room, 2023, and price > 800,000. Each condition goes in its own parenthesised block, combined with &.

million_dollar = hdb.filter(
    (pl.col("town") == "BISHAN")
    & (pl.col("flat_type") == "4 ROOM")
    & (pl.col("year") == 2023)
    & (pl.col("resale_price") > 800_000)
)

print(f"Matching transactions: {million_dollar.height}")

The exact count depends on the snapshot of the dataset, but in the 2023 HDB data this filter returns a handful of rows — "million-dollar HDBs" are rare but exist. The headline pattern is what matters.

Step 4.

Compute the median price per square metre across those matching rows and format a summary.

if million_dollar.height > 0:
    median_psm = million_dollar["price_per_sqm"].median()
    median_price = million_dollar["resale_price"].median()
    print(f"Bishan 4-room > S$800k, 2023:")
    print(f"  Transactions:       {million_dollar.height}")
    print(f"  Median price:       S${median_price:,.0f}")
    print(f"  Median price / sqm: S${median_psm:,.0f}")
else:
    print("No matching transactions")
Step 5.

Rewrite the whole thing as a single chained pipeline. Read it top to bottom and see how each line narrows the result.

summary = (
    hdb
    .with_columns(
        pl.col("month").str.slice(0, 4).cast(pl.Int32).alias("year"),
        (pl.col("resale_price") / pl.col("floor_area_sqm")).alias("price_per_sqm"),
    )
    .filter(
        (pl.col("town") == "BISHAN")
        & (pl.col("flat_type") == "4 ROOM")
        & (pl.col("year") == 2023)
        & (pl.col("resale_price") > 800_000)
    )
    .select("month", "town", "resale_price", "floor_area_sqm", "price_per_sqm")
    .sort("resale_price", descending=True)
)

print(summary)

That is the canonical shape of a Polars analysis: enrich, filter, select, sort. From here you could pipe the result into a chart, a CSV export, or an aggregation — without ever having to re-read the original 500,000 rows.

Mathematical Foundations: Boolean Algebra

Behind the scenes, Polars is evaluating boolean expressions across the whole column at once — a single logical operation applied to hundreds of thousands of rows simultaneously. The rules that govern those operations are the same boolean algebra you meet in a first discrete-maths course.

Let $A$ and $B$ be two boolean values (each is True or False). Then:

$$A \land B = \text{True} \iff A = \text{True and } B = \text{True}$$
$\land$ — "and", Polars & both must hold
$$A \lor B = \text{True} \iff A = \text{True or } B = \text{True}$$
$\lor$ — "or", Polars | at least one must hold

These combine to form the truth table you will carry in your head for the rest of your data career:

$A$$B$$A \land B$$A \lor B$
TTTT
TFFT
FTFT
FFFF
🔵 Theory: De Morgan's laws

The equivalence $\lnot(A \land B) \equiv (\lnot A) \lor (\lnot B)$ is a De Morgan's law. In Polars terms: "not (A and B)" is the same as "(not A) or (not B)". This matters when you flip a filter. "I want rows where the town is not one of these three" can be written either as ~pl.col("town").is_in([...]) or as "rows where the town is not A and not B and not C". De Morgan guarantees the two forms are equivalent.

Common Mistakes

⚠ Mistake 1: Missing parentheses around conditions

df.filter(pl.col("a") == 1 & pl.col("b") == 2) looks reasonable but is parsed as pl.col("a") == (1 & pl.col("b")) == 2. Always write (pl.col("a") == 1) & (pl.col("b") == 2).

⚠ Mistake 2: Using Python and instead of &

Python's built-in and/or return the first falsy/truthy value — they do not work element-wise on columns. Polars expressions must use & and |. If you write pl.col("a") == 1 and pl.col("b") == 2 Polars will raise a confusing error about "boolean ambiguity". The fix is always the same: use the bitwise operators.

⚠ Mistake 3: Forgetting to assign the result

Polars DataFrames are (for practical purposes) immutable. Writing hdb.filter(...) on a line by itself computes the filtered DataFrame and immediately discards it. Always capture the result: filtered = hdb.filter(...) or overwrite: hdb = hdb.with_columns(...).

⚠ Mistake 4: Bare strings in when / then

In pl.when(...).then("budget"), Polars interprets "budget" as a column name and looks for a column called "budget". If there is no such column you get a cryptic error. Wrap constants with pl.lit(...): pl.when(...).then(pl.lit("budget")).

Reflection

In under an hour you went from looking at a twelve-row dataset to filtering half a million rows of real Singapore property transactions down to the exact subset needed to answer a business question. The three verbs — filter, with_columns, select — and the glue of method chaining cover the vast majority of the data manipulation you will do in this course.

The biggest shift today was the move from imperative thinking ("do this, then this, then this") to declarative thinking ("here is what I want"). Polars expressions describe the answer; the engine figures out how to compute it. That separation is what makes the code both fast and reviewable.

In Lesson 1.3 we leave "one filter at a time" behind and meet group_by — the single most useful pattern in analytics. Instead of asking "how much did Bishan 4-rooms cost?", you will ask "what was the median price for every town and every flat type?", and Polars will produce the entire table in one call.
